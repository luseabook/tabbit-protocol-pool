# Tabbit Protocol Pool

这是一个独立新项目的文档起点，用于探索和实现：

- Tabbit Web 全协议调用，不依赖浏览器 UI 作为运行时通道。
- 多账号 session/cookie 池轮询。
- YYDS Mail/215 邮箱 API 接码。
- 账号注册初始化、活动权益领取、每日签到、额度/重置券维护。
- 对外保持 OpenAI / Anthropic 兼容网关能力。

当前阶段：**基础实现与文档契约并行推进**。已落地配置/脱敏、YYDS Mail 客户端、M01 协议客户端、M02 账号池、账号 JSON 元数据持久化、文件型 secret 引用存储、pooled request runner、受控 `LocalToolLoopRunner`、AccountProvisioner 离线注册/导入编排层、可配置 auth 当前可用版本目标：**手动 cookie 运维可用版本**。用户手动在 Tabbit 注册/登录并通过 `accounts import-session` 导入 cookie/session；运行中由 `verifySession` 识别 401/login_required，将账号标记为 `login_expired`，再由用户手动重新登录并重新导入 cookie/session。当前版本不承诺自动注册、Yoda/短信登录自动化或自动刷新登录态；`automated_session_refresh_strategy` 继续作为后续自动恢复 backlog，不作为 manual-cookie 当前发布目标的硬阻塞。

- `src/account-provisioner.js`：M04 账号注册初始化基础编排层，通过注入邮箱与协议操作执行 inbox 创建、验证码发送/等待、注册/登录提交、session 保存、账号导入、resume hook 和 session 验证；raw cookie/session 只写入 secret store。协议操作返回 `ok:false` 时会停在对应 provisioning 阶段并记录脱敏错误。默认 CLI 依赖已把显式配置的 `TABBIT_POOL_PROTOCOL_AUTH_SEND_CODE_PATH` / `TABBIT_POOL_PROTOCOL_AUTH_SUBMIT_CODE_PATH` 接到 `AccountProvisioner` 可用的 auth 协议方法，但真实注册/登录 endpoint/body success evidence 尚未捕获时，可通过 `tabbit-pool accounts import-session` 临时导入已登录账号 cookie。
- `src/benefits-maintainer.js`：M05 权益额度维护基础编排层，通过注入协议操作执行额度刷新、每日签到、活动 Pro 领取和重置券使用，支持批量维护账号数组或绑定的 accountStore，并能把明确的 login_required、rate_limited、network_error、upstream_error、protocol_changed、forbidden、quota_exhausted 维护错误转为账号状态；只执行已注入的协议操作，不硬编码未知 Tabbit 接口路径。默认 CLI 仅在显式配置 `TABBIT_POOL_PROTOCOL_QUOTA_USAGE_PATH` 时注入真实 `refreshQuota`，在显式配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_PATH` 时注入已验证的 `dailyCheckin`；活动 Pro、抽奖和真实重置券消耗仍不会自动执行，真实用券当前只通过显式 `useResetCoupon` probe/gateway 方法触发。

- `src/anthropic-compat.js`：纯函数式 Anthropic Messages 兼容处理，返回 Anthropic message/error JSON，透传非空 `tools`、`tool_choice`，没有真实工具定义时忽略 `tool_choice:{type:"auto"}` 这类 no-op 工具选项，保留请求中的 `tool_use` / `tool_result` content block 和响应中的内部 `tool_use` content block，并在 `stream:true` 时把数组或 async iterable `streamDeltas` 作为非公开 `stream.deltas` 元数据交给 HTTP 路由。

- `src/protocol-probe.js`：protocol probe fixture foundation，生成脱敏协议探针 fixture，支持 verifySession/sendMessage/listModels/refreshQuota/uploadAttachment、已校准只读 commerce GET 查询和 M05 side-effect probe 注入协议操作、默认写入 `stateDir/fixtures/protocol-probes/`，也可通过 `TABBIT_POOL_PROTOCOL_FIXTURE_DIR` 把脱敏 fixture 读写根移到独立可写目录；本地 fixture list/read 仍使用稳定 `fixtures/protocol-probes/<name>.json` ref 并拒绝路径穿越；fixture sanitizer 会屏蔽附件 `data`、sendMessage prompt/response text、stream delta、raw frame payload 和真实用户标识，避免把探针附件 payload、prompt 或用户数据写入仓库。
- `bin/tabbit-pool.js`：`tabbit-pool` 可执行入口，只负责把命令行参数转给 `runProtocolPoolCli()`。
- `docs/09-实现接口参考.md`：记录当前 `src/index.js` 导出的实现接口、默认值和错误分类。
- `docs/modules/M06-兼容网关/HTTP路由层.md`：记录原生 HTTP server 的路由、认证、JSON 解析、OpenAI/Anthropic SSE adapter 和测试契约。
- `test/`：离线单元测试，不依赖真实 Tabbit 或 YYDS 网络。

## 状态目录与剩余工作

默认 `TABBIT_POOL_STATE_DIR` 可能没有账号、`readiness.json` 或 protocol probe fixture，此时 `readiness`/`fixtures audit` 返回 `blocked` 是安全默认，不代表代码不可用。要复用本机已有脱敏 evidence，可显式选择外部状态目录后运行：

~~~powershell
$env:TABBIT_POOL_STATE_DIR = "E:\tabbit2api\output\tabbit-live-state"
node bin\tabbit-pool.js readiness doctor --json
node bin\tabbit-pool.js readiness --json
node bin\tabbit-pool.js fixtures audit --json
node bin\tabbit-pool.js fixtures audit --scope auth --json
node bin\tabbit-pool.js fixtures audit --scope benefits --json
node bin\tabbit-pool.js fixtures audit --scope session --json
node bin\tabbit-pool.js fixtures audit --scope upstream --json
~~~

当前发布目标的可用性看 `readiness doctor --json` 中的 `manualCookieMode`，或非 JSON 输出的 `manual_cookie_mode` 行。该状态只在基础 gateway/chat readiness、默认 fixture audit、成功 `verifySession` 和 401/login_required 过期 evidence 都满足时为 `ready`；它明确声明 `automatedSessionRefresh.requiredForCurrentRelease:false`。JSON 中 `manualCookieMode.missing` 和 `manualCookieMode.blockingMissing` 只表示当前手动 cookie 发布阻塞项，`manualCookieMode.backlogMissing` 表示后续自动恢复增强；非 JSON 行对应 `release_blocking_missing=<csv>` 和 `backlog_missing=<csv>`。因此 `calibrationBacklog.status:"blocked"` 仍可同时存在，用于跟踪注册/登录、M05 副作用、真实上游 stream boundary 和自动 session recovery 后续增强。

如果外部 `TABBIT_POOL_STATE_DIR` 只能读账号和 secret、不能写 fixture，可额外设置 `TABBIT_POOL_PROTOCOL_FIXTURE_DIR` 到一个可写的脱敏 fixture 目录。该变量只影响 `FileProtocolFixtureStore` 的 fixture 读写根，账号元数据、secret 和 `readiness.json` 仍来自 `TABBIT_POOL_STATE_DIR`。

`probe protocol --json` 打印前会再次经过 sanitizer；即使 runner 返回了 prompt、stream delta、raw frame、Bearer、cookie、session 或真实用户标识，也只能输出脱敏后的 JSON。

`fixtures audit --scope auth`、`fixtures audit --scope benefits`、`fixtures audit --scope session` 和 `fixtures audit --scope upstream` 都只读本地脱敏 fixture，并且不改变默认 gateway/chat readiness。auth scope 报告注册/登录 evidence 覆盖：发送验证码 coverage 要求成功 fixture 含 send/delivery 专属成功信号，泛 2xx、`ok/status/result:"success"` 只计入 transport success，不闭环；提交验证码/注册登录需要成功 fixture 且响应形状包含可供 `AccountProvisioner.extractSessionSecret()` 导入的 session material 字段。显式配置 auth send/submit path 后，默认 AccountProvisioner 协议适配器可以调用真实 `ProtocolTabbitClient` 方法，但这只代表运行路径已接通，不代表 auth 校准完成；仍必须通过 auth scope 的 delivery success 与 session material fixture gate。benefits scope 报告每日签到、活动 Pro 成功、真实重置券消耗和抽奖成功 evidence 覆盖，其中活动 Pro 成功必须有 participation/activity/claim/pro 专属成功字段，不能只凭 `ok/status/result:"success"` 闭环；真实用券消耗必须来自消费类 operation，并同时具备脱敏 `endpointHash/bodyHash/resultHash`、`safe:true`、`sanitized:true`、`rawPayload:false` evidence 和真实消费成功信号；`participateResetCouponActivity` 只作为活动参与 evidence，永不满足真实用券消耗 coverage，`drawLottery` 也必须有 draw/lottery 专属成功或非空奖品/命中记录；benefits scope 的读取和 `total/success/failed` 计数只覆盖 M05 side-effect 白名单 operation，不会被 `sendMessage`、`verifySession` 等无关 fixture 污染；session scope 报告 `verifySession` 成功 evidence 与上游 401/login_required 过期 evidence，并把本地 `session_missing` 单独计数，不当成上游过期；即使 lifecycle evidence 已 ready，仍必须有显式脱敏 `session_recovery_strategy` / `recoverSession` evidence，且该 evidence 声明 `safe:true`、`sanitized:true`、`rawPayload:false`、正整数 `observedWindowMs`、`resultHash` 脱敏哈希，并在 `result` 中证明 `expiredBeforeRecovery:true` 与 `recoveredVerifySession:true`，`automated_session_refresh_strategy` 才会 ready。upstream scope 只读取 `sendMessage` fixture，且只有同时带显式真实上游 evidence marker 与 stream/SSE/NDJSON 元数据的脱敏样本才会满足错误帧、取消或 backpressure coverage；非流式 `protocol_probe` 样本可用于默认 send readiness，但不满足 upstream stream boundary readiness；本地 HTTP route、compat 或单测 fake stream 不会被当成真实上游校准。非 JSON 的 auth/benefits audit 会打印 transport/strict evidence 计数和 `missing` 行；非 JSON 的 session audit 也会打印 `recovery_strategy` 和 `missing` 行；非 JSON 的 upstream audit 会打印真实上游错误帧、取消、backpressure 计数，便于人工巡检时直接看到哪些 stream boundary evidence 仍缺失。

session scope 的 `counts.rejectedRecoveryStrategyEvidence` 和 plain 输出 `recovery_strategy_rejected` 只表示存在 `recoverSession` / `session_recovery_strategy` 候选 fixture 但未通过严格 proof gate。该诊断不会满足 `automated_session_refresh_strategy`，也不会启用任何自动刷新或 re-auth 行为。

session scope 的 plain 输出会额外打印 `session_lifecycle	last_successful_at=<iso>	last_expired_at=<iso>	observed_window_ms=<ms>`，这些值只来自成功/过期 verifySession 脱敏 fixture 的聚合时间窗口，便于观察 cookie 失效窗口；它仍只是 observation evidence，不满足 `automated_session_refresh_strategy`，也不会启用自动刷新。

session scope 同时输出 `manualCookieOperations`，plain 输出为 `manual_cookie_mode`。当成功 `verifySession` 与上游 401/login_required 过期 evidence 都存在时，该字段可为 `ready`，表示当前 manual-cookie 运维链路具备“失效识别 -> 标记 login_expired -> 人工重新导入 cookie/session”的证据；它不要求也不启用自动刷新。JSON 中 `manualCookieOperations.blockingMissing` 只列当前手动 cookie 发布目标仍缺的 evidence，`manualCookieOperations.backlogMissing` 只列后续增强；plain `manual_cookie_mode` 对应输出 `release_blocking_missing=<csv>` 和 `backlog_missing=<csv>`，因此 `backlog_missing=automated_session_refresh_strategy` 不再被误读为当前发布硬阻塞。

仅凭 `kind:"protocol_probe"` 不构成真实上游 marker；即便样本包含 `streamDeltas`、SSE 或 NDJSON 元数据，也只能证明默认 streaming fixture 形状，必须额外声明安全脱敏的 Tabbit/live/upstream 来源、`upstreamEvidence.real:true` 或 `result.raw.upstream:true` 等真实上游 marker 才能进入 upstream scope 覆盖计数。泛 `source:"protocol-client"` / `evidenceSource:"protocol"` 这类协议客户端标签不计为真实 Tabbit 上游证据。

真实 `/api/v1/chat/completion` async SSE/NDJSON 分支会在 `ProtocolTabbitClient.sendMessage({ stream:true })` 结果中附加安全 `upstreamEvidence` marker；通过 `probe protocol --write-fixture` 写入后，fixture 可被 upstream scope 识别。本地 HTTP route、compat handler、旧显式 sendPath 和 fake stream 仍不会因为 `kind:"protocol_probe"` 自动计为真实上游 evidence。

`sendMessage` 探针可显式传入 `streamEvidence:{ mode, maxDeltas }` 触发有界 async stream evidence 捕获；CLI 只接受 `first_token_backpressure`、`cancel_after_first_delta`、`error_frame` 三种模式，`maxDeltas` 限制为 1 到 5。runner 只在结果已有 `source:"tabbit-live"`、`real:true`、`stream:true` marker 时消费 bounded delta，并只写入 `backpressure`、`firstTokenFlush`、`delayedSecondChunk`、`cancellation` 或 `streamErrorFrame` 等聚合布尔值，不保存 raw stream 文本、prompt、payload、cookie、session、token 或真实用户数据。若请求了 `streamEvidence` 但真实 async upstream marker、目标错误帧、取消信号或两段 backpressure delta 没有被捕获，probe 会 fail closed，写入脱敏 failed fixture 和 `STREAM_EVIDENCE_NOT_CAPTURED`，不会把普通 success 误报成真实上游证据。

`fixtures audit --scope upstream` 会把 `STREAM_EVIDENCE_NOT_CAPTURED` 失败样本统计到 `counts.streamEvidenceNotCaptured`，plain 输出为 `stream_evidence_not_captured`。该计数只表示有界捕获已经尝试但目标边界证据不足，即使样本带真实 upstream marker，也不会满足 `real_upstream_error_frame_fixture`、`real_upstream_cancellation_fixture` 或 `real_upstream_backpressure_fixture` readiness。

需要生成真实上游 stream boundary 捕获骨架时，可以直接用模板命令，例如 `node bin\tabbit-pool.js probe template --operation sendMessage --stream-evidence error_frame --json`。该命令只生成安全占位 input，sendMessage 内容默认为 `<redacted-message-content>`；`probe validate` 会接受该占位模板用于形状预检，但真实 `probe protocol --operation sendMessage` 会在调用 runner 前拒绝未替换的占位内容。实际运行前仍必须审查 `<redacted-input.json>`、替换为确认可公开的最小测试内容、执行 `probe validate --operation sendMessage --input-file <redacted-input.json> --json`，并且只持久化 protocol sanitizer 输出的脱敏 fixture。`probe validate --operation sendMessage --json` 会额外输出脱敏 `sendMessageReview` 摘要；其中 `protocolDispatchReady:true` 只表示至少存在一条非 `<redacted-message-content>` 的消息字符串，不证明内容已经安全、真实上游已成功或 readiness 已满足。doctor JSON 中所有 sendMessage capture command 都会带 `requiresReviewedInput:true` 与 `reviewRequirement:"replace_redacted_message_content"`；plain `capture_command` 对应 `review=replace_redacted_message_content`。

每日签到 coverage 接受 `signInResult:"success"`、`signedToday:true` 或 `signInResult:"already_signed"`；`already_signed` 只表示当天签到状态已完成，不代表重复发放额度，也不会放宽活动 Pro、抽奖或重置券的成功判定。

`readiness doctor --json` 现在会同时输出 core readiness 的 `remainingWork`、当前发布目标的 `manualCookieMode` 和扩展校准的 `calibrationBacklog`；`protocol.authSendCodePathConfigured` 与 `protocol.authSubmitCodePathConfigured` 只暴露布尔配置状态，不输出 endpoint path。非 JSON 输出也会打印 `auth_send_endpoint`、`auth_submit_endpoint`、`manual_cookie_mode`、`calibration_backlog`、`auth_backlog`、`benefits_backlog`、`session_backlog`、`upstream_backlog` 和占位符 `capture_command` 行。当 `status:"ready"` 且 `remainingWork:[]` 时，只表示基础 gateway/chat 上线门槛已清；当 `manualCookieMode.status:"ready"` 且 `manualCookieMode.blockingMissing:[]` 时，才表示当前手动 cookie 运维目标的失效识别 evidence 也已具备。`manualCookieMode.backlogMissing` 可继续列出 `automated_session_refresh_strategy`，但这是后续自动恢复增强，不阻塞 manual-cookie 当前发布目标。`calibrationBacklog.status:"blocked"` 或 plain 输出中 `calibration_backlog` 仍为 `blocked` 时，继续表示注册/登录、M05 副作用、session 自动恢复或真实上游 stream boundary 仍有后续校准项。auth endpoint 显示 `configured` 只说明运行路径有显式配置，不说明发送验证码或提交验证码成功语义已闭环。`calibrationBacklog.captureCommands` / plain `capture_command` 只给出带 `<account-id>` / `<redacted-input.json>` 占位符的安全 `templateCommand`、`validateCommand`、`confirmedValidateCommand` 和 `probeCommand`；auth、benefits、session 与 upstream capture command 会在 JSON 中输出 `prerequisites` 与 `prerequisitesStatus`，plain 行对应 `template=`、`validate=`、`confirm_validate=`、`probe=`、`prereq=ENV:configured|missing`。当前会覆盖 `TABBIT_POOL_PROTOCOL_AUTH_SEND_CODE_PATH`、`TABBIT_POOL_PROTOCOL_AUTH_SUBMIT_CODE_PATH`、`TABBIT_POOL_PROTOCOL_SIGN_IN_PATH`、`TABBIT_POOL_PROTOCOL_BENEFIT_COUPON_USE_PATH`、`TABBIT_POOL_PROTOCOL_ACTIVITY_PARTICIPATE_PATH`、`TABBIT_POOL_PROTOCOL_LOTTERY_DRAW_PATH`、`TABBIT_POOL_PROTOCOL_SESSION_VERIFY_PATH` 和 upstream `TABBIT_POOL_PROTOCOL_SEND_PATH`。prerequisite 只暴露 env 名和配置状态，既不输出 endpoint 值，也不证明 endpoint/body/side-effect/success 语义已校准。真实副作用仍必须先人工确认 input file 和 `confirmSideEffect`，并先用 `probe validate --operation <operation> --input-file <redacted-input.json> --json` 做只读 schema 与脱敏形状预检。准备运行副作用 `probe protocol --write-fixture` 前，再执行 `confirm_validate` 对应命令，离线确认 `confirmSideEffect:true`，然后再决定是否执行真实 probe。真实用券消耗缺口现在给出 `useResetCoupon` template/validate/confirm_validate/probe 命令，并要求先选择安全券码；离线 `consumeResetCoupon` 模板仅用于外部脱敏 evidence 导入。自动刷新缺口会给出离线 `recoverSession` template/validate 命令，但 `confirm_validate` 和 `probe` 保持为空；这只帮助准备安全脱敏 recovery evidence，不代表可触网刷新 endpoint 已校准，也不阻塞 manual-cookie 当前发布目标。upstream 缺口的 `templateCommand` 会直接带 `--stream-evidence <mode>` 生成对应捕获骨架。离线 `recoverSession` / `consumeResetCoupon` evidence 通过人工审查后，可追加 `--write-fixture` 让 `probe validate` 只写白名单脱敏 fixture；该写入不会读取账号、secret 或运行网络 probe，也不会把 input 中额外 raw payload 字段写入 state。只有安全脱敏恢复策略 evidence fixture 同时包含过期窗口、脱敏结果 hash 和恢复后 `verifySession` 成功布尔信号并进入 state 后，session scope 才会从 `manual_reimport_then_probe` 升级为已校准自动恢复。

`forbidden_403_fixture` 也会出现在 `calibrationBacklog.captureCommands` / plain `capture_command` 中，来源是默认 fixture audit 的 403 缺口，而不是 auth/benefits/session/upstream 扩展 scope。该命令使用只读 `verifySession` 模板和 `<account-id>` / `<redacted-input.json>` 占位符，只在实际观测到 403/forbidden 后保存 sanitizer 输出；命令本身不会让 403 readiness 变为 ready，也不能替代真实脱敏 403 fixture。

默认 fixture audit 的 send 侧缺口也会出现在 doctor capture command 中：`successful_sendMessage_fixture`、`streaming_text_fixture` 和 `tool_call_fixture` 都使用 `sendMessage`、`TABBIT_POOL_PROTOCOL_SEND_PATH` 前置状态和 `<redacted-input.json>` 占位符。`streaming_text_fixture` 的命令提示准备 `stream:true` 输入；`tool_call_fixture` 的命令只说明可捕获真实 tool-call 脱敏 fixture，或捕获“上游明确不支持原生工具字段”的脱敏 evidence，不代表 Tabbit 原生工具字段已经支持。sendMessage capture command 的 `review` 标记表示必须先替换 `<redacted-message-content>`，否则 `probe protocol` 会在调 runner 前拒绝；替换后还应复跑 `probe validate` 并查看 `sendMessageReview.protocolDispatchReady`。三者都必须由真实 sanitizer 输出进入 fixture store 后才会满足默认 audit 和 `manualCookieMode`，doctor 本身不会触网、写 fixture 或放宽 gate。

`readiness doctor --json` 的 `commands.accountPreflightReadOnly` 和 plain 输出的 `preflight_command	account_read_only` 只展示下一步只读预检命令：`node bin\tabbit-pool.js accounts probe <account-id> --read-only --json`。doctor 本身不会调用 verifier、不会保存账号状态、不会写 fixture；操作者可在准备真实脱敏 evidence 前，用该命令检查当前导入的 cookie/session 会被投影为 active、suspect 或 login_expired。

`readiness doctor --json` 的 `commands.codexE2EMark`、`commands.claudeE2EMark`、`commands.combinedE2EMark` 和 plain 输出的 `mark_command` 行，只是人工 Codex / Claude Code E2E 验收完成后的本地持久化提示：`readiness mark --codex-verified/--claude-verified --json`。doctor 不会执行这些命令，也不会写 `readiness.json`；在未完成真实 base_url/key/model 验收前，不应使用 mark command 伪造 `codex_e2e_verified` 或 `claude_code_e2e_verified`。

`readiness doctor --json` 会对 `successful_reset_coupon_consumption_fixture` 和 `automated_session_refresh_strategy` 额外给出 `writeFixtureCommand`；非 JSON `capture_command` 行对应打印 `write_fixture=`。该字段只用于 `probe validate --write-fixture` 的离线脱敏 evidence 入库，不会为 auth、每日签到、活动 Pro、抽奖、verifySession 或真实上游 sendMessage 生成替代 probe 命令。

upstream 三个 `captureCommands` 的 `templateCommand` 会直接生成对应 `--stream-evidence` 模板命令，同时在 JSON 中保留 `recommendedInput.streamEvidence`，plain `capture_command` 行会打印 `stream_evidence=<mode>:<maxDeltas>`：error-frame 使用 `error_frame:2`，取消使用 `cancel_after_first_delta:2`，backpressure 使用 `first_token_backpressure:2`。这些 sendMessage capture command 也会带 `review=replace_redacted_message_content`。这些提示只包含安全模式、review 要求和 delta 数量，不包含 prompt、endpoint、payload、cookie、session、token 或真实用户数据；实际运行前仍必须准备并审查 `<redacted-input.json>`，只保存 sanitizer 输出的聚合 marker。

剩余工程重点仍是注册/登录 Yoda 后验证码投递成功 evidence、短信码提交后的 session material evidence、活动 Pro 成功领取 body、抽奖成功响应、真实上游私有工具语义或最终本地 loop 策略，以及 session 过期/刷新、用券后额度刷新策略和上游取消/backpressure 的更多 fixture 校准。`tabbit-cookie.txt`、`output/`、浏览器 profile、raw 抓包和本地 state fixture 默认不提交。

## 密钥约定

任何真实密钥、账号密码、cookie、session token 都不写入仓库。运行时通过环境变量或本地加密状态注入：

~~~powershell
$env:YYDS_MAIL_API_KEY="AC-..."
~~~

示例中的 `AC-...`、`sk-tabbit-local` 都是占位符，不代表真实服务密钥。

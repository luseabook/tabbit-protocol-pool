# 12-账号风控与 403 排障

本文专门说明账号风控问题。目标不是把 403 当成普通网络错误无限重试，而是把账号状态、冷却、fixture 校准和人工动作拆开，避免一个异常账号拖垮整个本地网关。

## 403 在系统里的含义

`ProtocolTabbitClient.classifyProtocolError()` 当前把 HTTP 403 归类为：

~~~text
category: forbidden
retryable: false
cooldownMs: 30 min
~~~

在账号状态流转中，`forbidden` 更接近“账号或会话可疑”：

| 信号 | 分类 | 账号状态倾向 | 处理动作 |
|---|---|---|---|
| HTTP 401 | `login_required` | `login_expired` | 刷新或重新导入 session。 |
| HTTP 403 | `forbidden` | `suspect` 或 `cooldown` | 先隔离账号，跑 session probe，必要时重新登录。 |
| HTTP 429 | `rate_limited` | `cooldown` | 按 `Retry-After` 或默认冷却等待。 |
| quota / credit / usage 耗尽 | `quota_exhausted` | `quota_exhausted` | 切换账号，等待重置或使用重置券。 |
| 协议字段变了 | `protocol_changed` | `suspect` | 不轮询账号，先抓 fixture 更新解析。 |

## 风控策略

### 请求期

一次模型请求通过 `PooledRequestRunner` 执行：

1. 从账号池选择一个候选账号。
2. 调用 `ProtocolTabbitClient.sendMessage()`。
3. 成功则记录账号成功状态。
4. 失败则按错误分类更新账号状态。
5. 只有 retryable 且仍有候选账号时才 fallback。

`forbidden` 默认不是 retryable，不会在同一请求里盲目消耗更多账号。这样做的原因是 403 可能表示协议、会话或账号环境整体异常，连续换账号可能放大风控面。

### 账号状态

账号池建议保留以下状态：

| 状态 | 含义 | 是否参与请求 |
|---|---|---|
| `active` | 可用账号 | 是 |
| `cooldown` | 临时冷却 | 到期后再参与 |
| `quota_exhausted` | 额度耗尽 | 额度恢复或重置后再参与 |
| `login_expired` | session 失效 | 刷新登录后再参与 |
| `suspect` | 风控、403、协议疑似异常 | 默认不参与，需 probe 或人工确认 |
| `disabled` | 手动禁用 | 否 |
| `provisioning` | 初始化中 | 否 |

### 冷却与隔离

- 403：优先隔离为 `suspect`，或按明确策略进入 30 分钟 `cooldown`。
- 429：使用 `Retry-After`，没有时默认 60 秒。
- 网络/5xx：指数退避或短冷却，不直接怀疑账号。
- `protocol_changed`：不做账号轮询，先更新协议 fixture 和解析测试。

## 排障入口

### 查看账号池健康

~~~powershell
node bin/tabbit-pool.js health --json
~~~

输出关注：

- `status`
- `accounts.active`
- `accounts.byStatus.suspect`
- `alerts[]`

### 查看脱敏账号列表

~~~powershell
node bin/tabbit-pool.js accounts list --json
~~~

输出不会包含 cookie、session、token、`cookieJarRef`。

### 导入已登录账号 session

真实注册/登录 endpoint 未还原前，可以临时导入本机已登录账号的 cookie/session：

~~~powershell
node bin/tabbit-pool.js accounts import-session --id acct_default --email user@example.test --access-tier pro --cookie-file .\tabbit-cookie.txt --json
~~~

命令不请求 Tabbit 网络；raw cookie/session 只写入本地 secret store，stdout 只返回脱敏账号和 action events。导入后再执行 `accounts probe` 判断本地 secret 是否存在、真实 session verifier 是否已配置。

### 单账号 session 探测

~~~powershell
node bin/tabbit-pool.js accounts probe acct_default --json
~~~

用途：确认本地 secret 是否存在、session verifier 是否可用、账号是否应恢复 `active` 或转 `login_expired` / `suspect`。

### 403 建议

~~~powershell
node bin/tabbit-pool.js probe advice --category forbidden --json
node bin/tabbit-pool.js probe advice --status 403 --message "<403 response message>" --json
~~~

该命令不触网，只给下一步建议。带 `--message` 或 `--code` 时会先脱敏再输出细分建议。当前会把 403 细分为 `risk_control`、`signature_or_protocol`、`session_or_cookie`、`entitlement_or_model` 和 `unknown_forbidden`。

### 真实协议探针与 fixture

先生成安全模板：

~~~powershell
node bin/tabbit-pool.js probe template --operation sendMessage --json > .\probe-input.json
~~~

再对指定账号执行探针并写脱敏 fixture：

~~~powershell
node bin/tabbit-pool.js probe protocol --account acct_default --operation sendMessage --input-file .\probe-input.json --write-fixture --json
~~~

查看 fixture：

~~~powershell
node bin/tabbit-pool.js fixtures list --json
node bin/tabbit-pool.js fixtures show fixtures/protocol-probes/<ref>.json --json
~~~

fixture 用于回答：

1. 403 是 session 失效、权限不足、请求体字段错误，还是协议签名错误？
2. 响应体有没有稳定的 `code` / `message` / `reason` 可以更细分类？
3. 是否需要把当前账号转为 `login_expired`、`suspect`、`cooldown` 或 `disabled`？
4. 是否是全账号都会发生的 `protocol_changed`？

## 什么时候需要前端管理后台

### 当前不需要

MVP 阶段使用 CLI 更合适：

- 本地运行，默认只监听 `127.0.0.1`。
- 所有账号 secret 只在本地 store / secret ref 中存在。
- 排障输出可以强制脱敏并纳入单元测试。
- 真实 endpoint 还在校准，不应先做 UI 承诺。

### 后续可以做

当满足以下条件时再做 Web 管理后台：

1. 真实 send/session/benefits endpoint 已稳定。
2. 账号数量变多，CLI 表格不够用。
3. 需要人工确认账号状态、恢复账号、禁用账号、查看 quota 趋势。
4. 已有统一的只读脱敏 API 和写操作审计日志。

后台应该只包装 M08 能力，不直接读取或展示 raw cookie/session/token。

## 403 回归测试要求

涉及 403/风控逻辑时至少覆盖：

- `classifyProtocolError({ status:403 })` 返回 `category:"forbidden"`。
- `AccountPool.recordFailure()` 能把明确 forbidden 转为 `suspect` 或配置的冷却状态。
- `PooledRequestRunner` 不对非 retryable forbidden 做无意义轮询。
- `protocolProbeAdvice({ category:"forbidden" })` 给出可执行建议。
- CLI 输出不泄露 cookie/session/token/验证码。

## 当前缺口

- 真实 Tabbit 403 响应体还需要脱敏 fixture 校准。
- 403 是否一定表示风控，还是可能表示签名/路径/权限错误，需要靠 fixture 区分。
- 自动恢复策略已有真实 session verify endpoint（`GET /api/v0/user/base-info`）可作为基础；刷新/续期策略仍需继续观察。
- 前端管理后台暂不做；先保证 CLI、`/health` 和 fixture 可用。

# 06-数据字典

本文件定义实现前的数据契约。代码落地时字段可以细化，但不能静默改变语义；改变公开字段必须同步更新测试和 API 文档。

## Account

| 字段 | 类型 | 说明 | 是否敏感 |
|---|---|---|---|
| id | string | 本地账号 ID，建议使用 `acct_` 前缀 | 否 |
| email | string | 注册邮箱 | 部分敏感，日志可脱敏 |
| cookieJarRef | string | cookie jar 存储引用 | 是 |
| userId | string/null | Tabbit 用户 ID | 部分敏感 |
| status | enum | active / cooldown / quota_exhausted / login_expired / disabled / provisioning / suspect | 否 |
| accessTier | string | free / pro / unknown | 否 |
| proClaimed | boolean | 是否已领取活动 Pro | 否 |
| quotaState | QuotaState[] | 额度摘要，可按模型或权益维度记录 | 否 |
| resetCouponCount | number | 可用重置券数量 | 否 |
| lastCheckinAt | datetime/null | 最近签到时间 | 否 |
| lastMaintainedAt | datetime/null | 最近维护时间 | 否 |
| lastProvisioningStage | string/null | 最近账号初始化阶段，如 createInbox / waitForVerificationCode / saveSession | 否 |

## AccountStatus

| 状态 | 可被请求选择 | 说明 | 退出条件 |
|---|---:|---|---|
| provisioning | 否 | 正在注册或初始化 | 注册成功转 active；失败转 disabled/suspect |
| active | 是 | 可正常处理请求 | 失败后按类型转移 |
| cooldown | 否 | 临时冷却 | 到达 `cooldownUntil` 后重新探测 |
| quota_exhausted | 否 | 额度耗尽 | 额度刷新或重置券成功后转 active |
| login_expired | 否 | session/cookie 失效 | 刷新登录成功后转 active |
| suspect | 否 | 协议或风控异常，需人工确认 | 探针成功后转 active；确认无效后 disabled |
| disabled | 否 | 人工或策略禁用 | 人工启用后进入探测 |

## AccountStoreDocument

账号 JSON 元数据 store 当前使用 versioned document。raw cookie、session、token、password、Authorization、apiKey 不应写入该文件。

| 字段 | 类型 | 说明 |
|---|---|---|
| version | number | 当前为 1。 |

| accounts | Account[] | 经过 normalizeAccount() 与 secret 字段剥离后的账号元数据。 |

默认路径：

~~~text
<state-dir>/accounts.json
~~~

## SecretRef

SecretRef 是账号元数据中保存的本地 secret 引用字符串。当前主要字段是 `Account.cookieJarRef`。

| 规则 | 说明 |
|---|---|
| 类型 | string |
| 形态 | 相对路径，如 `secrets/acct_a.cookie` |
| 解析根目录 | `TABBIT_POOL_STATE_DIR` / `config.stateDir` |
| 禁止 | 空值、绝对路径、`..` 路径段、Windows drive-letter 路径 |
| 读取缺失 | `FileSecretStore.readSecret()` 返回 `null` |
| 写回账号 JSON | 只保留 ref，不写回 raw secret |

运行时 hydrate 后，secret 只进入临时账号对象的 `cookieHeader`，不进入 `accounts.json`。

## ProtocolFixtureAudit

`buildProtocolFixtureAudit()` 和 `tabbit-pool fixtures audit` 输出的聚合审计结构。它只包含计数、覆盖状态、缺口和固定下一步建议，不包含 raw request/response、cookie、session、token、prompt 或真实用户数据。

| 字段 | 类型 | 说明 |
|---|---|---|
| scope | enum/null | `protocol` 默认 scope 可省略；扩展 scope 为 `auth`、`benefits`、`session`。 |
| status | enum | ready / blocked。 |
| counts | object | 当前 scope 的聚合计数。auth scope 包含 `sendVerificationCode`、`submitRegistrationOrLogin`、`successfulSendVerificationCode`、`successfulSendVerificationCodeWithDeliverySignal`、`successfulSubmitRegistrationOrLogin`、`successfulSubmitRegistrationOrLoginWithSessionMaterial`、`success`、`failed`；session scope 包含 `verifySession`、`successfulVerifySession`、`expiredVerifySession`、`sessionMissing`、`recoveryStrategyEvidence`、`success`、`failed`。 |
| coverage | object | 覆盖项状态。auth scope 的 submit coverage 使用 `successfulSubmitRegistrationOrLoginWithSessionMaterial`；session scope 包含 `successfulSessionVerify` 和 `expiredSessionSignal`。 |
| lifecycle | object/null | 仅 session scope 输出，包含 `lastSuccessfulAt`、`lastExpiredAt`、`observedWindowMs`。 |
| recoveryStrategy | object/null | 仅 session scope 输出，当前为 `status:"blocked"`、`current:"manual_reimport_then_probe"`，`automatedRefresh:"not_calibrated"`。 |
| missing | string[] | 固定缺口名称。session scope 使用 `successful_verifySession_fixture`、`expired_verifySession_fixture` 和 `automated_session_refresh_strategy`。 |
| nextActions | string[] | 固定建议，不包含原始 fixture 内容。 |

auth scope 中，`successfulSubmitRegistrationOrLogin` 只表示 submit transport success；只有响应形状包含可导入 session material 字段时，才计入 `successfulSubmitRegistrationOrLoginWithSessionMaterial` 并让 submit coverage ready。session scope 中，`session_missing` 只表示本地 secret 缺失，不能作为上游 session 过期 evidence；只有 `verifySession` 的 401 或 `login_required` 失败 fixture 才能计入 `expiredSessionSignal`。`session_recovery_strategy` / `recoverSession` evidence 必须同时包含 `safe:true`、`sanitized:true`、`rawPayload:false`、正整数 `observedWindowMs`、`resultHash` 的 `sha256:` 脱敏哈希，以及 `result.expiredBeforeRecovery:true` 和 `result.recoveredVerifySession:true`，marker-only 恢复策略不计入 `recoveryStrategyEvidence`。

## ProtocolCalibrationCaptureCommand

`buildReadinessDoctorReport().calibrationBacklog.captureCommands[]` 和 `tabbit-pool readiness doctor` plain `capture_command` 行使用的安全命令提示结构。所有命令都只包含 `<account-id>`、`<redacted-input.json>` 等占位符，不包含真实账号、cookie、session、验证码、API key、Bearer token、raw payload、prompt 或真实用户数据。

| 字段 | 类型 | 说明 |
|---|---|---|
| missing | string | 对应缺口名称，如 `successful_sendVerificationCode_fixture`。 |
| scope | enum | `auth`、`benefits` 或 `session`。 |
| operation | string/null | 已校准 probe operation；未知真实用券消耗或自动刷新路径为 `null`。 |
| sideEffect | boolean | 是否为副作用校准项。 |
| templateCommand | string/null | 生成 redacted input 模板的命令；无已校准 operation 时为 `null`。真实上游 stream boundary 缺口会直接带 `--stream-evidence <mode>`，但仍只包含安全模式名和 delta 数量。 |
| validateCommand | string/null | 只读 schema/字段形状预检命令；无已校准 operation 时为 `null`。 |
| confirmedValidateCommand | string/null | 只读 `confirmSideEffect:true` 门禁命令；仅已校准 side-effect operation 有值，read-only operation 或未知 operation 为 `null`。 |
| probeCommand | string/null | 写入脱敏 fixture 的 probe 命令；无已校准 operation 时为 `null`。 |
| reason | string | 固定说明，解释缺口和安全前置条件。 |

plain 输出中字段名分别是 `template=`、`validate=`、`confirm_validate=`、`probe=`。`validateCommand` 与 `confirmedValidateCommand` 都不触网、不读取账号或 secret，也不证明真实 endpoint/body 已校准。

## 流式响应内部字段

这些字段只在协议客户端、compat handler 和 HTTP route adapter 之间流转，不是公开 JSON body 的稳定字段。端到端行为见 [流式 SSE 链路](10-流式SSE链路.md)。

| 字段 | 类型 | 所属对象 | 说明 | 是否公开 |
|---|---|---|---|---|
| streamDeltas | string[] / AsyncIterable<string \| ToolCallDelta> | ProtocolTabbitClient 成功结果 | 上游 SSE/NDJSON 中解析出的流式分片。数组表示 buffered 文本解析完成；AsyncIterable 表示可边读上游 response.body 边产出文本字符串或工具调用 delta。 | 否 |
| stream.deltas | string[] / AsyncIterable<string \| ToolCallDelta> | OpenAICompat / AnthropicCompat 返回对象 | handler 给 HTTP route adapter 的非公开元数据，不能写入 public JSON body；`ToolCallDelta` 会由 HTTP 层转成官方风格工具调用 SSE。 | 否 |
| raw.kind | "stream" | ProtocolTabbitClient raw | 表示响应来自 SSE/NDJSON parser foundation。 | 否 |
| raw.format | "sse" / "ndjson" | ProtocolTabbitClient raw | 记录上游 wire format，便于 fixture 和排错。 | 否 |
| raw.async | boolean | ProtocolTabbitClient raw | true 表示 response.body async producer 路径。 | 否 |
| raw.events | array | ProtocolTabbitClient raw | 结构化流事件；错误帧也会先追加到这里再分类/抛出。buffered OpenAI stream `tool_calls` 与 Anthropic stream `input_json_delta` 聚合也基于这些事件。保存 fixture 前必须脱敏。 | 否 |
| upstreamEvidence | object | 脱敏 protocol probe fixture | 可选真实上游 evidence marker，用于 `fixtures audit --scope upstream` 区分真实 Tabbit stream boundary 样本和本地 HTTP/compat/fake stream 测试。只允许保存布尔 marker、来源类别和脱敏摘要，不保存 raw payload、prompt、cookie、session、token 或真实用户数据。 | 否 |
| streamEvidence | object | sendMessage probe input | 显式 opt-in 的真实上游 stream boundary 捕获请求。只允许 `mode` 为 `first_token_backpressure`、`cancel_after_first_delta` 或 `error_frame`，`maxDeltas` 为 1 到 5；只影响 runner 是否有界消费 async `streamDeltas` 并写入聚合 marker，不允许保存 raw delta。 | 否 |

`upstreamEvidence` 只有与 stream/SSE/NDJSON 元数据同时存在时才会被 upstream scope 计入真实边界 evidence；泛 `kind:"protocol_probe"` 流式样本不再作为真实上游 marker，泛 `source:"protocol-client"` / `evidenceSource:"protocol"` 也不能单独计入，避免把本地或默认 streaming 样本误计为 Tabbit upstream 校准结果。真实 `/api/v1/chat/completion` async SSE/NDJSON 分支会在返回结果中附加安全的 `upstreamEvidence:{ source:"tabbit-live", real:true, stream:true, format }`，后续 `ProtocolProbeRunner` 写入的脱敏 fixture 可据此进入 upstream audit；该 marker 不包含 raw frame、prompt、cookie、session、token 或真实用户数据。

`ToolCallDelta` 是 async 路径的内部对象：

~~~ts
type ToolCallDelta = {
  type: "tool_call_delta";
  index: number;
  id?: string;
  name?: string;
  argumentsDelta?: string;
};
~~~

## 官方工具字段内部通道

这些字段来自 OpenAI / Anthropic 官方 API 风格请求，用于让 Codex、Claude Code 和三方 SDK 的工具定义不在兼容层被吞掉。当前已定义工具定义透传、工具回合输入保真、基础输出映射、async tool delta SSE 输出和受控本地工具 loop。真实 `/api/v1/chat/completion` 仍不支持原生工具字段；本地 loop 只有显式 `local_executes_tools` 且宿主注入 executor 时才执行，并受 allowlist、最大轮数、单次工具超时和结果截断约束。

| 字段 | 类型 | 来源 | 内部映射 | 协议请求字段 | 说明 | 是否公开 |
|---|---|---|---|---|---|---|
| tools | unknown[] | OpenAI Chat/Responses、Anthropic Messages | tools | tools | 工具定义数组，默认由 compat 与 runner 原样透传；旧显式 sendPath 写入协议 body，真实 `/api/v1/chat/completion` 返回 unsupported。`local_executes_tools` 模式会把定义转成文本约束并从真实协议请求中剥离。 | 否 |
| tool_choice | unknown | OpenAI Chat/Responses、Anthropic Messages | toolChoice | tool_choice | 工具选择策略。OpenAI 有 tools 且缺省时为 `"auto"`；Anthropic 有 tools 且缺省时为 `{ type:"auto" }`。没有真实 tools 时，compat 会忽略 `auto/none` 这类 no-op 选择，避免文本请求误触发工具未支持错误。 | 否 |
| parallel_tool_calls | boolean | OpenAI Chat/Responses | parallelToolCalls | parallel_tool_calls | 是否允许并行工具调用。仅在非空 tools 一起出现时进入内部通道；协议客户端写入 signed body 时转为 boolean。 | 否 |

## 工具回合输入保真字段

这些字段表示“模型请求工具 -> 客户端或本地 loop 执行工具 -> 把结果继续发回”的后续回合输入。默认 compat 层只负责保留结构并交给 runner；旧显式 sendPath 可继续写入 signed body，真实 `/api/v1/chat/completion` 的原生工具字段已校准为不支持。显式 `local_executes_tools` 且注入 executor 时，LocalToolLoopRunner 会追加 OpenAI Chat 风格 `role:"tool"` / `tool_call_id` 消息进入下一轮。

## 本地工具 loop 策略字段

这些字段来自 `config.compat.localToolLoop`，只影响 `TABBIT_POOL_TOOL_LOOP_MODE=local_executes_tools` 且宿主注入 executor 的场景。默认策略仍不执行本地工具。

| 字段 | 类型 | 默认值 | 说明 |
|---|---|---:|---|
| `allowedToolNames` | string[] | `[]` | 为空表示只按请求中的工具定义作为白名单；非空时，提交的工具定义必须全部在该列表内。 |
| `maxRounds` | number | `4` | 单次请求最多允许多少轮“模型 -> 工具 -> 模型”循环。 |
| `toolTimeoutMs` | number | `0` | 单个工具执行超时，0 表示不启用超时。超时返回 `timeout/LOCAL_TOOL_TIMEOUT`。 |
| `maxToolResultChars` | number | `16000` | 工具结果写回下一轮前的最大字符数，超出会追加 `[truncated]` 标记。 |

| 字段/结构 | 来源 | 保留位置 | 说明 | 是否公开 |
|---|---|---|---|---|
| assistant `tool_calls` | OpenAI Chat `messages[]` | runner `messages[]` -> protocol `body.messages[]` | 表示上一轮 assistant 请求一个或多个 function tool call。 | 否 |
| `role:"tool"` + `tool_call_id` | OpenAI Chat `messages[]` | runner `messages[]` -> protocol `body.messages[]` | 表示客户端执行工具后的结果消息，必须引用上一轮 tool call id。 | 否 |
| `type:"function_call"` | OpenAI Responses `input[]` | runner `messages[]` -> protocol `body.messages[]` | 表示上一轮 Responses output 中的 function call item。 | 否 |
| `type:"function_call_output"` | OpenAI Responses `input[]` | runner `messages[]` -> protocol `body.messages[]` | 表示客户端执行 function call 后提交的 output。 | 否 |
| `type:"tool_use"` content block | Anthropic Messages `messages[].content[]` | runner `messages[].content[]` -> protocol `body.messages[]` | 表示上一轮 assistant 请求 Anthropic tool use。 | 否 |
| `type:"tool_result"` content block | Anthropic Messages `messages[].content[]` | runner `messages[].content[]` -> protocol `body.messages[]` | 表示客户端执行 Anthropic tool use 后提交的结果。 | 否 |

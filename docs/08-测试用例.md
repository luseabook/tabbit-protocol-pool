# 08-测试用例

测试按“可离线回归优先”的原则组织：能用 fixture 验证的逻辑不依赖真实账号；只有端到端探针需要真实本地配置。

## 单元测试

### T01：YYDS Mail 创建邮箱请求构造

- 前置：设置占位 `YYDS_MAIL_API_KEY`。
- 操作：调用 `YYDSMailProvider.createInbox()`，拦截 fetch。
- 期望：请求 `POST /v1/accounts`，使用 `X-API-Key`，body 包含 `localPart/domain/subdomain` 中的有效字段。

### T02：YYDS Mail 读取验证码

- 前置：邮件列表 fixture 包含主题/正文验证码。
- 操作：调用 `waitForVerificationCode()`。
- 期望：在 timeout 内返回验证码字符串，不把正文写入日志。

### T03：YYDS Mail 限流退避

- 前置：模拟 `429` 和 `Retry-After: 2`。
- 操作：轮询邮件。
- 期望：下一次请求延后，错误摘要包含 `rate_limited`。

### T04：Tabbit sign key 获取

- 操作：GET `/chat/sign-key` 的 HTTP client 使用 fixture 返回。
- 期望：返回非空 key，并写入缓存元数据。

### T05：Tabbit 签名头生成

- 前置：固定 body、timestamp、signature、sign key。
- 操作：生成签名头。
- 期望：真实 chat completion 分支的 `x-nonce` 等于 `HMAC-SHA256(signKey, timestamp.signature.sha256(bodyText))`，与浏览器前端同输入输出一致；旧 canonical 签名骨架仍有独立回归。

### T06：协议错误分类

- 前置：准备 login_required/quota_exhausted/forbidden/rate_limited/network_error fixture；quota fixture 使用 `QUOTA_EXHAUSTED` / `INSUFFICIENT_QUOTA` 等 code 或 quota/usage/credit 耗尽文案，即使 HTTP status 是 429/5xx 也应优先识别为账号本地额度耗尽。
- 操作：调用 `classifyProtocolError()`。
- 期望：得到稳定 category、retryable、cooldownMs。

### T07：账号池轮询

- 前置：两个 active 账号。
- 操作：连续发送 4 次 pick。
- 期望：账号选择符合 round-robin 或健康优先策略。

### T08：额度耗尽切换

- 前置：账号 A 模拟 quota_exhausted，账号 B active。
- 操作：发送请求。
- 期望：A 被标记为 `quota_exhausted` 或冷却，B 完成请求。

### T09：重置券使用

- 前置：账号 quota_exhausted 且 `resetCouponCount > 0`。
- 操作：调用 `BenefitsMaintainer.useResetCoupon()`。
- 期望：无券或状态不匹配时 skipped；使用成功后账号恢复 active、券数扣减；失败返回 failed action 且不覆盖旧状态。

### T53：ProtocolTabbitClient stream response parser

- 前置：构造带显式 `sendPath` 的 `ProtocolTabbitClient`，fake fetch 先返回 sign key，再分别返回 `Content-Type: text/event-stream` 的 SSE body 和 `Content-Type: application/x-ndjson` 的 newline-delimited JSON body。
- 操作：调用 `sendMessage({ stream:true })`。
- 期望：buffered fake response 返回 `ok:true`，`contentBlocks` 聚合为单个最终 text block；`raw.kind` 为 `stream`，`raw.format` 区分 `sse` / `ndjson`，`raw.events` 保存结构化事件，`streamDeltas` 保留可提取文本分片；不会把 `data:` frame 或 JSONL 原文当成助手文本。

### T54：协议 response.body async stream producer 与网关透传

- 前置：构造带显式 `sendPath` 的 `ProtocolTabbitClient`，fake fetch 先返回 sign key，再返回 `Content-Type: text/event-stream` 且 body 是 Web `ReadableStream`；第一帧立即写入，第二帧等待 deferred promise。
- 操作：调用 `sendMessage({ stream:true })`，再通过 OpenAICompat/AnthropicCompat 和 `createProtocolPoolGateway()` 请求 `stream:true`。
- 期望：`sendMessage()` 在第二帧释放前返回 `ok:true`，`streamDeltas` 是 async iterable；消费第一项得到第一段 delta，第二项在 deferred 释放前不会产出；OpenAI Chat/Responses 与 Anthropic Messages handler 会把 async iterable 原样放入非公开 `response.stream.deltas`，public JSON body 不包含 `stream` 或 `streamDeltas`；HTTP gateway 对真实协议 async body 不设置 `Content-Length`，并在第二帧释放前先 flush 第一个本地 SSE delta。

### T55：ProtocolTabbitClient stream error frame 分类

- 前置：构造带显式 `sendPath` 的 `ProtocolTabbitClient`，fake fetch 先返回 sign key，再分别返回 buffered SSE 错误帧和 async SSE 错误帧。错误帧使用 `event:error`，payload 为 `{"error":{"code":"QUOTA_EXHAUSTED","message":"Current account quota exhausted"}}`。
- 操作：buffered 情况调用 `sendMessage({ stream:true })`；async 情况先消费第一段正常 delta，再释放错误帧并继续消费 `streamDeltas`。
- 期望：buffered 情况返回 `{ ok:false }`，`error.category` 为 `quota_exhausted`，`error.code` 为 `QUOTA_EXHAUSTED`，且 `retryable` 为 true；async 情况第二次迭代 reject `ProtocolTabbitError`，分类同上，并且 `raw.events` 保留正常 delta 和错误 frame。

### T56：HTTP async SSE 迭代错误帧

- 前置：OpenAI Chat/Responses/Anthropic compat handler 返回 2xx 成功 JSON，且 `stream.deltas` 是 async iterable：先 yield `"Hel"`，等待 deferred promise，随后 throw 带 `code:"QUOTA_EXHAUSTED"` 的错误。
- 操作：分别请求 `/v1/chat/completions`、`/v1/responses`、`/v1/messages` 且 `stream:true`，先读取第一段正常 SSE delta，再释放错误。
- 期望：HTTP 响应保持 `200 text/event-stream` 且没有 `Content-Length`；Chat Completions 在错误后输出 `data: {"error":...}` 与 `[DONE]`；Responses 在错误后输出 `event: response.failed` 与 `[DONE]`；Anthropic 在错误后输出 `event: error`。三者都不能尝试在 headers sent 后改写 JSON，也不能让客户端无错误帧异常断连。

### T57：ProtocolTabbitClient async stream consumer cancellation

- 前置：构造带显式 `sendPath` 的 `ProtocolTabbitClient`，fake fetch 先返回 sign key，再返回 `Content-Type: text/event-stream` 且 body 是 Web `ReadableStream`。该 stream 先释放第一帧，然后等待 deferred promise，并在 underlying source 的 `cancel(reason)` 中记录取消原因。
- 操作：调用 `sendMessage({ stream:true })`，消费 async `streamDeltas` 的第一段 delta 后调用 iterator `return()`。
- 期望：iterator 正常结束，底层 `ReadableStream.cancel()` 被调用，取消原因是 `"stream_deltas_cancelled"`；不会等待第二帧，也不会继续读取上游 body。

### T58：HTTP async SSE 客户端断开取消

- 前置：OpenAI Chat compat handler 返回 2xx 成功 JSON，且 `stream.deltas` 是可取消 async iterable：先 yield `"Hel"`，第二次 `next()` 永不完成，`return()` 会记录已取消。
- 操作：请求 `/v1/chat/completions` 且 `stream:true`，读取到第一段 content delta 后中止 fetch / cancel reader，等待服务端处理 close。
- 期望：HTTP writer 收到下游 close 后停止继续写 SSE，不输出错误 JSON，不等待永不完成的下一段 delta，并请求 `stream.deltas` iterator `return()`。

### T59：ProtocolTabbitClient 附件上传与 raw/base64 自动引用

- 前置：构造带显式 `attachmentUploadPath` 的 `ProtocolTabbitClient`，fake fetch 先返回 sign key，再返回附件上传响应；另构造同时带 `attachmentUploadPath=/proxy/v0/cos/presigned-upload-url` 与 `attachmentCompleteUploadPath=/api/v0/cos/complete-upload` 的客户端；另通过 `createProtocolPoolGateway()` 配置 `TABBIT_POOL_PROTOCOL_ATTACHMENT_UPLOAD_PATH` 和 `TABBIT_POOL_PROTOCOL_ATTACHMENT_COMPLETE_UPLOAD_PATH`，账号只保存 `cookieJarRef`。
- 操作：调用 `uploadAttachment({ account, attachment })`，调用 `sendMessage({ attachments:[raw/base64] })`，并通过 gateway 的 OpenAI Chat 入口提交 raw 附件。
- 期望：旧上传骨架请求 `baseUrl + attachmentUploadPath`，method 为 `POST`，body 为 `{ attachment }`，签名 payload 使用现有 canonical JSON 规则，headers 包含 `Content-Type` 和 hydrated `Cookie`；真实 COS 链按 presign -> PUT -> complete 顺序执行，两个 Tabbit POST 不生成 `x-signature` / `x-nonce`，响应归一化为 `attachment.id/name/mimeType/size/url?`。真实 `/api/v1/chat/completion` 分支中，已上传附件引用和 raw/base64 自动上传结果都会进入 `references[].metadata.file_id`；未完整配置上传链时，缺少 file id/path 的 raw/base64 附件返回 `unsupported_feature/ATTACHMENT_REFERENCE_REQUIRED`，不自动降级为纯文本发送。

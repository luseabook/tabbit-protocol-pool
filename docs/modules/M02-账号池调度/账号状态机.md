# 账号状态机

## 状态图

~~~text
provisioning ──成功──> active
      │                 │
      └─失败──────────> suspect/disabled

active ──额度耗尽────> quota_exhausted
active ──登录失效────> login_expired
active ──限流/上游错误→ cooldown
active ──协议异常────> suspect

cooldown ──到期探测成功──> active
quota_exhausted ──额度恢复/重置券──> active
login_expired ──刷新登录成功──────> active
suspect ──人工确认/探针成功──────> active
~~~

## 状态定义

| 状态 | 能否处理请求 | 进入原因 | 恢复方式 |
|---|---:|---|---|
| provisioning | 否 | 注册或导入中 | 初始化成功 |
| active | 是 | 可正常调用 | 无 |
| cooldown | 否 | 429、网络波动、短暂上游错误 | 到期后探测 |
| quota_exhausted | 否 | 额度不足 | 额度刷新或重置券 |
| login_expired | 否 | cookie/session 失效 | 重新登录或刷新 session |
| suspect | 否 | 风控、协议变更、不确定错误 | 人工确认或探针成功 |
| disabled | 否 | 人工禁用或不可恢复失败 | 人工启用 |

## 状态写入规则

- 每次状态变更都记录 `reason`、`requestId`、`observedAt`。
- 降级状态比升级状态优先，例如同一次请求同时出现 403 和解析失败，应先标为 `suspect`。
- 不能因为一次网络超时直接 disabled。
- 不能把 `quota_exhausted` 自动改为 `active`，除非额度查询或重置券动作成功。

## 测试

- active → quota_exhausted。
- active → cooldown → active。
- active → login_expired。
- suspect 需要人工或探针恢复。

# 账号元数据持久化

本文件记录 M02 当前已实现的 JSON 账号元数据 store。它解决的问题是：账号池的健康状态、失败审计和额度摘要需要跨请求保留，但 raw cookie、session、token、password 不能写进仓库或普通 JSON。

## 定位

~~~text
PooledRequestRunner
  ↓ recordSuccess / recordFailure
StoredAccountPool
  ↓ saveAccounts()
JsonAccountStore
  ↓ atomic write
<state-dir>/accounts.json
~~~

- `AccountPool` 仍是内存状态机。
- `StoredAccountPool` 在内存状态变更后写回 store。
- `JsonAccountStore` 只保存账号元数据与 secret 引用。
- cookie/session/token 原文由后续 secret store 或系统密钥链保存，账号 JSON 只保留 `cookieJarRef`。

## 文件位置

| 输入 | 结果 |
|---|---|
| `{ filePath }` | 使用显式文件路径。 |
| `{ stateDir }` | 使用 `<stateDir>/accounts.json`。 |
| 两者都缺失 | 抛出 `AccountStoreError(MISSING_ACCOUNT_STORE_PATH)`。 |

默认推荐位置来自 `loadConfig().stateDir`，不要放在 Git 仓库目录内。

## JSON 结构

当前版本写入 versioned document：

~~~json
{
  "version": 1,
  "updatedAt": "2026-07-02T00:00:00.000Z",
  "accounts": [
    {
      "id": "acct_a",
      "email": "aa@example.com",
      "status": "active",
      "accessTier": "unknown",
      "cookieJarRef": "secrets/acct_a.cookie",
      "quotaState": [],
      "resetCouponCount": 0,
      "failureStreak": 0,
      "audit": []
    }
  ]
}
~~~

兼容读取 legacy account array，但写回时统一写为上述 document。`loadAccounts()` 在文件不存在时返回空数组。

## Secret 字段规则

`sanitizeAccountForStorage(account)` 会剥离以下直接 secret 字段：

- `apiKey`
- `authorization`
- `cookie`
- `cookieHeader`
- `password`
- `secret`
- `session`
- `sessionToken`
- `token`

`cookieJarRef` 会保留，因为它是本地 secret 存储引用，不是 cookie 原文。引用本身也不要设计成可还原 secret 的字符串。

## 写入语义

- `saveAccounts(accounts)` 先归一化并脱敏账号，再写入临时文件，最后 rename 到目标路径。
- `updateAccounts(mutator)` 会读取当前账号，传给 mutator，要求 mutator 返回新的账号数组。
- `StoredAccountPool.recordSuccess()` 和 `recordFailure()` 都会先执行内存状态机，再调用 `persist()` 写回。
- `PooledRequestRunner.run()` 已等待 `recordSuccess()` / `recordFailure()` 的 Promise，因此 HTTP handler 返回前状态已经完成持久化。

## 状态变更示例

| 事件 | 写回效果 |
|---|---|
| 请求成功 | `status=active`，`failureStreak=0`，清空 `cooldownUntil` 和 `lastError`，追加 success audit。 |
| `quota_exhausted` | `status=quota_exhausted`，`failureStreak+1`，写入脱敏 `lastError`，追加 failure audit。 |
| `login_required` | `status=login_expired`，停止继续选择该账号。 |
| `rate_limited` / `network_error` / `upstream_error` | `status=cooldown`，设置 `cooldownUntil`。 |
| `protocol_changed` / `forbidden` | `status=suspect`，等待人工或探针确认。 |

## How to：初始化本地账号元数据

1. 选择一个仓库外状态目录，例如：

   ~~~powershell
   $env:TABBIT_POOL_STATE_DIR="E:\tmp\tabbit-protocol-pool-state"
   ~~~

2. 写入只含元数据的 `accounts.json`。不要写 cookie 原文：

   ~~~json
   {
     "version": 1,
     "updatedAt": "2026-07-02T00:00:00.000Z",
     "accounts": [
       {
         "id": "acct_a",
         "status": "active",
         "accessTier": "unknown",
         "cookieJarRef": "secrets/acct_a.cookie"
       }
     ]
   }
   ~~~

3. 用当前实现验证归一化结果：

   ~~~powershell
   node --input-type=module -e "import { JsonAccountStore } from './src/index.js'; const store = new JsonAccountStore({ stateDir: process.env.TABBIT_POOL_STATE_DIR }); console.log(JSON.stringify(await store.loadAccounts(), null, 2));"
   ~~~

4. 看到 `status`、`quotaState`、`failureStreak`、`audit` 等默认字段后，说明元数据 store 可被账号池加载。

## Troubleshooting

| 现象 | 原因 | 处理 |
|---|---|---|
| `INVALID_ACCOUNT_STORE_JSON` | `accounts.json` 不是合法 JSON。 | 先备份文件，再用 JSON formatter 修复语法。 |
| `INVALID_ACCOUNT_STORE_SHAPE` | 文档不是账号数组，也没有 `accounts` 数组。 | 改成 `{ "version": 1, "accounts": [] }` 结构。 |
| 保存后 cookie 字段消失 | 这是预期行为，直接 secret 字段会被剥离。 | 把 raw secret 放入后续 secret store，只在账号元数据中保留引用。 |
| 请求结束后状态未写回 | runner 未等待异步持久化或未使用 `StoredAccountPool`。 | 使用 `StoredAccountPool.load({ store })`，并通过 `PooledRequestRunner` 调用。 |

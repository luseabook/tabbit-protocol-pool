# 失败分类与冷却

## 目标

把协议客户端返回的错误转换为账号状态更新和请求 fallback 决策。

## 分类表

| ProtocolError.category | retryable | 账号状态动作 | 默认冷却 |
|---|---:|---|---|
| login_required | 否 | `login_expired` | 无 |
| quota_exhausted | 是 | `quota_exhausted` | 到额度刷新 |
| forbidden | 视情况 | `suspect` 或 `cooldown` | 30 min |
| rate_limited | 是 | `cooldown` | `Retry-After` 或 60s |
| upstream_error | 是 | `cooldown` | 10s-60s |
| network_error | 是 | 增加 failureStreak | 指数退避 |
| protocol_changed | 否 | `suspect` | 无 |
| unknown | 视情况 | `cooldown` | 60s |

## fallback 规则

- 只有 `retryable: true` 且仍有候选账号时，才切换账号。
- 同一请求的账号 fallback 次数应配置上限，默认 1。
- `protocol_changed` 不做账号轮询，因为所有账号可能都会失败。
- `login_required` 不重试同账号。

## 冷却计算

优先级：

1. 上游 `Retry-After`。
2. ProtocolError.cooldownMs。
3. 分类默认值。
4. failureStreak 指数退避上限。

## 测试

- 429 + Retry-After 转 cooldown。
- quota_exhausted 触发 fallback。
- protocol_changed 阻止 fallback。
- network_error 连续失败增加退避。

# M02-账号池调度

## 定位

AccountPool 负责在多账号之间选择可用账号，并根据协议客户端返回的结果更新账号状态。它是用户“一个号额度太少，想组号池轮询”的核心实现。

## 公开能力

| 能力 | 方法草案 | 说明 |
|---|---|---|
| 加载账号 | `JsonAccountStore.loadAccounts()` / `StoredAccountPool.load()` | 从本地 JSON 元数据 store 读取账号状态 |
| 选择账号 | `pickAccount(input)` | 根据模型、权益、健康度选择账号 |
| 记录成功 | `recordSuccess(input)` | 清理失败 streak，刷新最近成功时间 |
| 记录失败 | `recordFailure(input)` | 按错误分类更新状态 |
| 探测恢复 | `probeAccount(account)` | 冷却结束或维护后确认是否恢复 |

## 调度策略

默认策略建议：

1. 排除 disabled/provisioning/login_expired。
2. 排除 cooldown 未到期账号。
3. 按请求模型过滤权益不足账号。
4. 在可用账号中按 health score + round-robin 选择。
5. 单请求失败后只尝试有限 fallback，避免无限消耗账号。

## 完成标准

- 两个 active 账号能稳定轮询。
- 额度耗尽账号不会继续被选中。
- 登录失效账号不会触发重复失败。
- 所有状态变更都有可审计原因。

## 当前实现状态

- 已实现内存版 `AccountPool`，支持 `pickAccount()`、`recordSuccess()`、`recordFailure()`、`shouldFallback()`。
- 已实现状态归一化、候选解释、round-robin、premium 过滤、冷却过滤、失败审计。
- 已实现 `JsonAccountStore` 账号元数据读写，文件为 versioned JSON。
- 已实现 `StoredAccountPool`，可从 store 加载账号，并在 `recordSuccess()` / `recordFailure()` 后写回状态。
- `PooledRequestRunner` 已等待异步状态记录完成，便于持久化 store 参与请求闭环。
- raw cookie/session/token 不直接写入账号 JSON；只保留 `cookieJarRef` 等存储引用。

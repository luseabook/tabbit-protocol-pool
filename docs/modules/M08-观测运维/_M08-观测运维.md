# M08-观测运维

## 定位

观测运维模块帮助用户判断账号池是否健康、哪个账号失败、额度是否耗尽、协议是否变更。当前已实现离线可测的 `src/observability.js` 基础层，已默认接入 protocol-pool gateway 的 `/health`，并通过 `tabbit-pool` 提供本地运维 CLI foundation。

CLI foundation 的边界很明确：账号列表和健康检查只读取本地账号元数据；未配置协议 path 时，`maintain` 使用空协议客户端，只输出 skipped actions，不会触发真实 Tabbit 网络；配置 `TABBIT_POOL_PROTOCOL_QUOTA_USAGE_PATH=/api/commerce/quota/v1/usage` 后，`maintain` 会把 `refreshQuota` 接到已校准的真实额度 usage 查询；配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_PATH=/api/commerce/activity/v1/sign-in` 后，`maintain` 会把 `dailyCheckin` 接到已验证每日签到，若配置 status path 会先查 `signedToday`。配置 activity/newbie/placement/reward/lottery 只读 path 或其它 M05 side-effect probe path 只会启用对应 `probe protocol` operation 和 gateway protocol client factory，不会让 `maintain` 自动领取 Pro、抽奖或用券；POST 型 side-effect operation 还要求 `confirmSideEffect:true`。

`fixtures list/show/audit` 只读取本地脱敏 protocol probe fixture，`audit` 只输出覆盖计数、缺口和下一步建议；`readiness doctor` 会把当前 `stateDir`、协议 env 覆盖、readiness、默认 fixture audit、core 剩余工作、auth/benefits/session/upstream `calibrationBacklog` 和安全命令汇总成只读诊断报告；`probe advice` 根据错误分类/status/code/message 生成建议，并能细分 403 forbidden；`probe template` 只输出安全 JSON payload 模板，包含离线 `recoverSession` 和 `consumeResetCoupon` evidence 模板；`probe validate` 只读校验 input schema 和字段形状，`--require-confirmed-side-effect` 可在真实副作用捕获前离线要求 `confirmSideEffect:true`，`recoverSession` / `consumeResetCoupon` 只校验显式脱敏 evidence，也可通过 `--write-fixture` 把校验后的白名单离线 evidence 写入本地 fixture store。`probe protocol` 已提供协议探针 fixture foundation，并支持 `--input-json` / `--input-file` 给 `sendMessage`、`listModels`、`refreshQuota`、只读 commerce 查询和 M05 side-effect probe operation 传入探针 payload；CLI 会先校验已稳定的 operation 字段，默认不触发真实 Tabbit 网络；`recoverSession` 和 `consumeResetCoupon` 会被拒绝派发。`accounts import-session` 可把已登录账号的 cookie/session 写入本地 secret store 并创建 active 账号元数据；`accounts probe <id>` 已可复用 `AccountProvisioner.verifyAccount()` 检查本地 session 状态；`readiness mark` 可把 Codex / Claude Code 人工端到端验收结果写入本地 `readiness.json`；`serve/start` 会启动本地 protocol-pool OpenAI/Anthropic 兼容 gateway 并等待 SIGINT/SIGTERM 关闭；`smoke gateway` 会对本地 gateway 做 OpenAI/Anthropic 兼容路由 smoke 验收。

`ProtocolTabbitClient.verifySession()` 已校准 `GET /api/v0/user/base-info`，`ProtocolTabbitClient.refreshQuota()` 已校准 `GET /api/commerce/quota/v1/usage?user_id=...`，只读 commerce 状态/资源探针已校准五条 `/api/commerce` GET，M05 side-effect probe 已校准每日签到状态/POST、权益券列表、activity participate、usage reset coupon sku、抽奖机会/奖池/记录/POST。只有设置 `TABBIT_POOL_PROTOCOL_ENABLED=true` 或显式 `TABBIT_POOL_PROTOCOL_*_PATH` 后，默认 CLI dependencies 才会创建真实 `ProtocolTabbitClient`；当前已校准的真实文本发送路径是 `/api/v1/chat/completion`，模型目录默认 `scene=chat`。

## CLI 命令参考

### tabbit-pool accounts list [--json]

读取 `JsonAccountStore.loadAccounts()`，输出脱敏账号列表。

- `--json`：输出 `{ "accounts": [...] }`。
- 无 `--json`：输出 tab 分隔表格：`id`、`status`、`accessTier`、`email`、`quota`、`lastError`。
- 输出不会包含 `cookie`、`cookieHeader`、`cookieJar`、`cookieJarRef`、`session`、`sessionToken`、`token`。

示例：

~~~powershell
node bin/tabbit-pool.js accounts list --json
~~~

### tabbit-pool accounts import-session [--id <id>] [--email <email>] [--cookie-header <text> | --session <text> | --cookie-file <path> | --session-file <path>] [--json]

把本机已登录 Tabbit 的 cookie/session 导入账号池。命令不会请求 Tabbit 网络；它只调用 `AccountProvisioner.importSession()`，把原始 cookie/session 写入 `FileSecretStore`，并在账号元数据中保存 `cookieJarRef`。

建议优先使用文件输入，避免 shell history 保存 cookie：

~~~powershell
node bin/tabbit-pool.js accounts import-session --id acct_default --email user@example.test --access-tier pro --cookie-file .\tabbit-cookie.txt --json
~~~

约束：

- 四种 session 来源 `--cookie-header`、`--session`、`--cookie-file`、`--session-file` 必须且只能选一个。
- stdout 只返回脱敏账号和 action events，不输出 cookie、session、token 或 `cookieJarRef`。
- `--id` 推荐显式传入，省略时会生成 `acct_<uuid>`。

### tabbit-pool accounts probe <id> [--json]

调用 `AccountProvisioner.verifyAccount(accountId)` 验证单个账号的本地 session 状态，并输出脱敏 action log 与下一步建议。

- 可通过 `options.accountVerifier` 注入真实或测试 verifier。
- 默认 verifier 使用 `JsonAccountStore` + `FileSecretStore`；未配置协议 env 时使用空 `protocolClient`，配置 `TABBIT_POOL_PROTOCOL_SESSION_VERIFY_PATH` 后使用 `ProtocolTabbitClient.verifySession()`。
- 如果账号缺少本地 secret，默认 verifier 会把账号标记为 `login_expired`。
- 如果本地 secret 存在但未配置 `protocolClient.verifySession` 且未通过 env 启用协议客户端，默认 verifier 返回 skipped，不触发真实网络。
- JSON 输出为 `{ changed, account, events, advice }`，其中 `account` 和 `events` 都已脱敏。

示例：

~~~powershell
node bin/tabbit-pool.js accounts probe acct_default --json
~~~

      "suspect": 0,
      "unknown": 0
    }
  },
  "alerts": []

~~~


### tabbit-pool readiness [--json]

读取本地账号元数据和 protocol probe fixture，输出真实协议校准 readiness。命令会先 `listFixtures()`，再对带 `ref` 的摘要调用 `readFixture(ref)` 读取脱敏详情，以便识别 `result.error` 下的工具不支持和 403 证据。它不触发 Tabbit 网络，只判断当前还缺什么：

- `protocolCalibration`：协议开关、sendPath、sessionVerifyPath、active 账号、成功 verifySession fixture、成功 sendMessage fixture。
- `codexClaudeE2E`：Codex 与 Claude Code 人工端到端验收是否完成。
- `toolLoopDecision`：来自 `config.compat.toolLoopMode`，默认决策为 `client_executes_tools_first`；也可显式为 `disabled` 或 `local_executes_tools`。该检查只展示策略，不会触发真实网络或执行本地工具；`local_executes_tools` 仍要求 gateway 宿主注入 executor。
- `forbidden403`：是否已有 403/forbidden 脱敏 fixture。

示例：

~~~powershell
node bin/tabbit-pool.js readiness --json
~~~

### tabbit-pool readiness doctor [--json]

读取同一组本地状态，并额外组合 `fixtures audit` 结果，输出操作者可直接判断的诊断报告。该命令只读，不会调用 `accounts probe`、不会调用 `probe protocol`、不会写 `readiness.json`，也不会读取或输出 raw cookie/session。

JSON 输出包含：

- `status`：`ready`、`partial` 或 `blocked`；只有 readiness 和 fixture audit 都 ready 时才是 `ready`。
- `stateDir`：当前 CLI 使用的本地状态目录。
- `protocol`：只包含 `enabled`、`baseUrlConfigured`、`sendPathConfigured`、`sessionVerifyPathConfigured`、`authSendCodePathConfigured`、`authSubmitCodePathConfigured`、`compatStripClientTools`、`toolLoopMode`，不输出 auth endpoint path、`REQ_CTX` 或签名上下文。
- `readiness`：`buildCalibrationReadinessSnapshot()` 聚合结果。
- `fixtureAudit`：`buildProtocolFixtureAudit()` 聚合结果。
- `calibrationBacklog`：auth、benefits、session 与 upstream scope 的扩展审计；不改变 top-level `status`，用于显示注册/登录、M05 副作用、session lifecycle 和真实上游 stream boundary evidence 是否仍缺失。`captureCommands` 只输出带 `<account-id>` / `<redacted-input.json>` 占位符的安全 template/validate/confirm_validate/probe 命令提示；auth、benefits、session 与 upstream capture command 还会输出 `prerequisites` 与 `prerequisitesStatus`，只包含 env 名和 `configured` / `missing`，不包含 endpoint 值、body、cookie、session、token 或 raw payload。当前 prerequisites 覆盖 auth send/submit、daily sign-in、benefit coupon use、activity participate、lottery draw、session verify path 和 upstream send path。`validateCommand` 只做本地 schema 与形状预检，不触网也不证明真实协议已闭环。`confirmedValidateCommand` 只在已校准 side-effect operation 上有值，对应 plain `confirm_validate=`，用于在同一 input file 上只读确认 `confirmSideEffect:true`。真实上游 error-frame、cancellation 和 backpressure 缺口会给出 `sendMessage` 占位命令；没有已校准 operation 的缺口会返回 `validateCommand:null`、`confirmedValidateCommand:null`、`probeCommand:null` 和原因。
- `remainingWork`：readiness 与默认 fixture audit 的 `nextActions` 去重合并，只代表 core gateway/chat gate。
- `commands`：安全 PowerShell 命令示例，用于设置 `TABBIT_POOL_STATE_DIR`、重新运行 readiness/audit、auth/benefits/session/upstream scoped audit 和启动 gateway。

当 `status:"ready"` 且 `remainingWork:[]` 时，基础 gateway/chat readiness 已满足；如果 `calibrationBacklog.status:"blocked"`，仍需继续捕获注册/登录、M05 副作用、session lifecycle 或真实上游 stream boundary 的安全 evidence。非 JSON 输出同样会打印 `auth_send_endpoint`、`auth_submit_endpoint`、`calibration_backlog`、`auth_backlog`、`benefits_backlog`、`session_backlog` 和 `upstream_backlog` 的状态与 missing 计数，避免 plain 巡检只看到 core ready；`capture_command` 行会带 `prereq=ENV:configured|missing`，无前置条件时该字段为空。auth endpoint 和 prereq 的 `configured` / `missing` 只用于区分 env wiring 状态；即使相关 path 都为 `configured`，也不代表 endpoint、请求 body、副作用安全或成功响应语义已闭环，auth/benefits/session scope 仍必须由对应 sanitized fixture gate 决定。真实用券消耗缺口会打印 `useResetCoupon` template/validate/confirm_validate/probe 命令，并要求 `TABBIT_POOL_PROTOCOL_BENEFIT_COUPON_USE_PATH` 前置配置；运行前仍必须选择安全券码并确认 `confirmSideEffect:true`。session scope 当前只记录 `manual_reimport_then_probe`，自动刷新路径仍未校准，并通过 `automated_session_refresh_strategy` 缺口保持 blocked；doctor 只会为该缺口打印离线 `recoverSession` template/validate 命令且 `probe` 为空，不代表存在可触网刷新 endpoint。upstream capture command 只是帮助生成真实 `sendMessage` 探针路径，upstream scope 仍只统计带真实上游 evidence marker 的脱敏 `sendMessage` fixture，本地 HTTP/compat/fake stream 测试不计为真实上游 evidence。

对离线 `recoverSession` 缺口，JSON doctor 会额外返回 `writeFixtureCommand`，plain `capture_command` 会打印 `write_fixture=`。该命令只调用 `probe validate --write-fixture` 写入已校验的脱敏 evidence，不读取账号、secret、fixture body，不运行网络 probe，也不会替代 auth、每日签到、活动 Pro、抽奖、verifySession 或 upstream sendMessage 的真实校准流程。离线 `consumeResetCoupon` 模板仍可人工使用，但 doctor 的真实用券缺口已优先引导 `useResetCoupon` 协议捕获。

示例：

~~~powershell
node bin/tabbit-pool.js readiness doctor --json
~~~

该命令适合处理“默认状态目录 blocked，但外部脱敏 evidence state 已存在”的场景：

~~~powershell
$env:TABBIT_POOL_STATE_DIR = "E:\tabbit2api\output\tabbit-live-state"
node bin\tabbit-pool.js readiness doctor --json
~~~

### tabbit-pool readiness mark [--codex-verified] [--claude-verified] [--json]

把人工端到端验收结果写入本地 `readiness.json`。该命令不读取账号、不请求 Tabbit，只更新本地状态文件。

- `--codex-verified`：标记 Codex 已用本地 OpenAI Responses base URL 完成一次真实代码任务。
- `--claude-verified`：标记 Claude Code 已用本地 Anthropic base URL 完成一次真实代码任务。
- 两个 flag 可同时传入；每个被标记项都会写入 `verified:true` 和当前 `verifiedAt`。
- 至少需要一个 verification flag；否则 exitCode=2，且不会写文件。
- 输出不会包含 prompt、代码 diff、API key、cookie、session 或 token。

示例：

~~~powershell
node bin/tabbit-pool.js readiness mark --codex-verified --json
node bin/tabbit-pool.js readiness mark --claude-verified --json
~~~

### tabbit-pool serve/start [--host <host>] [--port <port>] [--json]

启动 protocol-pool 本地 OpenAI / Anthropic 兼容 gateway。`serve` 和 `start` 是等价入口；命令会调用 `createProtocolPoolGateway()`，再执行 `gateway.start({ host, port })`，直到收到 Ctrl+C / SIGTERM 后调用 `gateway.close()`。

- 默认 host/port 来自 `TABBIT_POOL_HOST` / `TABBIT_POOL_PORT`，未配置时是 `127.0.0.1:50124`。
- `--host` / `--port` 只覆盖本次启动；`--port` 必须是 1 到 65535 的整数，非法值返回 exitCode 2，且不会创建 gateway。
- `--json` 输出 `{ status, host, port, openaiBaseUrl, anthropicBaseUrl }`。输出不会包含 `TABBIT_POOL_API_KEY`、cookie、session 或 token。
- `0.0.0.0` / `::` 绑定时，展示用 base URL 会写成 `127.0.0.1`，避免把示例误当成公网发布建议。

示例：

~~~powershell
node bin/tabbit-pool.js serve --host 127.0.0.1 --port 50124
node bin/tabbit-pool.js start --json
~~~

JSON 输出示例：

~~~json
{
  "status": "listening",
  "host": "127.0.0.1",
  "port": 50124,
  "openaiBaseUrl": "http://127.0.0.1:50124/v1",
  "anthropicBaseUrl": "http://127.0.0.1:50124"
}
~~~

### tabbit-pool smoke gateway [--base-url <url>] [--api-key <key>] [--model <model>] [--json]

对已启动的本地 gateway 做兼容路由 smoke 验收。默认参数来自本地配置：`base-url=http://127.0.0.1:50124`、`api-key=TABBIT_POOL_API_KEY || sk-tabbit-local`、`model=tabbit/priority`。

检查顺序：

1. `GET /health`，不带认证。
2. `GET /v1/models`，使用 OpenAI `Authorization: Bearer ...`。
3. `POST /v1/chat/completions`，发送最小 OpenAI Chat 请求。
4. `POST /v1/responses`，发送最小 OpenAI Responses 请求。
5. `POST /v1/messages`，发送最小 Anthropic Messages 请求。

如果某一步失败，命令停止后续步骤，JSON 输出 `status:"failed"` 和 `failedStep`，exitCode=1。输出包含 URL、HTTP status 和 step 名称，但不会输出 API key、cookie、session 或 token。

注意：后三个 POST 会让 gateway 调用当前配置的上游协议客户端；当 gateway 已连接真实 Tabbit 时，这就是一次真实最小消息请求。

示例：

~~~powershell
node bin/tabbit-pool.js smoke gateway --json
node bin/tabbit-pool.js smoke gateway --base-url http://127.0.0.1:50124 --model tabbit/priority --json
~~~

### tabbit-pool maintain [--json]

逐个账号调用 `BenefitsMaintainer.maintainAccount(account)`，把 actions 转换为脱敏 action log。只要任一账号返回 `changed:true`，就保存更新后的账号数组。

未配置协议 path 时，默认 maintainer 使用空 `protocolClient`，所以四个动作都会安全跳过：

1. `refreshQuota`
2. `claimProIfAvailable`
3. `dailyCheckin`
4. `useResetCoupon`

这让 CLI 可以先上线排障和日志格式，不会在默认维护链里误调用未知 Tabbit endpoint。配置 `TABBIT_POOL_PROTOCOL_QUOTA_USAGE_PATH=/api/commerce/quota/v1/usage` 后，默认 maintainer 会读取账号 `cookieJarRef` 指向的本地 secret，并对有 `userId` 的账号执行真实 `refreshQuota`；成功时写回 `quotaState`、`accessTier`、`resetCouponCount` 和 `lastMaintainedAt`。配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_PATH=/api/commerce/activity/v1/sign-in` 后，默认 maintainer 会读取同一 secret 并执行已验证 `dailyCheckin`；若同时配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_STATUS_PATH`，会先查状态，`signedToday:true` 时不再 POST。即使配置了其它 M05 side-effect probe path，`claimProIfAvailable` 和 `useResetCoupon` 也仍返回 skipped；后续接入更多真实维护时，应先补充安全 evidence、明确成功/失败语义，再通过明确的协议方法和 fixture 回归扩展。

### tabbit-pool fixtures audit [--scope protocol|auth|benefits|session|upstream] [--json]

离线审计本地 protocol probe fixture 覆盖。命令先调用 `protocolFixtureStore.listFixtures()`，再对带 `ref` 的摘要调用 `readFixture(ref)` 读取脱敏详情，以便识别流式和工具调用证据；输出仍只包含聚合结果，不展示 raw request、raw response、邮箱、cookie、session、token 或验证码。

审计项：

- `successful_verifySession_fixture`：至少一个 `operation:"verifySession"` 且 `status:"success"` 的 fixture。
- `successful_sendMessage_fixture`：至少一个 `operation:"sendMessage"` 且 `status:"success"` 的 fixture。
- `streaming_text_fixture`：至少一个成功 sendMessage fixture 保留 `streamDeltas`、`raw.kind:"stream"`、`kind:"stream"`、`format:"sse"` 或 `format:"ndjson"`。
- `tool_call_fixture`：至少一个成功 sendMessage fixture 含 `tool_use`、`tool_call`、`function_call` 或 `tool_calls`，或一个带 `tools` 输入且明确显示上游不支持原生工具字段的 fixture。
- `forbidden_403_fixture`：至少一个 `adviceCategory:"forbidden"`、`error.status:403`、`result.error.status:403`、`httpStatus:403` 或 `statusCode:403` 的 fixture。

JSON 输出包含 `status`、`counts`、`coverage`、`missing`、`nextActions`。非 JSON 输出为 tab 分隔摘要。

扩展 scope：

- `--scope auth`：只读检查成功 `sendVerificationCode` 且响应形状含 send/delivery 专属成功字段的 fixture，以及成功 `submitRegistrationOrLogin` 且响应形状含可导入 session material 字段的 fixture；send-code 和 submit transport success 都会单独计数，但不直接让 coverage ready。plain 输出会同时打印 transport success、delivery success、session material success 和 `missing` 名称。
- `--scope benefits`：只读检查成功 `dailySignIn`、活动 Pro 成功、真实重置券消耗和抽奖成功 fixture；每日签到接受 `signInResult:"success"`、`signedToday:true` 或 `signInResult:"already_signed"`，其中 `already_signed` 只表示当天签到状态已完成；真实重置券消耗必须来自消费类 operation，并同时具备脱敏 `endpointHash/bodyHash/resultHash`、`safe:true`、`sanitized:true`、`rawPayload:false` evidence 和真实消费成功信号；`participateResetCouponActivity` 只作为活动参与 evidence，永不满足真实用券消耗 coverage；`drawLottery` 泛 `ok/status/result:"success"` 不计为抽奖成功，必须有 draw/lottery 专属成功或非空奖品/命中记录；读取和 `total/success/failed` 计数只覆盖 M05 side-effect 白名单 operation。plain 输出会同时打印 operation 计数、严格成功计数和 `missing` 名称。
- `--scope session`：只读检查成功 `verifySession`、上游 401/login_required 过期 fixture 和显式脱敏自动恢复策略 evidence；本地 `session_missing` 单独计数，不计为上游过期，并输出 `lifecycle` 与 `recoveryStrategy`。在刷新或安全 re-auth recovery path 未校准前，`automated_session_refresh_strategy` 会让该 scope 保持 blocked；只有 `session_recovery_strategy` / `recoverSession` evidence 同时声明 `safe:true`、`sanitized:true`、`rawPayload:false` 和已校准自动 re-auth/refresh 模式时，恢复策略才会 ready。非 JSON 输出同样打印 `recovery_strategy` 与 `missing` 行，避免只看表格时误判自动恢复已闭环。
- `--scope upstream`：只读检查同时带真实上游 evidence marker 与 stream/SSE/NDJSON 元数据的 `sendMessage` stream boundary fixture，覆盖 `real_upstream_error_frame_fixture`、`real_upstream_cancellation_fixture` 和 `real_upstream_backpressure_fixture`。非流式 `protocol_probe` sendMessage 只可满足默认 send readiness；本地 HTTP route、compat handler、unit/fake stream fixture 不计为真实上游 evidence；plain 输出会打印 `real_upstream`、`upstream_error_frame`、`upstream_cancellation`、`upstream_backpressure` 和 `missing` 名称。

示例：

~~~powershell
node bin/tabbit-pool.js fixtures audit --json
~~~

### tabbit-pool probe protocol --account <id> [--operation <name>] [--input-json <json> | --input-file <path>] [--write-fixture] [--json]

运行协议探针 fixture foundation。命令读取账号元数据和本地 secret，然后调用注入或显式 env 配置的 protocol client operation。默认依赖没有 protocol client factory，所以不会猜测未知 Tabbit endpoint，也不会触发真实网络；设置 `TABBIT_POOL_PROTOCOL_ENABLED=true` 或任一 `TABBIT_POOL_PROTOCOL_*_PATH` 后，默认 dependencies 会创建 `ProtocolTabbitClient`。

- `--account <id>`：必填，指定账号 ID。
- `--operation <name>`：可选，默认 `verifySession`；当前 runner 支持 `verifySession`、`sendMessage`、`listModels`、`refreshQuota`、`uploadAttachment`、`getLotteryExplorationMe`、`getNewbieExplorationMe`、`getPlacementResources`、`listRewardCardRecords`、`listLotteryHitRecords`、`getDailySignInStatus`、`dailySignIn`、`listBenefitCoupons`、`useResetCoupon`、`participateResetCouponActivity`、`participateActivity`、`getUsageResetCouponSku`、`getAvailableLotteryChanceCount`、`getActiveMainPools`、`listLotteryChanceRecords`、`drawLottery`；`recoverSession` 和 `consumeResetCoupon` 只允许 `probe template` / `probe validate` 离线使用，不能通过该命令派发。
- `--input-json <json>`：可选，传入 JSON object 作为 `ProtocolProbeRunner.probeAccount({ input })`；适合短 sendMessage/listModels/refreshQuota/uploadAttachment/只读 commerce 查询和 M05 side-effect probe 探针。
- `--input-file <path>`：可选，从 UTF-8 JSON 文件读取 input object；适合较长 messages payload。
- `--write-fixture`：可选，把脱敏 fixture 写入 `stateDir/fixtures/protocol-probes/`。
- `--json`：输出 `{ status, account, fixture, advice, fixtureRef? }`。

输入边界：`--input-json` 和 `--input-file` 只能二选一；输入必须是 JSON object，不能是数组、字符串或 `null`。CLI 会在调用 `ProtocolProbeRunner` 前校验稳定字段：`sendMessage.model` 如果出现必须是非空字符串；`sendMessage.messages` 如果出现必须是非空数组；`sendMessage.stream` 如果出现必须是 boolean，`probe validate` 只输出该布尔摘要，不输出消息内容；`listModels.force` 如果出现必须是 boolean；`refreshQuota.userId` 如果出现必须是非空字符串；`getNewbieExplorationMe.viewMode` 如果出现必须是 `event_gate`、`float_collapsed`、`float_expanded` 或 `activity_page`，`includeCompletions` / `includeRewards` 必须是 boolean；`getPlacementResources.placementCode` 和 `clientVersion` 如果出现必须是非空字符串；记录查询和权益券查询的 `userId` 必须是非空字符串，`offset` / `limit` 必须是非负整数；side-effect probe 的 `confirmSideEffect` 如果出现必须是 boolean，`sceneCodes` 必须是非空字符串数组，`requestNo` 必须是非空字符串且不超过 64 字符，`activityId` 必须是非空字符串，`body` 必须是 object；`useResetCoupon` 额外校验可选 `userId`、`couponCode`、`couponType` 为非空字符串；`uploadAttachment.attachment` 如果出现必须是 object，且 `filename`、`mimeType`、`data` 如果出现必须是非空字符串；`recoverSession` 必须提供显式 `session_recovery_strategy` evidence，要求 `operation:"recoverSession"`、`status:"success"`、`safe:true`、`sanitized:true`、`rawPayload:false` 和已校准的 re-auth/refresh mode；`consumeResetCoupon` 必须提供显式 `reset_coupon_consumption_evidence`，要求消费类 operation、`status:"success"`、`endpointHash/bodyHash/resultHash` 三个 `sha256:` 脱敏哈希、`safe:true`、`sanitized:true`、`rawPayload:false` 和真实消费成功信号。未知字段会保留给真实协议校准使用。`probe validate --require-confirmed-side-effect` 会在不读取账号、secret 或 fixture 的前提下，对已知副作用 operation 额外要求 `confirmSideEffect:true`。`probe validate --write-fixture` 只支持 `recoverSession` 和 `consumeResetCoupon`，写入前复用同一校验，写入时只保留白名单 evidence/result 字段并丢弃 input 中的额外 raw payload；对 `sendMessage`、auth、M05 side-effect 等非离线 evidence operation 会在触碰 fixture store 前拒绝。缺少参数值、JSON 非法、输入文件不存在、两种输入同时出现或 schema 非法都会返回 exitCode 2，错误输出会脱敏，不回显原始 payload，也不会调用 runner；`probe protocol --operation recoverSession` 和 `probe protocol --operation consumeResetCoupon` 会返回 exitCode 2，因为它们是离线 evidence operation。

}
~~~

常见结果边界：

| status | error.category | 含义 | 下一步 |
|---|---|---|---|
| `success` | 无 | 注入的协议 operation 成功返回。 | 将脱敏 fixture 加入回归测试，再把真实 endpoint 行为接入 runtime。 |
| `failed` | `account_not_found` | 本地账号元数据中没有该 ID。 | 先导入账号或修正 `--account`。 |
| `failed` | `session_missing` | 账号存在，但 `cookieJarRef` 指向的本地 secret 不存在。 | 重新导入/刷新本地 Tabbit session。 |
| `skipped` | `protocol_missing` | 未配置 protocol client factory 或该 operation hook。 | 这是默认安全行为；接入真实探针前不要猜 endpoint。 |
| `failed` | `invalid_request` | operation 名称不在当前 runner 支持列表中。 | 修正 `--operation` 并补测试。 |

示例：

~~~powershell
node bin/tabbit-pool.js probe protocol --account acct_default --operation verifySession --write-fixture --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation sendMessage --input-json '{"model":"tabbit/priority","messages":[{"role":"user","content":"ping"}]}' --write-fixture --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation listModels --input-file .\probe-input.json --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation uploadAttachment --input-json '{"attachment":{"filename":"probe.txt","mimeType":"text/plain","data":"base64-probe-payload"}}' --write-fixture --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation getNewbieExplorationMe --input-json '{"viewMode":"activity_page"}' --write-fixture --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation getPlacementResources --input-json '{"placementCode":"home.input_below"}' --write-fixture --json

node bin/tabbit-pool.js probe protocol --account acct_default --operation getDailySignInStatus --input-json '{"sceneCodes":["desktop_pet"]}' --write-fixture --json
~~~

### tabbit-pool probe template [--operation <name>] [--json]

输出可直接传给 `probe protocol --input-json` 或保存后传给 `--input-file` 的安全 JSON object 模板。该命令不读取账号、不触发网络，也不输出 cookie、token、验证码或真实邮箱。`--json` 当前与默认输出一致，都会打印格式化 JSON，便于复制到文件。

支持的 operation：

- `verifySession`：输出 `{}`。
- `sendMessage`：输出 `{ "model": "tabbit/priority", "messages": [{ "role": "user", "content": "ping" }], "stream": true }`，便于直接用于真实上游 stream boundary 捕获前的 input 文件审查。
- `listModels`：输出 `{ "force": true }`。
- `refreshQuota`：输出 `{}`，默认使用账号元数据中的 `userId`。
- `getLotteryExplorationMe`：输出 `{}`。
- `getNewbieExplorationMe`：输出 `{ "viewMode": "activity_page", "includeCompletions": true, "includeRewards": true }`。
- `getPlacementResources`：输出 `{ "placementCode": "home.input_below" }`。
- `listRewardCardRecords`：输出 `{ "offset": 0, "limit": 10 }`。
- `listLotteryHitRecords`：输出 `{ "offset": 0, "limit": 20 }`。
- `getDailySignInStatus`：输出 `{ "sceneCodes": ["desktop_pet"] }`。
- `dailySignIn`：输出 `{ "confirmSideEffect": false, "requestNo": "desktop-pet-sign-in-probe", "sceneCodes": ["desktop_pet"] }`。
- `listBenefitCoupons`：输出 `{ "couponType": "weekly_reset_coupon", "offset": 0, "limit": 50 }`。
- `participateResetCouponActivity`：输出 `{ "confirmSideEffect": false, "requestNo": "reset-coupon-activity-probe" }`。
- `participateActivity`：输出 `{ "confirmSideEffect": false, "body": {} }`。
- `useResetCoupon`：输出 `{ "confirmSideEffect": false, "couponCode": "coupon-code", "couponType": "weekly_reset_coupon", "requestNo": "reset-coupon-use-probe" }`；真实运行前必须替换为安全选择的券码并显式确认副作用。
- `getUsageResetCouponSku`：输出 `{}`。
- `getAvailableLotteryChanceCount`：输出 `{ "activityId": "activity-id" }`。
- `getActiveMainPools`：输出 `{ "activityId": "activity-id" }`。
- `listLotteryChanceRecords`：输出 `{ "activityId": "activity-id", "offset": 0, "limit": 20 }`。
- `drawLottery`：输出 `{ "confirmSideEffect": false, "body": {} }`。
- `recoverSession`：输出 `{ "kind": "session_recovery_strategy", "operation": "recoverSession", "status": "success", "evidence": { "strategy": "automated_reauth", "automatedRefresh": "calibrated_reauth_probe", "safe": true, "sanitized": true, "rawPayload": false } }`；这是离线 session recovery strategy evidence 模板，只能用于 `probe validate` 和后续脱敏 fixture 准备。
- `consumeResetCoupon`：输出 `{ "kind": "reset_coupon_consumption_evidence", "operation": "consumeResetCoupon", "status": "success", "evidence": { "endpointHash": "sha256:<redacted-endpoint>", "bodyHash": "sha256:<redacted-body>", "resultHash": "sha256:<redacted-result>", "safe": true, "sanitized": true, "rawPayload": false }, "result": { "resetCouponConsumed": true, "consumeResult": "success" } }`；这是离线真实用券消耗 evidence 模板，只能用于 `probe validate` 和后续脱敏 fixture 准备。
- `uploadAttachment`：输出 `{ "attachment": { "filename": "probe.txt", "mimeType": "text/plain", "data": "base64-probe-payload" } }`；写入 fixture 时 `attachment.data` 会被屏蔽。

缺少 `--operation` 时默认 `verifySession`。不支持的 operation 返回 exitCode 2，并只列出支持的 operation，不回显用户输入。`recoverSession` 模板不代表自动刷新 endpoint 已校准；`consumeResetCoupon` 模板只用于外部脱敏 evidence 导入，真实协议捕获应使用 `useResetCoupon`，且 `probe protocol --operation consumeResetCoupon` 会被拒绝。

示例：

```powershell
node bin/tabbit-pool.js probe template --operation sendMessage --json > .\probe-input.json
node bin/tabbit-pool.js probe protocol --account acct_default --operation sendMessage --input-file .\probe-input.json --write-fixture --json
```

### tabbit-pool probe advice [--category <category>] [--status <status>] [--code <code>] [--message <text>] [--json]

根据错误分类、HTTP status、错误 code 或错误 message 输出下一步建议，不读取账号、不触发网络。403 会输出 `forbidden.kind`，用于区分风控、签名/协议、session/cookie、权益/模型和未知 forbidden。

示例：

~~~powershell
node bin/tabbit-pool.js probe advice --category protocol_changed --json
~~~

输出：

~~~json
{
  "category": "protocol_changed",
  "severity": "warning",
  "recommendation": "Capture a fresh fixture from Tabbit Web, compare request/response shape, and update protocol parsing tests before changing runtime behavior."
}
~~~

### 退出码

| 场景 | exitCode | 输出 |
|---|---:|---|
| 命令成功 | 0 | stdout |
| 未知命令 | 2 | stderr 帮助文本 |
| fixture ref 缺失、越界或文件不存在 | 2 | stderr 脱敏错误消息 |
| probe input 缺失、冲突、文件不存在、JSON 非法或 schema 非法 | 2 | stderr 脱敏错误消息，不调用 runner |
| serve/start port 非法或缺少参数值 | 2 | stderr 脱敏错误消息，不创建 gateway |
| smoke gateway 任一步 HTTP/fetch 失败 | 1 | stdout 中 `status:"failed"` 和 `failedStep`，不泄露 key |
| readiness mark 缺少 verification flag | 2 | stderr 脱敏错误消息，不写 readiness.json |
| probe template operation 不支持 | 2 | stderr 只列出支持的 operation |
| 运行时错误 | 1 | stderr 脱敏错误消息 |

## 脱敏边界

- `/health` 默认只返回统计摘要，不返回账号列表。
- `accounts list` 返回账号列表，但只返回展示字段，且所有敏感字段会被移除或脱敏。
- `redactAccountForDisplay()` 可返回账号 ID、状态、额度、冷却时间和脱敏错误摘要。
- 不返回 `cookie`、`cookieHeader`、`cookieJar`、`cookieJarRef`、`session`、`sessionToken`、`token`。
- 邮箱通过 `redactSensitiveValue()` 变成前缀 + `***@domain`。
- action log error message 和 CLI runtime error 会额外屏蔽 4 到 8 位裸数字，避免验证码进入日志。
- `fixtures show` 会对读取到的 fixture 再执行 `sanitizeProtocolProbeFixture()`，并移除展示账号中的 `cookieJarRef`、`cookieHeader`、`session`、`sessionToken`、`token`。
- `fixtures audit` 只输出覆盖计数、缺口名称和固定 nextActions，不输出 fixture request/response/error 原文。
- `probe protocol --input-json/--input-file` 的解析错误和 schema 错误不会回显原始 JSON payload，避免把 token、cookie 或验证码写入 stderr。
- `probe template` 只输出固定安全样例，不读取账号 secret；不支持的 operation 错误也不会回显用户输入。

# M04-账号注册初始化

## 定位

AccountProvisioner 负责把“邮箱接码 + Tabbit 注册/登录 + session 保存 + 首次权益初始化”串成一个可重试、可审计的流程。当前已实现离线可测的编排层；真实 Tabbit session 验证已校准为 `GET /api/v0/user/base-info`。发送验证码、提交验证码和注册/登录现在已有可配置 `ProtocolTabbitClient` 协议入口，默认 CLI 依赖会在显式配置 auth send/submit path 后把这些方法暴露给 `AccountProvisioner`，并可通过 `probe protocol` 的 auth evidence pipeline 写入脱敏 fixture；但真实 endpoint path、method、body 和 success response 仍待安全 evidence 校准，因此编排层继续只依赖注入的 `protocolClient` 方法，不硬编码 URL。

## 流程

~~~text
createInbox
  ↓
sendVerificationCode（Tabbit 接口待还原）
  ↓
waitForVerificationCode
  ↓
submitRegistrationOrLogin（Tabbit 接口待还原）
  ↓
saveSession
  ↓
claimPro / refreshQuota
  ↓
account active
~~~

当前代码中的固定顺序：

1. `mailProvider.createInbox()`
2. 创建 `provisioning` 账号元数据并写入 account store
3. `protocolClient.sendVerificationCode()`
4. `mailProvider.waitForVerificationCode()`
5. `protocolClient.submitRegistrationOrLogin()`
6. `secretStore.writeSecret(cookieJarRef, sessionSecret)`
7. 可选 `benefitsMaintainer.maintainAccount()`
8. 写入 `active` 账号元数据

## 公开能力

| 能力 | 方法 |
|---|---|
| 创建账号 | `createAccount(input)`，已实现离线编排，依赖注入的邮箱和协议操作。 |
| 导入账号 | `importSession(input)`，已实现 secret-first 保存。 |
| 重试初始化 | `resumeProvisioning(accountId)`，已实现状态检查和可注入 resume hook。 |
| 验证 session | `verifyAccount(accountId)`，已实现读取 secret 并调用可注入 verifier。 |

## 构造依赖

~~~js
const provisioner = new AccountProvisioner({
  accountStore,
  secretStore,
  mailProvider,
  protocolClient,
  benefitsMaintainer,
});
~~~

- `accountStore` 必须提供 `loadAccounts()` 和 `saveAccounts()`。
- `secretStore` 必须提供 `readSecret()` 和 `writeSecret()`。
- `mailProvider.createInbox()` 与 `mailProvider.waitForVerificationCode()` 只在 `createAccount()` 中需要。
- `protocolClient.sendVerificationCode()`、`submitRegistrationOrLogin()`、`resumeProvisioning()`、`verifySession()` 都是注入点，不在编排层硬编码 URL。默认 `ProtocolTabbitClient` 只有显式配置 `authSendCodePath` / `authSubmitCodePath` 后才会请求网络，默认 CLI 依赖也只在对应 path 存在时把 auth send/submit 方法接给 `AccountProvisioner`；缺失配置会保持 `protocol_missing` 或未配置 operation 语义。auth evidence 捕获应优先通过 `probe template/protocol --operation sendVerificationCode|submitRegistrationOrLogin`，并显式设置 `confirmSideEffect:true` 后写入脱敏 fixture；auth probe 可以使用没有 session secret 的 provisioning 账号元数据。
- `benefitsMaintainer` 可选；存在 `maintainAccount()` 时，注册保存 session 后会执行首次权益初始化。

## 完成标准

- 成功注册后账号状态为 `active`。
- 失败阶段可定位到邮箱、验证码、注册、登录、权益初始化中的具体一步。
- 不在日志中输出完整邮箱验证码、cookie、token。
- raw cookie/session 只写入 `secretStore`，账号 JSON 元数据只保留 `cookieJarRef`。
- 写入 secret 失败时不会持久化 `active` 账号。
- 真实发送验证码/提交验证码 success evidence 未捕获前，不把 M04 注册/登录标记为真实闭环完成；配置 path 和 AccountProvisioner wiring 只证明运行路径可用，不替代 delivery success 与 session material fixture gate。

# 验证码注册流程

## 目标

用 YYDS Mail 自动接收验证码，完成 Tabbit 账号注册或验证码登录。

## 待还原接口

Tabbit 的发送验证码、提交验证码、注册/登录响应格式尚未在文档中确认。当前代码只提供显式配置入口，不把默认 body 当作真实协议事实。真实启用前必须通过浏览器网络面板、自动浏览器 evidence 或前端代码还原：

- 发送验证码接口路径、method、body。
- 验证码用途字段（注册、登录、绑定邮箱等）。
- 提交验证码接口路径、method、body。
- 成功响应中 session/cookie/token 的来源。
- 失败响应中的错误码和可重试性。

当前 `AccountProvisioner` 不硬编码这些 endpoint。它只调用注入的 `protocolClient.sendVerificationCode()` 与 `protocolClient.submitRegistrationOrLogin()`；默认 `ProtocolTabbitClient` 已提供可配置入口，默认 CLI 依赖也会在对应 path 显式配置后，把这两个方法接入可传给 `AccountProvisioner.createAccount()` 的协议适配器：

| 配置 | 默认 | 说明 |
|---|---|---|
| `TABBIT_POOL_PROTOCOL_AUTH_SEND_CODE_PATH` | 空 | 发送验证码 path；空值返回 `MISSING_AUTH_SEND_CODE_PATH`，不触网。 |
| `TABBIT_POOL_PROTOCOL_AUTH_SEND_CODE_METHOD` | `POST` | 发送验证码 method，会转为大写。 |
| `TABBIT_POOL_PROTOCOL_AUTH_SUBMIT_CODE_PATH` | 空 | 提交验证码/注册登录 path；空值返回 `MISSING_AUTH_SUBMIT_CODE_PATH`，不触网。 |
| `TABBIT_POOL_PROTOCOL_AUTH_SUBMIT_CODE_METHOD` | `POST` | 提交验证码 method，会转为大写。 |

默认请求 body 仅为校准占位：发送验证码 `{ "email": "<email>" }`，提交验证码 `{ "email": "<email>", "code": "<verification code>" }`。如果真实抓包 body 不同，应通过显式 `body` / `input.authSendCodeBody` / `input.authSubmitCodeBody` 传入，并补脱敏 fixture 与回归测试后再固化。配置 path 和默认 AccountProvisioner wiring 只说明运行调用链已接通，不代表真实发送/提交成功语义已闭环。`readiness doctor --json` 会以 `protocol.authSendCodePathConfigured` / `protocol.authSubmitCodePathConfigured` 暴露布尔配置状态，plain 输出对应 `auth_send_endpoint` / `auth_submit_endpoint` 的 `configured` 或 `missing`，但不会输出真实 path，也不会改变 auth fixture gate。auth capture command 会额外输出 `prerequisites` / `prerequisitesStatus`，plain `capture_command` 行对应 `prereq=TABBIT_POOL_PROTOCOL_AUTH_SEND_CODE_PATH:configured|missing` 或 `prereq=TABBIT_POOL_PROTOCOL_AUTH_SUBMIT_CODE_PATH:configured|missing`；这些字段只暴露 env 名与状态，不包含 endpoint 值，也不证明 endpoint、请求 body 或成功响应语义已校准。

auth evidence 捕获已接入现有 protocol probe fixture pipeline。可用 `probe template --operation sendVerificationCode` 和 `probe template --operation submitRegistrationOrLogin` 生成占位 input；模板默认 `confirmSideEffect:false`，真实执行前必须人工替换为已确认的脱敏 body，并显式设置 `confirmSideEffect:true`。`probe protocol` 仍要求传入一个账号 id 以便记录归属，但 auth operation 可以在没有本地 session secret 的 provisioning 账号上执行；runner 会在调用协议 client 前校验 email/code/body/confirmSideEffect，且写入 fixture 时会屏蔽邮箱、验证码、cookie、session、token 和 raw payload。捕获后用 `fixtures audit --scope auth --json` 只读检查成功发送验证码与成功提交验证码 fixture 覆盖；发送验证码 coverage 要求成功响应形状包含 `codeSent`、`emailSent`、`sendResult` 等 send/delivery 专属成功信号，泛 2xx、`ok:true` 或 `status/result:"success"` 只保留为 transport success 计数；提交验证码 coverage 还要求成功响应形状包含 `cookieHeader`、`cookieJar`、`cookie`、`session`、`sessionToken` 或 `token` 等可导入 session material 字段，避免把无法激活账号的 2xx 样本误判为闭环。默认 `fixtures audit --json` 仍只检查 gateway/chat readiness 的 verify/send/stream/tool/403 覆盖。

提交成功响应会归一化 `cookieHeader`、`cookie`、`session`、`sessionToken`、`token`、`userId`、`accessTier` 等字段，供 `AccountProvisioner.extractSessionSecret()` 保存到 secret store。raw session/cookie 只允许进入本地 secret store，不进入账号元数据或文档。

## 流程契约

~~~ts
type CreateAccountInput = {
  accountId?: string;
  localPartPrefix?: string;
  domain?: string;
  subdomain?: string;
  passwordStrategy?: "none" | "generated";
  timeoutMs: number;
  cookieJarRef?: string;
};
~~~

成功返回：

~~~ts
type ProvisionAction = {
  name: string;
  status: "success" | "skipped" | "failed";
  changed: boolean;
  detail?: string;
  error?: { message: string; code?: string; category?: string };
};

type ProvisionResult = {
  account: Account | null;
  changed: boolean;
  actions: ProvisionAction[];
};
~~~

`createAccount()` 在创建 inbox 后会先保存 `status:"provisioning"` 的账号元数据。验证码等待超时、发送失败或提交失败会保留 provisioning 账号并写入脱敏 `lastError`，便于后续 `resumeProvisioning(accountId)` 或人工排查。

## 错误处理

| 阶段 | 错误 | 动作 |
|---|---|---|
| 创建邮箱 | API key 无效 | 停止，提示配置 |
| 发送验证码 | Tabbit 限流 | 冷却，换邮箱前先等待 |
| 等待邮件 | timeout | 可重试发送验证码 |
| 提交验证码 | code_invalid | 重新等待或重新发送 |
| 登录初始化 | session_missing | 标记 suspect，保存脱敏响应 fixture |
| 保存 session | 写入失败 | 保持 provisioning，不产生 active 账号 |

## 测试

- [x] 邮箱创建成功但验证码超时：账号保持 `provisioning`，错误摘要脱敏。
- [x] 验证码提取成功并提交：按固定顺序调用注入方法。
- [x] 成功后进入保存会话：先写 secret，再写 active 账号元数据。
- [x] 登录初始化缺少 session：账号标记 `suspect`，不写 secret。
- [x] 可配置发送验证码/提交验证码协议入口：缺配置不触网，协议 `ok:false` 会让 AccountProvisioner 停在对应失败阶段。
- [x] 默认 CLI 依赖会在显式配置 auth send/submit path 后，把可配置协议入口暴露给 AccountProvisioner 编排层。
- [x] auth protocol probe pipeline：支持生成安全模板、显式确认 side effect、写入脱敏 fixture。
- [ ] 真实发送验证码/提交验证码接口路径、body、错误码和 success session 来源仍待安全 evidence 还原；submit success evidence 必须证明 session material shape。

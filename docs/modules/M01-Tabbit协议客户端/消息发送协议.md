# 消息发送协议

## 目标

还原 Tabbit Web Chat 的消息发送和响应解析，使外部兼容网关不再依赖浏览器 UI。

## 网关内部输入

~~~ts
type ProtocolMessage = {
  role: "system" | "user" | "assistant" | "tool";
  content?: string | unknown[];
  tool_call_id?: string;
  tool_calls?: unknown[];
  type?: string;
  [key: string]: unknown;
};

type SendMessageInput = {
  account: Account;
  model: string;
  messages: ProtocolMessage[];
  attachments?: unknown[];
  references?: unknown[];
  metadatas?: Record<string, unknown> | null;
  stream?: boolean;
};
~~~

为兼容 Codex、Claude Code 和三方 SDK 的工具回合，`messages` 当前允许保留官方结构：OpenAI Chat assistant `tool_calls`、tool message `tool_call_id`，OpenAI Responses `function_call` / `function_call_output` item，以及 Anthropic `tool_use` / `tool_result` content block。旧显式 `sendPath` 骨架会把这些结构写入 signed body；本地工具执行不发生在 ProtocolTabbitClient 内，而是在可选 LocalToolLoopRunner wrapper 中完成。真实 `/api/v1/chat/completion` 分支拒绝 `tools`、`toolChoice` 或 `parallelToolCalls`，返回 `unsupported_feature/TOOL_FIELDS_UNSUPPORTED`。

## 真实发送接口

已校准的文本发送接口：

~~~text
POST https://web.tabbit.ai/api/v1/chat/completion
Accept: text/event-stream
Content-Type: application/json
Cache-Control: no-cache
~~~

请求体：

~~~json
{
  "chat_session_id": "<session uuid>",
  "message_id": null,
  "content": "<prompt text>",
  "selected_model": "Default",
  "parallel_group_id": null,
  "task_name": "chat",
  "agent_mode": false,
  "metadatas": {
    "html_content": "<p>&lt;escaped prompt&gt;</p>"
  },
  "references": [],
  "entity": {
    "key": "d41d8cd98f00b204e9800998ecf8427e",
    "extras": {
      "type": "tab",
      "url": ""
    }
  }
}
~~~

字段规则：

- `chat_session_id` 必须来自 `sendMessage({ chatSessionId })`、账号元数据或 `TABBIT_POOL_PROTOCOL_CHAT_SESSION_ID`。
- `tabbit/priority` 映射为真实 `selected_model:"Default"`；`Default` 在当前模型目录中为 `free_unlimited`。
- prompt 来自 `content` 或 `messages` 归一化文本；`metadatas.html_content` 会做 HTML escape，并把换行转成 `<br>`。
- `references` 和 `entity` 保留真实空 tab 结构；已上传附件引用会追加到 `references`，document/image 的真实结构见下方附件策略。
- 该分支使用浏览器校准签名头，见 [签名头生成](签名头生成.md)。

## 响应解析

需要支持两类结果：

1. 非流式完整响应：解析出最终 assistant 文本。
2. 流式响应：当前已具备 SSE/NDJSON buffered 聚合解析和 response.body async delta producer。buffered 路径能把已配置 `sendPath` 返回的 SSE `data:` delta 或 newline-delimited JSON delta 聚合为最终 assistant 文本，并能把 buffered OpenAI stream `tool_calls` chunks、Anthropic stream `tool_use` / `input_json_delta` chunks 聚合为内部 `tool_use` block，同时保留解析后的 raw events 与数组 `streamDeltas`；async 路径会在 `stream:true`、2xx、支持的 Content-Type 且 `response.body` 可读时立即返回 async iterable `streamDeltas`，消费时边读 body 边产出文本 delta。对外网关会把这些 delta 映射为多个本地 SSE delta；async iterable 会走 chunked flush。消费端提前调用 async iterator `return()` 时，Web `ReadableStream` 分支会取消底层 reader 并释放锁。

真实 `/api/v1/chat/completion` 返回 `text/event-stream`。已捕获的成功事件包括 `ready`、`message_start`、`message_chunk`、`message_finish`、`finish`、`close`；高级模型不可用时会返回 `event:error`，例如 `code:492` 与“模型 GPT-5.5 仅高级用户可用”的文案。解析器会从 `content`、`data.content`、`delta` 等字段提取文本，并把错误帧分类为 `ProtocolTabbitError`。

旧显式 sendPath 仍支持 SSE/NDJSON parser foundation；外部 HTTP SSE 事件映射见 [流式 SSE 链路](../../10-流式SSE链路.md)。支持的文本 delta 字段包括 `delta`、`text`、`content`、`message.content`、`data.delta`、`data.text`、`data.content`、`choices[0].delta.content` 和 `choices[0].message.content`；buffered 工具 delta 支持 OpenAI `tool_calls` 形状（例如 `choices[0].delta.tool_calls`）和 Anthropic `content_block_start` + `input_json_delta.partial_json` 形状。无法提取文本或工具调用时仍归类为 `protocol_changed`。可提取的文本分片会进入数组或 async iterable `streamDeltas`，供 OpenAI/Anthropic SSE adapter 保留上游分片边界；buffered tool calls 会进入 `contentBlocks`。

流式错误帧不会被静默忽略。当前会识别 SSE `event:error`、对象 `type:"error"` / `event:"error"`、`error`、`errorCode`、以及 `code + message` 形状；buffered 路径会让 `sendMessage()` 返回失败，async 路径会在 `streamDeltas` 迭代到错误帧时抛出 `ProtocolTabbitError`，并保留 raw event。分类复用 `classifyProtocolError()`，因此 quota/usage/credit 耗尽信号会成为 `quota_exhausted`。消费端主动取消 async iterator 的本地 reader 取消语义已有测试覆盖；真实 Tabbit 错误帧全集、HTTP 断连传播和 backpressure 语义仍需 fixture 校准，并可通过 `fixtures audit --scope upstream` 只读跟踪缺口。`readiness doctor` 会为 `real_upstream_error_frame_fixture`、`real_upstream_cancellation_fixture` 和 `real_upstream_backpressure_fixture` 输出 `sendMessage` 占位 capture command，plain 行会显示 `prereq=TABBIT_POOL_PROTOCOL_SEND_PATH:configured|missing`；该命令只帮助准备脱敏探针输入，不替代真实上游 marker fixture。

## 错误分类

| 上游现象 | ProtocolError.category | 账号池动作 |
|---|---|---|
| 401/登录页面/无 token | login_required | 标记 `login_expired` |
| 额度不足文案或错误码 | quota_exhausted | 标记 `quota_exhausted`，尝试下一账号；优先于 429/5xx 状态码 |
| 403/签名失败/风控 | forbidden | 冷却或 suspect |
| 429 | rate_limited | 按 `Retry-After` 冷却 |
| 5xx | upstream_error | 短冷却，可重试 |
| 网络超时 | network_error | 可重试，不立即禁用账号 |
| 响应格式无法解析 | protocol_changed | 停止自动重试，要求更新协议 |

## 附件策略

当前消息发送支持“引用已上传附件”，并在完整配置 COS 上传链时支持 raw/base64 附件自动上传后发送。真实抓包和前端 bundle 校准到的上传链路是：

1. `POST /proxy/v0/cos/presigned-upload-url` 获取预签名 COS URL。
2. `PUT <presigned COS url>` 上传文件内容。
3. `POST /api/v0/cos/complete-upload` 完成上传并拿到可引用的 file id。

- `uploadAttachment()` 未配置路径时返回 `protocol_missing/MISSING_ATTACHMENT_UPLOAD_PATH`，不会触网。
- 只配置 `attachmentUploadPath` 时，`uploadAttachment()` 保留旧签名 POST 骨架：先获取 sign key，再对 `POST + attachmentUploadPath + { attachment }` 生成签名头，携带可用 `Cookie`，并把响应中的 `id/attachmentId/fileId`、`name/filename/fileName`、`mimeType/contentType/type`、`size/bytes` 归一化为 `attachment` 对象。
- 同时配置 `attachmentUploadPath=/proxy/v0/cos/presigned-upload-url` 和 `attachmentCompleteUploadPath=/api/v0/cos/complete-upload` 时，`uploadAttachment()` 走真实 COS 三步链路。两个 Tabbit POST 使用 `Content-Type: application/json`、`trace-id` 和 Cookie，不请求 sign key，不发送 `x-signature` / `x-nonce`；COS PUT 只带文件 `Content-Type`。
- `/api/v1/chat/completion` 分支中，`sendMessage({ attachments })` 接受已上传附件引用：`path/file_id/fileId/id/metadata.file_id` 会作为真实 `file_id`。
- 如果附件缺少 file id/path 但包含 `data`、`base64`、`raw` 或 `body`，且完整 COS 上传链已配置，`sendMessage()` 会先上传附件，再把返回的 file id 写入 `references`。
- document 引用结构为 `{ "type": "document", "title": "<name>", "content": "", "metadata": { "file_id": "<file id>" } }`。
- image 引用结构为 `{ "type": "image", "title": "<name>", "content": "<optional url/content>", "metadata": { "file_id": "<file id>", "source_url": "<optional source url>" } }`。
- 未完整配置上传链时，缺少 file id/path 的 raw/base64 附件返回明确兼容错误 `unsupported_feature/ATTACHMENT_REFERENCE_REQUIRED`。
- 旧显式 sendPath 骨架收到附件引用仍返回 `unsupported_feature/ATTACHMENTS_UNSUPPORTED`。
- 不丢弃附件后继续发送纯文本，避免客户端误以为附件已处理。

## 测试

- 文本成功响应 fixture。
- SSE/NDJSON stream 分片 fixture：`sendMessage({ stream:true })` 能把 buffered `text/event-stream` 的 `data:` delta 或 newline-delimited JSON delta 聚合为最终文本，raw 保存结构化 events，数组 `streamDeltas` 保存可提取文本分片，而不是把 wire text 当成助手输出。
- buffered tool call fixture：OpenAI 分帧 `choices[0].delta.tool_calls` 能按 index 聚合为内部 `tool_use` block；Anthropic `content_block_start` + `input_json_delta.partial_json` 能按 content block index 聚合为内部 `tool_use` block；二者都会把 `stopReason` 设为 `tool_use`。
- response.body async stream fixture：Web `ReadableStream` 先释放第一帧、延迟第二帧时，`sendMessage({ stream:true })` 必须在第二帧释放前返回 async iterable `streamDeltas`；第一段 delta 可立即消费，第二段必须等上游继续写入。
- stream error frame fixture：buffered `event:error` 帧会被分类为 `{ ok:false, error }`；async `event:error` 帧会让 `streamDeltas` reject，同时 raw events 保留错误帧。
- async stream cancellation fixture：消费第一段 async `streamDeltas` 后调用 iterator `return()`，底层 Web `ReadableStream` source 必须收到 `cancel("stream_deltas_cancelled")`。
- upstream boundary audit：`fixtures audit --scope upstream` 只统计带真实上游 evidence marker 的脱敏 `sendMessage` fixture；本地 HTTP/compat/fake stream 测试不满足 `real_upstream_error_frame_fixture`、`real_upstream_cancellation_fixture` 或 `real_upstream_backpressure_fixture`。`readiness doctor` 的 upstream capture command 只能提供 `probe template` / `probe validate` / `probe protocol` 占位步骤，不能把缺口标记为 ready。
- 七类错误分类 fixture。
- 显式附件上传路径 fixture：`uploadAttachment()` 会签名请求 configured `attachmentUploadPath`，携带 hydrated `Cookie`，并归一化附件 id/name/mimeType/size。
- 真实 COS 上传 fixture：`uploadAttachment()` 在同时配置 presign/complete path 时按 presign -> PUT -> complete 顺序上传，且不生成 `x-signature` / `x-nonce`。
- 已上传附件引用测试：`sendMessage({ attachments })` 在真实分支把 document/image 映射为 `references[].metadata.file_id`，并保留外部传入的 `references` 与 `metadatas`。
- raw/base64 附件自动上传测试：完整配置 COS 上传链时先上传再发送；未完整配置时返回 `ATTACHMENT_REFERENCE_REQUIRED`，不自动降级为纯文本发送。
- 工具回合输入保真 fixture：`sendMessage()` 的 signed body 保留 OpenAI `tool_calls` / `tool_call_id` / `function_call_output` 和 Anthropic `tool_result` 结构，签名基于最终 body。

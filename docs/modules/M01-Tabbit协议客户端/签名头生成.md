# 签名头生成

## 目标

复刻 Tabbit Web 请求签名，使 Node HTTP client 能直接调用需要签名的 Tabbit 接口。

## 已知线索

- sign key 接口：`GET https://web.tabbit.ai/chat/sign-key`。
- 真实消息发送请求头：`x-timestamp`、`x-nonce`、`x-signature`、`x-req-ctx`、`unique-uuid`。
- 真实发送接口 `POST /api/v1/chat/completion` 使用浏览器校准签名：`x-signature` 是 UUID/randomUUID 风格字符串，`x-nonce` 是对签名 payload 做 HMAC-SHA256 的 hex digest。
- 旧的 method/path/query/canonical-body 签名骨架仍保留给未校准的显式路径测试；真实 chat completion 分支使用浏览器算法。

## 接口草案

~~~ts
type SignRequestInput = {
  method: string;
  path: string;
  query?: URLSearchParams | Record<string, string>;
  body?: unknown;
  signKey: string;
  timestamp?: number;
  nonce?: string;
};

type SignedHeaders = {
  "x-timestamp": string;
  "x-nonce": string;
  "x-signature": string;
  "x-req-ctx"?: string;
  "unique-uuid"?: string;
};
~~~

## 已校准算法

真实 `POST /api/v1/chat/completion` 分支：

~~~text
bodyText = JSON.stringify(requestBody)
bodyHash = SHA256(bodyText).hex()
x-signature = randomUUID()
x-timestamp = Date.now() 毫秒
payload = `${x-timestamp}.${x-signature}.${bodyHash}`
x-nonce = HMAC-SHA256(signKey, payload).hex()
~~~

同时发送：

- `accept: text/event-stream`
- `content-type: application/json`
- `cache-control: no-cache`
- `x-req-ctx: MS4zLjI2KDEwMTAzMDI2KQ==`，可通过 `TABBIT_POOL_PROTOCOL_REQ_CTX` 覆盖
- `unique-uuid`，按浏览器模块中带时间戳位的 UUID 风格生成
- `Cookie`，只来自运行时 hydrate 的本地 secret

## 还原记录

1. Playwright 自动打开已登录 Tabbit 页面并捕获真实 `POST /api/v1/chat/completion`。
2. 在 webpack module 中定位 `x-signature` / `x-nonce` / `x-timestamp` / `unique-uuid` / `x-req-ctx` 生成逻辑。
3. 对同一 body 对比浏览器签名输入和输出，确认 `x-nonce = HMAC(signKey, timestamp.signature.sha256(bodyText))`。
4. 使用固定 timestamp、signature、signKey 和 bodyText 写入离线 golden test。
5. Node 实现与浏览器输出一致后，接入 `ProtocolTabbitClient.sendMessage()` 的 restored chat completion 分支。

## 边界条件

- 真实 chat completion 签名使用原始 `JSON.stringify()` 后的 body text hash，不能再对该分支改用 canonical JSON，否则签名会变化。
- timestamp 使用毫秒；服务端容忍窗口未知，探针应使用当前时间。
- `x-signature` 与 `unique-uuid` 是不同 header；前者参与 HMAC，后者是浏览器上下文标识。
- sign key 当前来自 `/chat/sign-key`；是否按账号或版本变化仍需长期观察。
- 失败时不能反复请求造成账号风控，应返回 `protocol_changed` 或 `forbidden`。

## 测试

- 固定输入输出 golden test。
- body 为空、对象、数组、字符串时的 canonical test。
- timestamp 过期时的错误分类 test。
- sign key 刷新失败时的 fallback test。

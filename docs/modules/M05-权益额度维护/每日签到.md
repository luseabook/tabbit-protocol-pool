# 每日签到

## 目标

按账号执行每日签到，领取赠送额度，并避免重复签到造成无意义请求。

## 策略

- 使用账号本地时区或服务端日期作为签到窗口，具体以 Tabbit 接口为准。
- `lastCheckinAt` 在同一服务端日期内存在时跳过。
- 签到失败按错误分类处理，不影响模型请求主链路。

## 待还原点

- 签到响应是否包含额度变化。
- 重复签到的错误码或幂等返回。

## 已校准协议

真实桌宠签到状态与签到 POST 已还原为显式 probe：

~~~text
GET  /api/commerce/activity/v1/sign-in/status?scene_codes=desktop_pet
POST /api/commerce/activity/v1/sign-in
~~~

状态查询返回 `sign_in_date` 和 `results[]`，桌宠场景码为 `desktop_pet`，query 使用 repeated `scene_codes`。签到 POST body 为：

~~~json
{
  "request_no": "desktop-pet-sign-in-<caller-generated>",
  "scene_codes": ["desktop_pet"]
}
~~~

`request_no` 由调用方生成，生产调用点使用短字符串形态；`ProtocolTabbitClient.dailySignIn()` 和 CLI schema 都限制为非空且不超过 64 字符。POST 使用已登录 Cookie、`trace-id` 和 `Content-Type: application/json`；不请求 sign-key，不发送 `x-signature` / `x-nonce`。`ProtocolTabbitClient.dailySignIn()` 只在调用方显式传入 `confirmSideEffect:true` 时触网；默认 CLI maintainer 只有在显式配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_PATH` 后才会注入 `dailyCheckin`，并生成 `daily-sign-in-YYYYMMDD-<12 hex>` 短 request_no。若同时配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_STATUS_PATH`，维护前会先查 `signedToday`，已签则不再 POST。

`ProtocolTabbitClient.dailySignIn()` 会把返回中的 `sign_in_date`、首个 `results[].signed_today`、`results[].sign_in_result`、`signed_days` 和 `total_signed_days` 归一化为 `signInDate`、`signedToday`、`signInResult`、`signedDays` 和 `totalSignedDays`，方便显式 probe fixture 判断签到结果。`fixtures audit --scope benefits` 对每日签到接受 `signInResult:"success"`、`signedToday:true` 或 `signInResult:"already_signed"` 作为成功覆盖；这个放宽只适用于 `dailySignIn` 的幂等日状态，不适用于活动 Pro、抽奖或重置券消耗。

2026-07-03 live 脱敏 evidence `output/tabbit-benefits-side-effects/side-effect-live-20260703T082435232Z.sanitized.json` 验证：短 `request_no` 的签到 POST 返回 `sign_in_result:"success"`，随后状态从 `signed_today:false` 变为 `signed_today:true`，`signed_days` 和 `total_signed_days` 都递增。quota usage 在该轮没有证明可归因的额度变化，因此高层维护只把签到本身视为成功，不声明额度增加。

2026-07-04 live probe 在已完成当天签到的账号上返回 `signInResult:"already_signed"`，脱敏 fixture 通过 benefits scope 审计计入 `successful_daily_sign_in_fixture`。该证据只说明签到 endpoint/body 与当天完成状态已闭环，不声明重复 POST 会再次发放额度。

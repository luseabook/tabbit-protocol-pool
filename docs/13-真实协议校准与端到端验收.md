# 13-真实协议校准与端到端验收

本文记录真实 Tabbit send endpoint / session verify endpoint / 签名 / 请求体校准结果，以及继续做 Codex / Claude Code 真实端到端验证、工具 loop 取舍和 403 响应体细分的验收流程。


## 0. 本地 readiness 预检

在跑真实网络探针前，先执行离线预检：

~~~powershell
node bin/tabbit-pool.js readiness --json
~~~

该命令只读取本地账号元数据、protocol probe fixture 摘要和本地 `readiness.json` 验收标记，不触发 Tabbit 网络。它会告诉你四项是否 ready：

1. `protocolCalibration`：是否已配置 protocol、sendPath、sessionVerifyPath、active 账号、真实 chat completion 所需 chat session context、成功 verifySession fixture 和成功 sendMessage fixture。
2. `codexClaudeE2E`：Codex / Claude Code 人工验收是否已记录。
3. `toolLoopDecision`：当前工具 loop 策略，默认客户端先执行工具，也可显式关闭或启用受控本地 loop。
4. `forbidden403`：是否已有 403/forbidden 脱敏 fixture。

`status:"blocked"` 时先按 `nextActions[]` 逐项补齐。Codex 或 Claude Code 真实端到端验收完成后，用下面命令写入本地标记：

~~~powershell
node bin/tabbit-pool.js readiness mark --codex-verified --json
node bin/tabbit-pool.js readiness mark --claude-verified --json
# 两项都刚完成时也可以一次写入
node bin/tabbit-pool.js readiness mark --codex-verified --claude-verified --json
~~~

标记写入 `TABBIT_POOL_STATE_DIR/readiness.json`，只记录 `verified:true` 和 `verifiedAt`，不保存 prompt、代码 diff、cookie 或 API key。

### 0.1 外部脱敏 evidence state 诊断

默认状态目录通常位于用户本地应用数据目录。如果该目录没有账号元数据、`readiness.json` 或 protocol probe fixture，`readiness` / `fixtures audit` 返回 `blocked` 是安全默认，不代表实现失败。要复用本机已经存在的脱敏 evidence state，应显式选择外部状态目录，再运行只读诊断：

~~~powershell
$env:TABBIT_POOL_STATE_DIR = "E:\tabbit-live-state"
$env:TABBIT_POOL_PROTOCOL_ENABLED = "true"
$env:TABBIT_POOL_COMPAT_STRIP_CLIENT_TOOLS = "true"

node bin\tabbit-pool.js production preflight --json
node bin\tabbit-pool.js readiness doctor --json
node bin\tabbit-pool.js readiness --json
node bin\tabbit-pool.js fixtures audit --json
node bin\tabbit-pool.js fixtures audit --scope session --json
node bin\tabbit-pool.js fixtures audit --scope upstream --json
~~~

`readiness doctor` 只读取当前 `stateDir` 下的账号摘要、脱敏 fixture 和 `readiness.json`，并返回协议 env 是否配置、readiness 状态、默认 fixture audit 状态、core 剩余工作、auth/benefits/session/upstream `calibrationBacklog` 和安全命令。它不会调用 `accounts probe`、不会运行 `probe protocol`、不会写 `readiness.json`，也不会输出 cookie、session、API key、`cookieJarRef`、`TABBIT_POOL_PROTOCOL_REQ_CTX`、`TABBIT_POOL_PROTOCOL_CHAT_SESSION_ID` 或 raw fixture payload。

真实 `/api/v1/chat/completion` 的当前发送能力需要两层状态同时成立：`verifySession` 成功说明登录 cookie 仍可用；`chat_session_context` ready 才说明真实发送能拿到 `chat_session_id`。该上下文可以来自单次 input、账号元数据、`TABBIT_POOL_PROTOCOL_CHAT_SESSION_ID`，也可以来自默认启用的 `TABBIT_POOL_PROTOCOL_CHAT_SESSION_AUTO_CREATE`：网关会先调用已校准的 `/newtab` Next Server Action 创建聊天会话，再发送 completion。账号列表显示 `chatSessionConfigured:false` 不再等于不可发送；应查看 doctor evidence 中的 `chatSessionContext.autoCreateConfigured`。如果 doctor 的 `protocolCalibration.missing` 包含 `chat_session_context`，修复路径是导入经审查的真实 chat session id、设置进程级默认 id，或配置/恢复已验证的自动建会话 action。

注意 `remainingWork: []` 只代表基础 gateway/chat readiness gate 没有剩余阻塞；注册/登录、M05 副作用、session lifecycle 和真实上游 stream boundary evidence 由 `calibrationBacklog` 跟踪。若 `calibrationBacklog.status` 仍为 `blocked`，必须继续运行对应 `fixtures audit --scope auth --json` / `fixtures audit --scope benefits --json` / `fixtures audit --scope session --json` / `fixtures audit --scope upstream --json` 和安全 probe 流程。`calibrationBacklog.captureCommands` 和非 JSON doctor 的 `capture_command` 行可作为下一步命令提示，但只包含 `<account-id>` / `<redacted-input.json>` 占位符，并按 template、validate、confirm_validate、probe 四步排列；真实副作用 probe 前必须先审查 input file 和 `confirmSideEffect`，先运行 `probe validate --operation <operation> --input-file <redacted-input.json> --json` 只读检查 schema、字段存在性和 body key 摘要，准备写入副作用 fixture 前再运行 `confirm_validate` 对应命令确认 `confirmSideEffect:true`，最后才决定是否运行 `probe protocol --write-fixture`。真实用券消耗缺口现在走 `useResetCoupon` template/validate/confirm_validate/probe 流，并要求 `TABBIT_POOL_PROTOCOL_BENEFIT_COUPON_USE_PATH`；离线 `consumeResetCoupon` 仅用于外部脱敏 endpoint/body/result hash evidence 导入。自动刷新仍只提供离线 `recoverSession` template/validate evidence 流且 `probe` 为空；离线 evidence 经人工审查后可在同一 validate 命令上追加 `--write-fixture` 写入白名单脱敏 fixture，不读取账号、secret 或网络，也不会保留 input 中的额外 raw payload 字段。真实上游 stream boundary 在已配置 send path 后会给出 sendMessage template/validate/probe 命令，但 audit 仍要求脱敏真实上游 evidence marker。session scope 的恢复策略当前仍是 `manual_reimport_then_probe`，自动刷新路径未校准。

默认 fixture audit 中的 `forbidden_403_fixture` 缺口也会进入 doctor 的 `captureCommands`，scope 显示为 `protocol`，operation 使用只读 `verifySession`。这只是 403/forbidden 真实响应出现后的安全捕获指引：命令只带 `<account-id>` / `<redacted-input.json>` 占位符和 env 配置状态，不输出 endpoint、cookie、session、token 或 raw payload；只有保存了 sanitizer 输出且 fixture audit 能识别为真实 403/forbidden 后，403 readiness 才会满足。

同一默认 fixture audit 的 `successful_sendMessage_fixture`、`streaming_text_fixture` 和 `tool_call_fixture` 缺口也会进入 doctor 的 `captureCommands`，scope 显示为 `protocol`，operation 使用 `sendMessage`。这些命令只给安全模板、schema validate 和 `<account-id>` probe 占位符；JSON 会带 `requiresReviewedInput:true` 与 `reviewRequirement:"replace_redacted_message_content"`，plain 行会打印 `review=replace_redacted_message_content`。`streaming_text_fixture` 建议准备 `stream:true` 输入，`tool_call_fixture` 允许真实 tool-call 脱敏 evidence 或明确 unsupported-native-tool 的脱敏 evidence。它们不会证明 prompt/body/上游语义已校准，也不会让默认 audit 或 `manualCookieMode` 在缺少真实 sanitized fixture 时变为 ready。

doctor 同时会给 Codex / Claude Code 人工 E2E 标记命令：JSON 字段为 `commands.codexE2EMark`、`commands.claudeE2EMark`、`commands.combinedE2EMark`，plain 输出为 `mark_command`。这些命令只在真实 base_url/key/model 验收完成并记录结果后用于写入本地 `readiness.json`；doctor 本身不写 state，也不能替代真实 E2E 运行证据。

当前发布目标另由 `manualCookieMode` 表达：这是“用户手动注册/登录 Tabbit、手动导入 cookie/session、失效后由 `verifySession` 识别 401/login_required 并标记 `login_expired`、再由用户手动重新导入”的运维版本。它不承诺自动注册、Yoda/短信登录自动化或自动刷新登录态；`automated_session_refresh_strategy` 只作为后续自动恢复 backlog。`manualCookieMode.status` 只有在基础 gateway/chat gate、真实 chat completion 的 `chat_session_context`（显式 id 或自动建会话）、默认 fixture audit、成功 `verifySession` evidence 和 401/login_required 过期 evidence 都满足时才会 ready。JSON doctor 的 `manualCookieMode.missing` / `manualCookieMode.blockingMissing` 只列当前 manual-cookie 发布阻塞项，`manualCookieMode.backlogMissing` 只列后续增强；plain `manual_cookie_mode` 行对应 `release_blocking_missing=<csv>` 与 `backlog_missing=<csv>`。生产接入前还应运行 `tabbit-pool production preflight --json`；该命令要求 gateway key 不是默认 `sk-tabbit-local`（来自 `TABBIT_POOL_API_KEY` 或 `<stateDir>/secrets/gateway-api-key.txt`）、doctor top-level ready 与 `manualCookieMode.blockingMissing:[]`，但不会把自动刷新 backlog 当作当前版本阻塞。

真实上游三个 `captureCommands` 会直接给出带 `--stream-evidence` 的 `templateCommand`，并保留推荐捕获模式：`real_upstream_error_frame_fixture` -> `error_frame:2`，`real_upstream_cancellation_fixture` -> `cancel_after_first_delta:2`，`real_upstream_backpressure_fixture` -> `first_token_backpressure:2`。JSON 字段为 `recommendedInput.streamEvidence`，plain 输出为 `stream_evidence=<mode>:<maxDeltas>`；由于这些命令都是 sendMessage probe，JSON 同样会带 `reviewRequirement:"replace_redacted_message_content"`，plain 同样会打印 `review=replace_redacted_message_content`。这些提示只包含聚合模式和审查要求，不包含 prompt、endpoint、payload、cookie、session、token 或真实用户数据，也不能替代真实 Tabbit 样本。

对 `successful_reset_coupon_consumption_fixture` 和 `automated_session_refresh_strategy`，JSON doctor 会直接输出 `writeFixtureCommand`，plain `capture_command` 会输出 `write_fixture=`。这两个命令等价于在已通过人工审查的离线 evidence `probe validate` 命令上追加 `--write-fixture`，只写白名单脱敏 fixture；其它 capture command 不会生成该字段，仍必须按真实协议 probe 或 fixture audit gate 校准。

外部 evidence state 的边界：

- raw 抓包、浏览器 profile、`tabbit-cookie.txt` 和未审查输出仍只留本机，不进入仓库。
- `stateDir/fixtures/protocol-probes/*.json` 必须是 sanitizer 写出的 protocol probe fixture，不能手工粘贴 raw 请求/响应。
- `probe protocol --json` 打印前会再次经过 sanitizer；即使 runner 返回了 prompt、stream delta、raw frame、Bearer、cookie、session 或真实用户标识，也只能输出脱敏后的 JSON。
- `stateDir/readiness.json` 只记录 Codex / Claude Code 人工验收标记，不代表 endpoint 自动校准成功。
- `fixtures audit` 负责证据覆盖；`readiness` 负责上线阻塞快照；两者都只读本地状态，不触发真实网络。
- 外部 `TABBIT_POOL_STATE_DIR` 可读但不可写时，可以设置 `TABBIT_POOL_PROTOCOL_FIXTURE_DIR` 到独立可写目录承接脱敏 fixture；账号、secret 和 `readiness.json` 仍来自 `TABBIT_POOL_STATE_DIR`。

## 总体原则

- 不猜真实 endpoint：所有真实路径必须来自本机抓包或 `probe protocol` 脱敏 fixture。
- 不提交 raw secret：cookie、session、token、验证码、真实 API key 只留在本地 state/secret store。
- 先 fixture，后 runtime：每次调整签名、请求体、响应解析前，先保存脱敏样本并补回归测试。
- 先本地回环，后客户端：先确认 `/health`、`/v1/models`、`/v1/chat/completions`、`/v1/responses`、`/v1/messages` 本地 curl/fetch 可用，再跑 Codex / Claude Code。

## 1. 真实 Tabbit send/session endpoint / 签名 / 请求体校准

当前已校准：

- 页面域名与协议 base URL：`https://web.tabbit.ai`。
- 真实发送接口：`POST /api/v1/chat/completion`。
- 真实聊天会话创建：`POST /newtab` Next Server Action `00b19386a3892f62370bef2ffacfbd5b58580fcb2a`，body 为 `[]`，返回 React Flight action result 中的 `chat_session_id`。
- 真实会话验证接口：`GET /api/v0/user/base-info`。
- 模型目录接口：`GET /proxy/v1/model_config/models?a=0&scene=chat`。
- 额度 usage 查询接口：`GET /api/commerce/quota/v1/usage?user_id=<user_id>`。
- 只读 commerce 状态/资源接口：
  - `GET /api/commerce/activity/v1/lottery/me`
  - `GET /api/commerce/activity/v1/newbie-exploration/me`
  - `GET /api/commerce/placement/v1/resources?placement_code=home.input_below`
  - `GET /api/commerce/reward/v1/card-records?user_id=<user_id>`
  - `GET /api/commerce/lottery/v1/hit-records?user_id=<user_id>`
- M05 显式 probe 接口：
  - `GET /api/commerce/activity/v1/sign-in/status?scene_codes=desktop_pet`
  - `POST /api/commerce/activity/v1/sign-in`
  - `GET /api/commerce/benefit/v1/coupon/list?user_id=<user_id>&coupon_type=weekly_reset_coupon`
  - `POST /api/commerce/activity/v1/participate`
  - `GET /api/commerce/product/v1/sku/usage-reset-coupon`
  - `GET /api/commerce/lottery/v1/available-chances?user_id=<user_id>&activity_id=<activity_id>`
  - `GET /api/commerce/lottery/v1/active-main-pools?activity_id=<activity_id>`
  - `GET /api/commerce/activity/v1/lottery/chance-records?activity_id=<activity_id>`
  - `POST /api/commerce/lottery/v1/draw`
- `Default` 是当前账号可用的 `free_unlimited` 模型，`tabbit/priority` 会映射到 `Default`。
- 浏览器签名公式：`x-nonce = HMAC-SHA256(signKey, "${timestamp}.${x-signature}.${sha256(bodyText)}")`，其中 `x-signature` 为 UUID/randomUUID 风格字符串。
- `GET /api/v0/user/base-info` 成功响应包含 `success:true` 和 `user_info.id`；无效 cookie 返回 401 和登录提示，归类为 `login_required` / `accountStatus:"login_expired"`。
- quota usage 成功响应包含 `member_level`、`usage_percentage`、`remaining_reset_hours`、`current_cycle_start`、`current_cycle_end`、`unused_reset_coupon_count` 等字段；请求需要已登录 Cookie 和 `unique-uuid`，当前 evidence 未观察到 `x-signature` / `x-nonce`。
- 只读 commerce GET 需要已登录 Cookie、`accept: application/json`、`unique-uuid` 和可选 `x-req-ctx`，当前 evidence 未观察到 `x-signature` / `x-nonce`；`newbie-exploration.view_mode` 合法值为 `event_gate`、`float_collapsed`、`float_expanded`、`activity_page`；`placement_resources` 已验证默认 `placement_code=home.input_below`。
- M05 side-effect probe 来自生产 chunk 与 live invalid-body/invalid-business 回放：签到/抽奖类 POST 使用 `trace-id + Content-Type`；reset-coupon activity participate 使用 `x-req-ctx + unique-uuid + Content-Type`；当前 evidence 未观察到 `x-signature` / `x-nonce`。这些 POST 在实现中要求 `confirmSideEffect:true`；其中每日签到已验证，显式配置 `TABBIT_POOL_PROTOCOL_SIGN_IN_PATH` 后可由默认 `maintain` 自动执行，其它 POST 不会被默认 `maintain` 自动调用。
- 消息内已上传附件引用结构：document 使用 `{ type:"document", title, content:"", metadata:{ file_id } }`，image 使用 `{ type:"image", title, content, metadata:{ file_id, source_url? } }`。
- 附件上传真实链路：`POST /proxy/v0/cos/presigned-upload-url`、`PUT <presigned COS url>`、`POST /api/v0/cos/complete-upload`；完整配置两个 Tabbit path 后，runtime 会在真实发送前自动上传 raw/base64 附件并引用 file id。
- `/session/:id` + Next action 是会话分页/history，不是发送接口。

### 1.1 准备账号和状态目录

确认账号已经导入到本地账号 store，并且账号有 `cookieJarRef` 指向本地 secret。真实注册/登录 endpoint 未还原前，先用已登录账号导入：

~~~powershell
# 推荐：把浏览器/客户端里的 Cookie header 保存到文件，避免 shell history 留 cookie
node bin/tabbit-pool.js accounts import-session --id acct_default --email user@example.test --access-tier pro --cookie-file .\tabbit-cookie.txt --json

node bin/tabbit-pool.js accounts list --json
node bin/tabbit-pool.js accounts probe acct_default --json
~~~

`--chat-session-id` 是可选覆盖项；它会写入账号元数据，作为真实发送接口的默认 `chat_session_id`，CLI 输出会移除该字段。默认协议配置也可在发送前自动创建新聊天会话，因此普通导入只需要有效 cookie/session。仅在需要复用特定浏览器会话时使用已登录页面或脱敏成功请求形状里的真实 session uuid，不要随机生成。

如果 `accounts probe` 返回 `session_missing`，先刷新或重新导入 session。

旧的代理方案只适用于需要代理的网络环境；如果仍使用 Node 直连探针，也应显式配置同一个本机代理：

~~~powershell
$env:NODE_USE_ENV_PROXY = "1"
$env:HTTPS_PROXY = "http://127.0.0.1:7890"
$env:HTTP_PROXY = "http://127.0.0.1:7890"
~~~

`TABBIT_POOL_PROTOCOL_ENABLED=true` 会填充已校准公共默认值：`TABBIT_POOL_PROTOCOL_BASE_URL=https://web.tabbit.ai`、`TABBIT_POOL_PROTOCOL_SIGN_KEY_PATH=/chat/sign-key`、`TABBIT_POOL_PROTOCOL_MODEL_CATALOG_PATH=/proxy/v1/model_config/models`、`TABBIT_POOL_PROTOCOL_SEND_PATH=/api/v1/chat/completion`、`TABBIT_POOL_PROTOCOL_SESSION_VERIFY_PATH=/api/v0/user/base-info` 和当前默认 `TABBIT_POOL_PROTOCOL_REQ_CTX`。`TABBIT_POOL_PROTOCOL_FETCH_TRANSPORT` 默认仍是 `node`，需要显式设为 `powershell` 才会使用 .NET 出站栈。未启用协议 flag 且未配置对应 path 时，`accounts probe` 仍会返回 `protocol_missing` 或 skipped，这是安全行为。

session lifecycle evidence 单独审计：

~~~powershell
node bin/tabbit-pool.js fixtures audit --scope session --json
~~~

该命令只读本地脱敏 protocol probe fixture，检查成功 `verifySession`、上游 401/login_required 过期 evidence，以及显式脱敏 session recovery strategy evidence。`session_missing` 只说明本地 secret 不存在，会单独计数，但不计为上游过期。输出包含 `lifecycle` 时间窗口、`manualCookieOperations` 和 `recoveryStrategy`；在捕获安全刷新或 re-auth 路径前，`automatedRefresh` 保持 `not_calibrated`，并以 `automated_session_refresh_strategy` 让 session scope 继续 blocked。成功/过期 lifecycle coverage ready 时，`manualCookieOperations.status` 可为 ready，表示当前手动 cookie 版本可以观测失效并要求人工重新导入；`manualCookieOperations.blockingMissing` 只列当前手动 cookie 发布缺口，`manualCookieOperations.backlogMissing` 则保留 `automated_session_refresh_strategy` 等后续增强，这不代表自动恢复策略已可用。恢复策略 evidence 必须显式声明安全、已脱敏、不含 raw payload，并提供正整数 `observedWindowMs`、`resultHash` 脱敏哈希、`expiredBeforeRecovery:true` 和 `recoveredVerifySession:true`，marker-only 恢复策略不会解除 blocked。

如果本地已有 `recoverSession` / `session_recovery_strategy` 候选 fixture 但缺少上述 proof 字段，`fixtures audit --scope session --json` 会把它计入 `counts.rejectedRecoveryStrategyEvidence`，plain 输出为 `recovery_strategy_rejected`。该计数只用于定位 evidence 质量问题，不满足 `automated_session_refresh_strategy`，也不代表存在可用自动刷新路径。

`fixtures audit --scope session` 的 plain 输出会打印 `session_lifecycle	last_successful_at=<iso>	last_expired_at=<iso>	observed_window_ms=<ms>`。该行只暴露成功 verifySession 与 401/login_required 过期 fixture 的聚合观察窗口，用来帮助估算 cookie 过期窗口；它不包含 raw fixture 内容，也不满足 `automated_session_refresh_strategy`。

同一 plain 输出还会打印 `manual_cookie_mode	<status>	mode=manual_reimport_then_probe	expired_session_action=login_expired_then_manual_reimport	automated_refresh_required=false	release_blocking_missing=<csv>	backlog_missing=<csv>`。这行是当前手动 cookie 运维目标的摘要：登录态失效后标记 `login_expired`，人工重新导入 cookie/session；`release_blocking_missing` 是当前版本阻塞项，`backlog_missing=automated_session_refresh_strategy` 表示自动刷新仍是后续项。

`automated_session_refresh_strategy` 的 doctor capture command 只提供离线 evidence 模板和校验：

~~~powershell
node bin/tabbit-pool.js probe template --operation recoverSession --json > .\probe-recoverSession.json
node bin/tabbit-pool.js probe validate --operation recoverSession --input-file .\probe-recoverSession.json --json
node bin/tabbit-pool.js probe validate --operation recoverSession --input-file .\probe-recoverSession.json --write-fixture --json
~~~

`recoverSession` 不是网络探针 operation；`probe protocol --operation recoverSession` 会被 CLI 拒绝。`--write-fixture` 只写入 `session_recovery_strategy` 的白名单 evidence/result 字段，不写入 input 文件中的 raw cookie/session/token/prompt。只有当安全 re-auth 或 refresh 路径的脱敏 `session_recovery_strategy` / `recoverSession` fixture 被写入 stateDir，并通过 `fixtures audit --scope session --json` 计入 `recoveryStrategyEvidence` 后，session 恢复策略才可视为 ready。

### 1.4 生成最小 sendMessage 探针输入

~~~powershell
node bin/tabbit-pool.js probe template --operation sendMessage --json > .\probe-sendMessage.json
~~~

真实上游 stream boundary 可以直接生成三种安全捕获骨架，避免手工拼错 `streamEvidence`：

~~~powershell
node bin/tabbit-pool.js probe template --operation sendMessage --stream-evidence error_frame --json > .\probe-sendMessage-error-frame.json
node bin/tabbit-pool.js probe template --operation sendMessage --stream-evidence cancel_after_first_delta --json > .\probe-sendMessage-cancel.json
node bin/tabbit-pool.js probe template --operation sendMessage --stream-evidence first_token_backpressure --json > .\probe-sendMessage-backpressure.json
~~~

这些模板仍只是占位 input。sendMessage 模板默认使用 `<redacted-message-content>`；`probe validate` 可接受该占位模板用于形状预检，但真实 `probe protocol --operation sendMessage` 会在调用 runner 前拒绝未替换的占位内容。运行真实 probe 前必须替换为已确认可公开的最小测试内容，审查整个 `<redacted-input.json>`，并先执行 `probe validate --operation sendMessage --input-file <redacted-input.json> --json`。validate JSON 中的 `sendMessageReview.protocolDispatchReady:true` 只表示至少存在一条非占位消息字符串；若仍为 `false`，不要执行真实 probe。最终只保存 sanitizer 写出的脱敏 fixture。

默认模板：

~~~json
{
  "model": "tabbit/priority",
  "messages": [
    { "role": "user", "content": "<redacted-message-content>" }
  ],
  "stream": true
}
~~~

`stream:true` 是默认值，用于让同一 input 文件可直接进入真实上游 error-frame、cancellation 和 backpressure 捕获流程；如果只需要非流式文本探针，可人工改为 `stream:false`。也可以加 `tools` 或工具回合输入，但第一次校准建议从最小文本请求开始。

要显式捕获真实上游 stream boundary evidence，可在确认消息内容安全后使用上述 `probe template --operation sendMessage --stream-evidence <mode> --json` 生成 input，或给同一 input 增加 `streamEvidence`，并先跑只读 validate：

~~~json
{
  "model": "tabbit/priority",
  "messages": [
    { "role": "user", "content": "<redacted-message-content>" }
  ],
  "stream": true,
  "streamEvidence": {
    "mode": "first_token_backpressure",
    "maxDeltas": 2
  }
}
~~~

~~~powershell
node bin/tabbit-pool.js probe validate --operation sendMessage --input-file .\probe-sendMessage.json --json
~~~

`streamEvidence` 是 opt-in 捕获开关：`mode` 只允许 `first_token_backpressure`、`cancel_after_first_delta`、`error_frame`，`maxDeltas` 只能是 1 到 5。runner 只在真实 restored async 结果已有 `upstreamEvidence:{ source:"tabbit-live", real:true, stream:true }` 且 `streamDeltas` 为 async iterable 时消费 bounded delta；fixture 只保存 `backpressure`、`firstTokenFlush`、`delayedSecondChunk`、`cancellation` 或 `streamErrorFrame` 聚合布尔 marker，不保存 raw stream、prompt、payload、cookie、session、token 或真实用户数据。没有可靠真实 marker 时不要用该机制声称 upstream evidence ready。

请求 `streamEvidence` 后必须捕获到目标边界才算 probe 成功：没有真实 async upstream marker、`first_token_backpressure` 少于两段 delta、`cancel_after_first_delta` 没拿到首段 delta，或 `error_frame` 未观察到 stream error，都会以 `STREAM_EVIDENCE_NOT_CAPTURED` 写入脱敏 failed fixture。该失败可用于排查捕获前置条件，但不满足 `real_upstream_*_fixture` readiness。

`fixtures audit --scope upstream` 会把这类失败样本汇总为 `counts.streamEvidenceNotCaptured`，plain 输出为 `stream_evidence_not_captured`。这个数字只用于区分“未尝试捕获”和“已尝试但证据不足”，不会让错误帧、取消或 backpressure 任一 coverage 变为 ready。

如果从 `readiness doctor` 继续执行，优先复制对应 `capture_command` 的 `template=` 命令生成 `<redacted-input.json>`；该命令已经带 `--stream-evidence`。plain 行中的 `stream_evidence` 仍会提示模式：error-frame 用 `error_frame`，取消用 `cancel_after_first_delta`，backpressure/first-token flush 用 `first_token_backpressure`；plain 行中的 `review=replace_redacted_message_content` 表示必须先替换 sendMessage 占位消息。先运行 `probe validate --operation sendMessage --input-file <redacted-input.json> --json`，确认只输出字段摘要，并检查 `sendMessageReview.redactedMessageContentPresent:false` 与 `sendMessageReview.protocolDispatchReady:true`；替换成确认可公开的最小测试内容后，才执行真实 `probe protocol --write-fixture`。未替换占位内容会被 CLI 拒绝，validate 通过也不代表已获得真实上游 evidence。

`cancel_after_first_delta` 会在第一段 delta 后主动请求 iterator `return()`，用于证明下游取消路径可被 fixture 表达；`error_frame` 会把 async stream 抛错记录为 failed fixture 和 `streamErrorFrame:true`，但错误 message 使用固定安全描述，不能把上游 raw frame、prompt 或 delta 文本写入 fixture。

`fixtures audit` 会检查上线前五类证据是否齐全：成功 `verifySession`、成功 `sendMessage`、流式文本、工具调用或明确不支持原生工具字段的证据、403/forbidden 响应。它只读本地脱敏 fixture，不触发网络，也不输出 raw payload。

`streamEvidence` 只是帮助真实 async stream probe 把已观察到的边界行为转成聚合 marker，不能替代真实 Tabbit 样本本身。

仅有 `kind:"protocol_probe"` 的流式样本也不满足真实上游 marker 要求；它必须额外携带脱敏 Tabbit/live/upstream 来源、`upstreamEvidence.real:true` 或 `result.raw.upstream:true` 等明确真实上游标记，且不得包含 raw stream、prompt、payload、cookie、session、token 或真实用户数据。泛 `source:"protocol-client"` / `evidenceSource:"protocol"` 只能说明协议客户端路径，不计为真实 Tabbit 上游证据。真实 `/api/v1/chat/completion` async SSE/NDJSON 结果现在会自动附加安全 `upstreamEvidence` marker，`probe protocol --write-fixture` 写入后可被 upstream scope 识别；旧显式 sendPath、本地 HTTP route、compat handler 和 fake stream 仍不会因为 `kind:"protocol_probe"` 被计为真实上游 evidence。

当前已用本地 `acct_default` 和 `Default` 模型拿到成功 `sendMessage` fixture，并先保存到 `output/` 下的临时 state，未提交 raw 抓包、cookie 或本地 fixture。后续若要把 fixture 纳入仓库回归，应只提交脱敏后的最小样本。

## 3. 是否需要网关本地工具执行 loop

当前结论：**先不默认做本地 loop，优先保持官方协议结构；需要时用受控 opt-in loop。**

运行时策略由 `TABBIT_POOL_TOOL_LOOP_MODE` / `config.compat.toolLoopMode` 表达，默认 `client_executes_tools_first`。可显式设为 `disabled`，表示剥离工具字段；也可设为 `local_executes_tools`，但只有 gateway 宿主注入 `executeLocalToolUse` / `localToolExecutor.execute` 时才会执行本地工具。真实 `/api/v1/chat/completion` 原生工具字段仍返回 `TOOL_FIELDS_UNSUPPORTED`。本地执行 guardrails 来自 `TABBIT_POOL_LOCAL_TOOL_ALLOWLIST`、`TABBIT_POOL_LOCAL_TOOL_MAX_ROUNDS`、`TABBIT_POOL_LOCAL_TOOL_TIMEOUT_MS`、`TABBIT_POOL_LOCAL_TOOL_MAX_RESULT_CHARS`。

| 场景 | 是否需要本地 loop | 理由 |
|---|---|---|
| Codex / Claude Code 自己执行工具 | 不需要 | 客户端会收到 tool call、执行工具、再把 tool result 发回；网关只要保真即可。 |
| 普通 SDK 调用方自己执行 function | 不需要 | 与 OpenAI / Anthropic 官方 API 使用方式一致。 |
| 客户端只会发 tools，但不会执行工具 | 需要 | 使用 `local_executes_tools`，网关 wrapper 执行注入工具并继续多轮，否则会停在 tool call。 |
| Assistants 风格 requires_action | 视接口而定 | 根项目已有 Assistants 工具流程，可作为后续迁移参考。 |
| 想内置受控工具，例如 fetch / js / shell | 需要但要单独设计 | 当前只提供注入 executor 和 guardrails，不内置任何高风险工具；内置工具仍需单独安全设计。 |

本地 loop 的验收口径：

1. 工具白名单明确，不执行未知工具。
2. 总轮数有限制，默认 4，可通过 env 覆盖。
3. 单工具执行可配置超时，超时返回 `timeout/LOCAL_TOOL_TIMEOUT`。
4. 工具输出会截断；具体脱敏由宿主 executor 或外围日志层负责。
5. OpenAI Chat、Responses、Anthropic Messages 三套入口经同一 runner wrapper 覆盖测试。
6. 客户端已有工具执行能力时，可以关闭本地 loop，避免重复执行。

## 4. 真实 403 响应体 fixture 细分

当前代码新增 `classifyForbiddenSignal()`，把 403 细分为：

| kind | 典型信号 | 建议账号动作 |
|---|---|---|
| `risk_control` | risk、abuse、suspicious、captcha、security、风控、人机验证 | `suspect`，隔离账号，避免立即重试。 |
| `signature_or_protocol` | signature、sign-key、timestamp、nonce、hmac、payload、签名 | `protocol_probe`，优先校准签名/请求体，不轮询账号。 |
| `session_or_cookie` | cookie、session、login、auth、expired、token、登录、会话 | `login_expired`，刷新或导入 session。 |
| `entitlement_or_model` | premium、pro、plan、tier、permission、model、权限、权益 | `cooldown` 或切换模型/账号权益。 |
| `unknown_forbidden` | 只有 Forbidden 或无结构 body | `suspect`，先抓 fixture。 |

CLI 可直接带上下文获取建议，输出会脱敏：

~~~powershell
node bin/tabbit-pool.js probe advice --status 403 --message "<403 response message>" --json
~~~

输出示例：

~~~json
{
  "category": "forbidden",
  "severity": "error",
  "recommendation": "Isolate this account...",
  "forbidden": {
    "kind": "risk_control",
    "accountAction": "suspect",
    "retryable": false
  }
}
~~~

真实 403 fixture 校准时必须记录：

- HTTP status
- response body 的脱敏 `code/message/reason/type`
- 当前 operation：`sendMessage` / `verifySession` / `listModels`
- 是否同账号复现
- 是否多账号复现
- 是否同一签名 payload 可复现
- 账号状态动作建议

## 完成定义

当前手动 cookie 运维可用版本的口径是：用户手动导入 cookie/session，基础 gateway/chat readiness 满足，`verifySession` 成功与 401/login_required 过期 evidence 均存在，`manualCookieMode.status` 为 ready 且 `manualCookieMode.blockingMissing:[]`；自动注册、Yoda/短信登录和自动 session refresh 不在当前完成定义内。`manualCookieMode.backlogMissing:["automated_session_refresh_strategy"]` 可以继续存在，表示后续自动恢复增强尚未校准。

完整真实协议校准继续以以下 4 项全部完成为准：

- 至少一个真实 `verifySession` 成功 fixture 进入回归测试。
- 至少一个真实 `sendMessage` 成功 fixture 进入回归测试。
- 至少一个真实流式文本 fixture 进入回归测试。
- 至少一个真实工具调用 fixture 或明确“Tabbit 上游不支持原生工具字段”的 fixture 进入回归测试。
- Codex 完成一次代码任务，并记录配置与结果，同时 `readiness mark --codex-verified` 已写入本地验收标记。
- Claude Code 完成一次代码任务，并记录配置与结果，同时 `readiness mark --claude-verified` 已写入本地验收标记。
- 403 至少有一个真实响应体 fixture，并能归入上述 kind 之一。
- 如启用默认权益维护，quota usage 和每日签到 maintain 结果已使用脱敏 evidence 验证；只读 commerce 探针和其它 M05 side-effect probe 可通过 `probe protocol` 单独执行；POST 型 side-effect operation 需要 `confirmSideEffect:true`，默认 maintain 仍不触发活动 Pro 领取、抽奖或真实用券。
- 文档、测试、API 兼容层同步更新，`npm test` 通过。

# 01-需求文档

## 需求总览

| 优先级 | 模块 | 用户价值 | 验收口径 |
|---|---|---|---|
| P0 | Tabbit 协议客户端 | 去掉浏览器 UI 运行时依赖 | 纯 Node HTTP 请求完成一次模型回复 |
| P0 | 账号池调度 | 多账号分摊额度 | 两个账号轮询，可识别失败并切换 |
| P0 | 配置与密钥管理 | 避免密钥泄漏 | 所有 secrets 只在 env/local store 中存在 |
| P0 | 协议回归样本 | 降低 Tabbit 前端变更风险 | 固定样本可验证签名、响应解析和错误分类 |
| P1 | YYDS 邮箱接码 | 自动读取注册验证码 | 创建邮箱、轮询邮件、提取验证码 |
| P1 | 账号注册初始化 | 降低手动注册成本 | 用邮箱验证码完成账号注册/登录并保存 session |
| P1 | 权益额度维护 | 自动使用活动权益 | 签到、领取 Pro、查询/重置额度可跑通 |
| P1 | 兼容网关 | 复用现有客户端生态 | OpenAI Chat/Responses、Anthropic Messages 可用，官方工具字段可透传 |
| P2 | 观测运维 | 便于排错与长期运行 | 可查看账号状态、额度、错误原因 |

## 用户场景

### 场景 A：日常模型调用

用户启动本地网关后，Codex/Claude Code 访问本地 API。系统自动选择一个可用 Tabbit 账号完成请求。

### 场景 A1：编码工具携带工具定义与工具结果

Codex、Claude Code 或三方 SDK 按官方 API 风格发送 `tools` / `tool_choice`。兼容层必须保留这些字段并传到协议请求体，避免编码工具因为请求字段被吞掉而退化；模型返回 tool call 时先映射为官方兼容输出；客户端执行工具后继续提交的 OpenAI `tool` message / Responses `function_call_output` / Anthropic `tool_result` 也必须保留结构并进入协议请求体。默认由客户端执行工具；如果宿主显式启用 `local_executes_tools` 并注入本地 executor，网关可用受控本地 loop 执行白名单工具并继续多轮。

### 场景 B：账号额度耗尽

某账号返回额度耗尽或模型不可用时，账号池将其标记为 `quota_exhausted` 或进入冷却，并切换到下一个账号。外部客户端最多感知一次正常模型响应或一个兼容错误，不需要知道内部账号 ID。

### 场景 C：每日维护

系统定时执行签到、权益领取、额度刷新、重置券检查。若发现账号额度耗尽且有重置券，可自动使用并恢复账号状态。

### 场景 D：新账号初始化

系统使用 YYDS Mail 创建邮箱，调用 Tabbit 注册流程，读取验证码，完成注册、登录、领取活动 Pro，并写入账号池。

## 功能需求

### F1 协议直连

- 获取并刷新 `/chat/sign-key`。
- 复刻 `x-timestamp`、`x-nonce`、`x-signature`。
- 还原消息发送请求体和流式响应协议。
- 支持模型目录查询与模型能力映射。
- 支持附件时，明确上传接口和引用结构；未知时先拒绝附件并返回兼容错误。

### F2 账号池

- 支持多账号配置与持久化。
- 每个账号维护 session/cookie、用户 ID、权限、额度、冷却状态。
- 支持 round-robin、健康优先、额度优先策略。
- 支持失败分类：登录失效、额度耗尽、权益不足、风控、网络错误、协议变更。
- 单个请求最多跨账号重试一次或按配置限制，避免循环消耗额度。

### F3 邮箱接码

- 通过 `X-API-Key` 调用 YYDS Mail。
- 创建临时 inbox。
- 轮询邮件列表。
- 读取邮件详情/源码。
- 从主题或正文提取验证码。
- 对 `429` 使用 `Retry-After` 控制退避。

### F4 账号注册初始化

- 生成邮箱。
- 请求发送验证码。
- 读取验证码。
- 提交注册或验证码登录。
- 保存账号 session。
- 首次权益领取与额度刷新。

### F5 权益与额度维护

- 查询当前权益/Pro 状态。
- 领取活动 Pro。
- 每日签到。
- 查询额度。
- 查询重置券。
- 额度耗尽时按策略使用重置券。

### F6 兼容网关

- 对外提供 `/health`。
- 对外提供 `/v1/models`。
- 对外提供 `/v1/chat/completions`。
- 对外提供 `/v1/responses`。
- 可选提供 Anthropic `/v1/messages`。
- OpenAI Chat/Responses 支持透传 `tools`、`tool_choice`、`parallel_tool_calls`。
- Anthropic Messages 支持透传 `tools`、`tool_choice`。
- 基础工具调用输出映射：OpenAI Chat `tool_calls`、OpenAI Responses `function_call`、Anthropic `tool_use`。
- 工具调用流式兼容：Chat SSE 输出 `tool_calls` delta，Responses SSE 输出 function_call item/arguments events，Anthropic SSE 输出 `tool_use` / `input_json_delta`；buffered 上游 OpenAI stream `tool_calls` 可聚合为内部工具调用，async 上游工具 delta 可直接转换为本地 chunked SSE 工具事件。
- HTTP 路由层只负责认证、JSON 解析和响应写回，兼容语义由 OpenAICompat/AnthropicCompat 处理。
- 客户端只看到统一模型名和兼容错误，不暴露账号细节。

## 非功能需求

- **本地优先**：默认监听 `127.0.0.1`，不新增公网部署路径。
- **可回归**：协议还原必须有录制样本、fixture 和单元测试。
- **可维护**：所有上游错误先归类，再映射到 OpenAI/Anthropic 错误。
- **可恢复**：session 过期、签名失效、额度耗尽都要能更新状态并给出下一步动作。
- **可脱敏**：日志、状态导出、测试 fixture 不保存真实 cookie、token、API key。

## 产品取舍

- 先做协议发信闭环，再做注册自动化；否则无法证明脱离 UI 的核心链路成立。
- 先用本地 JSON 元数据 + secret 引用存储账号池，暂不做 Web 后台；排障和协议校准优先使用 `tabbit-pool` CLI、`/health` 与脱敏 fixture。
- 先支持单机本地运行，暂不考虑分布式账号池。
- 对于不确定的 Tabbit 接口，先记录抓包样本和契约，不在文档中伪造已验证结论。

## 待确认问题

1. 新项目最终名称是否沿用 `tabbit-protocol-pool`？
2. 账号凭据存储用 SQLite、JSON 加密文件，还是系统密钥链？
3. 注册自动化需要支持哪些邮箱域名策略：随机临时域名、固定 localPart、还是自定义域名？
4. 是否需要和现有 Tabbit2API 合并，还是保持独立服务？
5. 账号池重试上限是否按请求配置，还是固定最多切换 1 次？
6. 真实 Tabbit 原生工具字段已校准为不支持；后续需要继续校准的是真实上游工具语义是否存在其他私有协议，或是否只保留本地工具 loop 方案。

## 脱敏接口

### redactSensitiveValue(value)

对字符串中的敏感片段做不可逆展示脱敏。当前覆盖：

- AC- 开头的 YYDS Mail key。
- Authorization: Bearer token。
- cookie、session、token 类键值。
- 邮箱 localPart。
- 英文 verification code / code 与中文验证码后面的数字。

### redactObject(value)

递归复制对象或数组并脱敏敏感字段，不修改输入对象。字段名匹配 apiKey、authorization、cookie、session、token、verification、code、password、secret 时会强制脱敏其字符串值。

### fingerprintSensitiveValue(value)

返回 sha256: 前缀的 8 位稳定指纹，用于日志相关性，不用于还原原文。

### createInbox(input)

请求 POST /accounts，使用 X-API-Key，body 只包含非空 localPart、domain、subdomain。返回归一化对象：

~~~ts
type Inbox = {
  id: string;
  address: string;
  tempToken?: string;
  expiresAt?: string;
};
~~~

### listMessages(address)、getMessage(id, address)、getSource(id, address)

分别请求：

~~~text
GET /messages?address=...
GET /messages/{id}?address=...
GET /sources/{id}?address=...
~~~

标准错误 envelope success:false 会变成 MailProviderError；429 会读取 Retry-After 并写入 retryAfterMs。

### waitForVerificationCode(input)

~~~ts
type WaitForVerificationCodeInput = {
  address: string;
  timeoutMs?: number;   // 默认 60000
  intervalMs?: number;  // 默认 2000
};
~~~

轮询邮件列表，优先读取消息详情，再读取原始源码。找到唯一验证码时返回 { code, source, message }。超时抛 MailProviderError(MAIL_TIMEOUT)，验证码不唯一抛 MailProviderError(AMBIGUOUS_CODE)。

### extractVerificationCode(message)

依次从 subject、text/body、html 转文本、raw/source 中提取 4 到 8 位数字验证码。单一命中返回 { code, source }；无命中抛 CODE_NOT_FOUND；同一来源多命中抛 AMBIGUOUS_CODE。

### canonicalJson(value)

稳定 JSON 序列化。对象 key 递归排序，undefined 字段被跳过。签名 payload 依赖该函数。

### buildSignaturePayload(input) 与 createSignedHeaders(input)

当前实现使用 HMAC-SHA256 作为可测试默认骨架，payload 格式：

~~~text
METHOD
/path?sorted=query
timestamp
nonce
canonical-json-body
~~~

createSignedHeaders 默认返回旧骨架的 x-timestamp、x-nonce、x-signature；当传入 `bodyText` 或 `signature` 时，使用已校准的浏览器签名分支：

~~~text
x-signature = signature
x-timestamp = timestamp
x-nonce = HMAC-SHA256(signKey, `${timestamp}.${signature}.${sha256(bodyText)}`)
~~~

真实 `/api/v1/chat/completion` 分支会走该浏览器算法；旧 method/path/query/canonical body 骨架仍保留给未校准显式路径和附件上传骨架。

### getSignKey({ force })

请求 signKeyPath 并缓存 key。响应可以是纯字符串，也可以是 key/signKey/data.key/data.signKey 结构。非 2xx、空 key 或解析异常会抛 ProtocolTabbitError。

### listModels({ force })

请求 modelCatalogPath，并固定携带 query `a=0` 与默认 `scene=chat`。返回归一化模型列表，第一项始终是 `tabbit/priority`。

### normalizeModelCatalog(body)

支持数组、`models`、`data`、`items` 多种上游包装结构。真实 Tabbit 目录使用 `models/status` 包装和 `display_name`、`supports_images`、`supports_tools`、`model_access_type` 字段。输出字段包括 id、selectedModel、displayName、tabbit_display_name、supports_tools、supports_images、model_access_type、available_in_tabbit_catalog；`tabbit/priority` 会映射到当前可用的 `Default` 发送模型。

### verifySession(input)

可配置真实 session verifier。输入：`{ account?, session? }`。构造参数未配置 `sessionVerifyPath` 时返回 `{ ok:false, category:"protocol_missing", code:"MISSING_SESSION_VERIFY_PATH", accountStatus:"suspect" }` 且不触网。配置后如果没有 `session`、`account.cookie` 或 `account.cookieHeader`，返回 `{ ok:false, category:"session_missing", code:"SESSION_MISSING", accountStatus:"login_expired" }`，也不触网。会话材料存在时会调用 `getSignKey()`，对 `sessionVerifyMethod` / `sessionVerifyPath` 生成签名头，附带 `Cookie`，并请求 `baseUrl + sessionVerifyPath`。真实校准路径是 `GET /api/v0/user/base-info`；成功响应中的 `user_info.id` 会归一化为 `userId`。2xx 响应归一化为 `{ ok:true, userId?, accessTier?, raw }`；401 等错误返回 `{ ok:false, category, code?, message, httpStatus, accountStatus, error }`。

### sendMessage(input)

~~~ts
type SendMessageInput = {
  account?: { cookie?: string; cookieHeader?: string };
  model: string;

  stream?: boolean;
~~~

当前行为：

- 真实 `/api/v1/chat/completion` 分支中，attachments 非空时优先识别已上传附件引用，至少提供 `path`、`file_id`、`fileId`、`id` 或 `metadata.file_id` 之一；document 会映射为 `{ type:"document", title, content:"", metadata:{ file_id } }`，image 会映射为 `{ type:"image", title, content, metadata:{ file_id, source_url? } }`。缺少 file id/path 但包含 `data`、`base64`、`raw` 或 `body` 的附件，在完整配置 `attachmentUploadPath + attachmentCompleteUploadPath` 时会先自动上传再发送；未完整配置时返回 `unsupported_feature/ATTACHMENT_REFERENCE_REQUIRED`。旧显式 sendPath 骨架收到附件引用仍返回 `unsupported_feature/ATTACHMENTS_UNSUPPORTED`。
- sendPath 未配置时返回 { ok:false, error: ProtocolTabbitError(MISSING_SEND_PATH) }。
- 成功时先获取 sign key，再对 body 签名，POST 到 sendPath。
- 当 `sendPath === "/api/v1/chat/completion"` 时，使用已校准真实 Tabbit 请求体和浏览器签名头。请求体包含 `chat_session_id`、`message_id:null`、`content`、`selected_model`、`parallel_group_id:null`、`task_name:"chat"`、`agent_mode:false`、`metadatas.html_content`、`references` 和空 tab `entity`。`chat_session_id` 来自 input、账号字段或 `defaultChatSessionId`；缺失时返回 `MISSING_CHAT_SESSION_ID`。`tabbit/priority` 会发送为 `selected_model:"Default"`。如果该真实分支收到 `tools`、`toolChoice` 或 `parallelToolCalls`，会直接返回 `unsupported_feature/TOOL_FIELDS_UNSUPPORTED`，不会获取 sign key 或发送请求。
- 真实分支 headers 包含 `accept:text/event-stream`、`content-type:application/json`、`cache-control:no-cache`、`x-req-ctx`、`unique-uuid`、`x-timestamp`、`x-nonce`、`x-signature` 和运行时 hydrated `Cookie`。
- 旧显式 sendPath 骨架的请求 body 基础字段为 `{ model, messages, stream }`。`messages` 不限于纯文本 `{ role, content }`，compat 可传入 OpenAI Chat 的 assistant `tool_calls` / `role:"tool"` / `tool_call_id`、OpenAI Responses 的 `function_call` / `function_call_output` input item，以及 Anthropic 的 `tool_use` / `tool_result` content block。旧显式 sendPath 会把这些结构原样写入 signed body。当 `tools` 为数组时写入 `body.tools`；当 `toolChoice` 非 null/undefined 时写入 `body.tool_choice`；当 `parallelToolCalls` 非 null/undefined 时写入布尔型 `body.parallel_tool_calls`。签名基于包含这些字段的最终 body。当前只做字段透传，不在协议客户端内执行工具。
- 响应文本可来自 text、content、message.content、message.text、data.text、data.content、choices[0].message.content。
- 响应工具调用可来自 OpenAI 风格 `tool_calls` / `choices[0].message.tool_calls` 或 Anthropic 风格 content block `{ type:"tool_use", id, name, input }`。`normalizeMessageResponse()` 会把它们归一化为内部 `{ type:"tool_use", id, name, input }` block，并把 `tool_calls` / `function_call` finish reason 映射为内部 `stopReason:"tool_use"`。
- `Content-Type: text/event-stream` 响应会被解析为 `{ kind:"stream", format:"sse", events, text }`，忽略 `[DONE]`，从 `delta`、`text`、`content`、`message.content`、`data.delta`、`data.text`、`data.content`、`choices[0].delta.content`、`choices[0].message.content` 聚合文本；buffered events 中的 OpenAI `choices[0].delta.tool_calls` 会按 index 聚合 `id/name/arguments` 并生成内部 `tool_use` block；Anthropic `content_block_start` 的 `tool_use` 与 `content_block_delta` 的 `input_json_delta.partial_json` 会按 content block index 聚合为内部 `tool_use` block。
- `Content-Type: application/x-ndjson`、`application/jsonl` 或 `application/stream+json` 会被解析为 `{ kind:"stream", format:"ndjson", events, text }`，按行保留结构化 events，并复用同一组字段聚合文本和 buffered OpenAI/Anthropic tool call delta。
- buffered 路径中，`normalizeMessageResponse()` 在可提取流式文本分片时返回 `streamDeltas:string[]`，供 compat handler 作为非公开 `stream.deltas` 元数据传给 HTTP SSE adapter。
- async 路径中，`sendMessage({ stream:true })` 在 2xx、支持的 SSE/NDJSON Content-Type 且 `response.body` 可读时立即返回 `streamDeltas:AsyncIterable<string | ToolCallDelta>`，`contentBlocks` 初始为空文本，`raw.async === true`；真实 `/api/v1/chat/completion` 分支会附加安全 `upstreamEvidence:{ source:"tabbit-live", real:true, stream:true, format }`，供脱敏 protocol probe fixture 后续进入 upstream audit marker gate。迭代时会边读 body 边追加 `raw.events`。文本 delta 以字符串 yield；OpenAI `tool_calls` 与 Anthropic `tool_use` / `input_json_delta` 会以 `{ type:"tool_call_delta", index, id?, name?, argumentsDelta? }` yield，供 HTTP 层转成 Chat/Responses/Anthropic 官方流式工具事件。如果消费端在 Web `ReadableStream` 分支调用 async iterator `return()`，协议客户端会调用底层 reader `cancel("stream_deltas_cancelled")` 后释放锁。
- stream error frame 会被识别并分类。buffered 路径会让 `sendMessage()` 返回 `{ ok:false, error }`；async 路径会在 `streamDeltas` 迭代到错误帧时抛出 `ProtocolTabbitError`，并且该 raw event 已追加到 `raw.events`。识别形状包括 SSE `event:error`、对象 `type:"error"` / `event:"error"`、`error`、`errorCode`、以及 `code + message`。
- 这些解析只发生在已显式配置 sendPath 的响应上，不猜测 endpoint。真实 `/api/v1/chat/completion` 已验证成功文本 SSE、已上传附件引用和 raw/base64 自动上传后发送；完整工具调用真实语义、更多错误帧、HTTP 客户端断连到上游取消的传播和 backpressure 仍需继续 fixture 校准。
- 响应既没有助手文本也没有工具调用时返回 protocol_changed。

### classifyProtocolError(input)

| 输入 | category | retryable | cooldownMs |
|---|---|---:|---:|
| Error 且无 status | network_error | true | 1000 |
| 401 | login_required | false | 0 |
| body/code/message 命中 quota/credit/usage 耗尽信号 | quota_exhausted | true | 0 |
| 403 | forbidden | false | 1800000 |

## AccountPool 接口

### normalizeAccount(account)

补齐账号默认值：id 字符串、status=active、accessTier=unknown、quotaState=[]、resetCouponCount=0、failureStreak=0、audit=[]。

### isAccountSelectable(account, options)

判断账号是否可选。disabled、provisioning、login_expired、quota_exhausted、suspect 默认不可选；cooldown 只有 cooldownUntil 仍在未来时不可选；requiresPremium 要求 accessTier 为 pro 或 premium。

### scoreAccount(account, options)

初始 100 分。failureStreak 每次扣 10 分；可见 quotaState.remaining 最高加 25 分；最近成功会在 10 分内按时间衰减加分。

### new AccountPool({ accounts, now })

内存账号池。需要跨请求持久化时，使用 `StoredAccountPool` 搭配 `JsonAccountStore`；纯 `AccountPool` 状态仍只保存在实例内。

### pickAccount(input)

~~~ts
type PickAccountInput = {
  model?: string;
  requiresPremium?: boolean;
  excludeAccountIds?: string[];
};
~~~

按可选性过滤，再结合 round-robin cursor 与最高 score 选择账号。无账号时抛 AccountPoolError(NO_AVAILABLE_ACCOUNT)，错误 detail 中包含候选摘要。

### recordSuccess(accountId, options)

把账号恢复 active，清空 failureStreak、cooldownUntil、lastError，写入 lastSuccessAt 和 audit success 记录。

### recordFailure(accountId, error, options)

根据错误 category 更新状态：

| category | 新状态 |
|---|---|
| login_required | login_expired |
| quota_exhausted | quota_exhausted |
| protocol_changed / forbidden | suspect |
| rate_limited / upstream_error / network_error / unknown | cooldown |

### shouldFallback(input)

只有 retryable 为 true、category 不是 protocol_changed/login_required/forbidden、retryCount 小于 retryLimit，且存在未尝试候选账号时才返回 fallback:true。

### StoredAccountPool

## LocalToolLoopRunner 接口

### new LocalToolLoopRunner(options)

| 参数 | 类型 | 说明 |
|---|---|---|
| runner | { run(input): Promise } | 必填，通常包装 PooledRequestRunner。 |
| mode | string | 默认 `client_executes_tools_first`；支持 `disabled` 和 `local_executes_tools`。 |
| executeToolUse | function/null | `local_executes_tools` 执行工具所需的宿主注入 executor。 |
| maxRounds | number | 默认 4，必须为正安全整数。 |
| maxToolResultChars | number | 默认 16000，工具输出超限会截断。 |

| toolTimeoutMs | number | 默认 0；大于 0 时，单个工具执行超过该毫秒数会返回 timeout。 |

### run(input)

`client_executes_tools_first` 是默认兼容策略，保持原生工具字段透传给底层 runner；`disabled` 会把 `tools`、`toolChoice`、`parallelToolCalls` 清为 `null` 后调用底层 runner。

`local_executes_tools` 只在请求带非空工具定义且注入 executor 时执行本地 loop。它会把工具定义转成 system 文本约束，剥离真实协议不支持的原生工具字段，把 `stream` 强制为 false 调用底层 runner，解析直接 `contentBlocks[].type:"tool_use"` 或 JSON 文本 tool_use/tool_calls envelope，调用 executor 后追加 `{ role:"tool", tool_call_id, content }` 继续下一轮。默认不内置 shell、web、js、fetch 等工具；请求工具定义超出 `allowedToolNames`、或模型请求未知工具名时返回 `LOCAL_TOOL_NOT_ALLOWED`，缺少 executor 返回 `LOCAL_TOOL_EXECUTOR_MISSING`，缺少工具定义但 `toolChoice` 强制工具时返回 `LOCAL_TOOL_DEFINITIONS_REQUIRED`，单工具执行超时返回 `timeout/LOCAL_TOOL_TIMEOUT`，超过轮数返回 `LOCAL_TOOL_LOOP_LIMIT_EXCEEDED`。`toolChoice:"none"`、`toolChoice:"auto"` 和孤立 `parallelToolCalls` 在无工具定义时按 no-op 剥离后走普通文本。

~~~ts
    status: "success" | "skipped" | "failed";
    changed: boolean;
    detail?: string;
    error?: { message: string; code?: string; category?: string };
  }>;
};
~~~

验证码等待、协议动作抛错或协议动作返回 `{ ok:false, error }` 时保留 provisioning 账号并写入带 stage 的脱敏 lastError。注册/登录响应缺少 session material 时账号转 `suspect`。错误摘要会通过通用脱敏逻辑，并额外屏蔽裸 4 到 8 位数字，避免验证码进入 action error。

### importSession(input)

导入已有 cookie/session。输入可包含 cookieHeader、cookieJar、cookie、session、sessionToken 或 token。缺少 session material 时返回 failed action 且不写账号；secret 写入失败时也不保存 active 账号。默认 cookieJarRef 为 `secrets/<accountId>.cookie`。

### resumeProvisioning(accountId)

只处理 `status:"provisioning"` 账号。非 provisioning 返回 skipped；缺少 `protocolClient.resumeProvisioning` 返回 skipped；注入 hook 返回 `{ account }` 时保存该账号。

### verifyAccount(accountId)

读取 `cookieJarRef` 对应 secret 并调用 `protocolClient.verifySession({ account, session })`。secret 缺失时账号转 `login_expired`。verifier 成功时账号转 `active`，可更新 userId、accessTier、lastVerifiedAt；verifier 失败或抛错时写入脱敏 lastError。

## BenefitsMaintainer 接口

### normalizeQuotaState(entry, options)

把 quota fixture 或协议返回项归一化为数据字典中的 QuotaState：

~~~ts
type QuotaState = {
  model: string;
  remaining: number | null;
  limit: number | null;
  unit: string;
  resetAt: string | null;
  exhausted: boolean;
  source: string;
};
~~~

remaining/limit 可从数字字符串转换；缺失时为 null。没有显式 exhausted 时，remaining===0 视为 exhausted。

### new BenefitsMaintainer(options)

~~~ts
type BenefitsMaintainerOptions = {
  protocolClient: {
    refreshQuota?: (account) => Promise<object>;
    dailyCheckin?: (account) => Promise<object>;
    claimProIfAvailable?: (account) => Promise<object>;
    useResetCoupon?: (account) => Promise<object>;
  };
  accountStore?: {
    loadAccounts(): Promise<Account[]>;
    saveAccounts?: (accounts: Account[]) => Promise<Account[]>;
  };
  now?: () => Date | string | number;
};
~~~

protocolClient 必填，但其中单个协议方法可缺失。缺失方法对应动作返回 skipped，不会猜测真实 Tabbit endpoint。协议方法抛出明确 category 错误时，会按维护失败状态映射更新账号。accountStore 可选；传入后 `maintainAllAccounts()` 可无参数读取账号，并在有变化且存在 `saveAccounts()` 时持久化。

### refreshQuota(account)

调用 `protocolClient.refreshQuota(account)` 并更新 quotaState、resetCouponCount、accessTier、lastMaintainedAt。quota exhausted 时账号转 `quota_exhausted`；额度恢复时可转 `active`。未分类失败保留旧 quota 并返回 failed action；明确 category 失败会按维护失败状态映射写入 status、lastMaintainedAt 和 lastError。

### dailyCheckin(account)

同 UTC 日期已有 lastCheckinAt 时 skipped；否则调用 `protocolClient.dailyCheckin(account)`，成功后更新 lastCheckinAt 和 lastMaintainedAt。若协议抛出 `category:"login_required"`，账号转 `login_expired` 并写入脱敏 lastError。

### claimProIfAvailable(account)

accessTier 为 pro/premium 或 proClaimed=true 时 skipped；否则调用 `protocolClient.claimProIfAvailable(account)`，成功后更新 accessTier 与 proClaimed。

### useResetCoupon(account)

仅在 status 为 quota_exhausted 且 resetCouponCount > 0 时调用协议方法。成功后账号转 active，resetCouponCount 扣减，并可写入新的 quotaState。若协议抛出 `rate_limited` / `network_error` / `upstream_error`，账号转 `cooldown`，保留 resetCouponCount，并写入 cooldownUntil 与脱敏 lastError。

### maintainAccount(account)

按固定顺序执行 refreshQuota、claimProIfAvailable、dailyCheckin、useResetCoupon。单动作失败不会阻断后续动作。返回：

~~~ts
type MaintainAccountResult = {
  account: Account;
  changed: boolean;
  actions: Array<{
    name: string;
    status: "success" | "skipped" | "failed";
    changed: boolean;
    detail?: string;
    error?: { message: string; code?: string; category?: string };
  }>;
};
~~~

### 维护失败状态映射

只有错误对象显式携带下列 `category` 时，维护层才修改账号状态；未分类错误保持账号状态，只返回 failed action。

| error.category | account.status | 额外字段 |
|---|---|---|
| `login_required` | `login_expired` | `lastError`、`lastMaintainedAt` |
| `quota_exhausted` | `quota_exhausted` | `lastError`、`lastMaintainedAt` |
| `protocol_changed` / `forbidden` | `suspect` | `lastError`、`lastMaintainedAt` |
| `rate_limited` / `network_error` / `upstream_error` | `cooldown` | `lastError`、`lastMaintainedAt`、`cooldownUntil` |

`lastError.message` 和 action error message 都会经过脱敏；4 到 8 位验证码数字仍由观测层输出时再清理。
### maintainAllAccounts(accounts?)

批量执行账号维护。传入账号数组时只返回维护后的数组，不直接写盘；未传入账号数组时会调用构造时注入的 `accountStore.loadAccounts()`。返回：

~~~ts
type MaintainAllAccountsResult = {
  accounts: Account[];
  changed: boolean;
  results: Array<{
    accountId: string | null;
    account: Account;
    changed: boolean;
    actions: MaintenanceAction[];
  }>;
};
~~~

无参数模式下，只有任一账号 `changed:true` 时才调用 `accountStore.saveAccounts(accounts)`；全部动作 skipped/failed 且账号未变化时不会写盘。若需要持久化变化但 accountStore 缺少 `saveAccounts()`，会抛 `BenefitsMaintainerError`，code 为 `MISSING_ACCOUNT_STORE_SAVE`。

### buildProtocolFixtureAudit(input)

离线审计本地 protocol probe fixture 覆盖，返回 `{ status, observedAt, counts, coverage, missing, nextActions }`。该 helper 只使用调用方传入的 fixture 数组，不读取网络或 raw secret。默认 scope 是 `protocol`，用于现有 gateway/chat readiness；`scope:"auth"` 是注册/登录校准专用审计；`scope:"benefits"` 是 M05 side-effect 校准专用审计；`scope:"session"` 是 session lifecycle 校准专用审计；`scope:"upstream"` 是真实上游 stream boundary 校准专用审计。auth/benefits/session/upstream scope 都不改变默认 readiness 判定。

`coverage` 包含五项：

- `sessionVerify`：成功 `verifySession` fixture。
- `successfulSendMessage`：成功 `sendMessage` fixture。
- `streamingText`：成功 `sendMessage` fixture 中的 `streamDeltas`、`raw.kind:"stream"`、`kind:"stream"`、`format:"sse"` 或 `format:"ndjson"`。
- `toolCall`：成功 `sendMessage` fixture 中的 `tool_use`、`tool_call`、`function_call`、`tool_calls`，或带 `tools` 输入且明确显示上游不支持原生工具字段的 fixture。
- `forbidden403`：`adviceCategory:"forbidden"`、`error.status:403`、`httpStatus:403` 或 `statusCode:403`。

缺口名称固定为 `successful_verifySession_fixture`、`successful_sendMessage_fixture`、`streaming_text_fixture`、`tool_call_fixture`、`forbidden_403_fixture`。`tabbit-pool fixtures audit --json` 使用该 helper，并只输出聚合结果。

`scope:"auth"` 返回 `scope:"auth"`，覆盖项为：

- `authSendVerificationCode`：成功 `sendVerificationCode` fixture，且响应形状包含 send/delivery/verification/code-send 专属成功字段，例如 `codeSent:true`、`emailSent:true` 或 `sendResult:"sent"`。`counts.successfulSendVerificationCode` 只表示 send-code transport success，`counts.successfulSendVerificationCodeWithDeliverySignal` 才是该 coverage 使用的计数。
- `authSubmitRegistrationOrLogin`：成功 `submitRegistrationOrLogin` fixture，且响应形状包含 `cookieHeader`、`cookieJar`、`cookie`、`session`、`sessionToken` 或 `token` 等可供 `AccountProvisioner.extractSessionSecret()` 导入的 session material 字段。`counts.successfulSubmitRegistrationOrLogin` 只表示 submit transport success，`counts.successfulSubmitRegistrationOrLoginWithSessionMaterial` 才是该 coverage 使用的计数。

auth scope 的缺口名称固定为 `successful_sendVerificationCode_fixture` 和 `successful_submitRegistrationOrLogin_fixture`。它只用于检查注册/登录真实 evidence 是否闭环；默认 `fixtures audit` 不要求 auth evidence。

`scope:"benefits"` 返回 `scope:"benefits"`，覆盖项为：

- `dailySignIn`：成功 `dailySignIn` fixture，要求 `signInResult/sign_in_result:"success"`、`signedToday/signed_today:true` 或 `signInResult/sign_in_result:"already_signed"`。`already_signed` 只适用于每日签到幂等完成状态，不适用于 Pro、抽奖或重置券。
- `proActivitySuccess`：成功 `participateActivity` fixture，且包含明确 participation/activity/claim success 信号；不能只凭 2xx 或 `ok:true` 推断。
- `resetCouponConsumption`：真实消耗重置券 fixture，要求明确 consumed/used/deducted 或 consume result success 信号，且 operation 必须是真实消费类 operation；同时必须包含脱敏 `endpointHash`、`bodyHash`、`resultHash`、`safe:true`、`sanitized:true`、`rawPayload:false` evidence。`participateResetCouponActivity` 只作为活动参与 evidence，即使响应含 consumed/used 字段也不计为成功。
- `lotteryDrawSuccess`：成功 `drawLottery` fixture，且包含明确 `drawResult/draw_result` 或 `lotteryResult/lottery_result` success，或非空 prize、award、reward、hit record 信号；泛 `ok/status/result:"success"` 不计为抽奖成功。

benefits scope 的缺口名称固定为 `successful_daily_sign_in_fixture`、`successful_pro_activity_fixture`、`successful_reset_coupon_consumption_fixture` 和 `successful_lottery_draw_fixture`。它只用于检查 M05 副作用 evidence 是否闭环；读取 fixture 详情和 `counts.total/success/failed` 只覆盖 M05 side-effect 白名单 operation，避免默认 protocol、auth 或 session fixture 污染 benefits 缺口判断；默认 `fixtures audit` 不要求 benefits evidence。

upstream scope 的缺口名称固定为 `real_upstream_error_frame_fixture`、`real_upstream_cancellation_fixture` 和 `real_upstream_backpressure_fixture`。它只读取 `sendMessage` fixture，并要求 fixture 同时带显式真实上游 evidence marker 与 stream/SSE/NDJSON 元数据；非流式 `protocol_probe` sendMessage 可满足默认 send readiness，但不满足 upstream stream boundary readiness。本地 HTTP route、compat handler、unit/fake stream fixture 不计为真实上游 evidence。默认 `fixtures audit` 不要求 upstream evidence。

`scope:"session"` 返回 `scope:"session"`，覆盖项为：

- `successfulSessionVerify`：成功 `verifySession` fixture。
- `expiredSessionSignal`：失败 `verifySession` fixture，且 error/result error/status/httpStatus/statusCode 显示 401 或 `login_required`。

session scope 的缺口名称固定为 `successful_verifySession_fixture`、`expired_verifySession_fixture` 和 `automated_session_refresh_strategy`。`session_missing` 只表示本地 secret 缺失，会计入 `counts.sessionMissing`，但不计入 `expiredSessionSignal`。输出还包含 `lifecycle.lastSuccessfulAt`、`lifecycle.lastExpiredAt`、`lifecycle.observedWindowMs`、`manualCookieOperations` 和 `recoveryStrategy`。`manualCookieOperations.missing` / `blockingMissing` 只列当前手动 cookie 运维发布目标缺口；`manualCookieOperations.backlogMissing` 列后续增强缺口，例如自动恢复未校准时的 `automated_session_refresh_strategy`。默认恢复策略为 `{ status:"blocked", current:"manual_reimport_then_probe", automatedRefresh:"not_calibrated" }`；只有显式脱敏 `session_recovery_strategy` / `recoverSession` evidence 声明 `safe:true`、`sanitized:true`、`rawPayload:false`、正整数 `observedWindowMs`、`resultHash` 脱敏哈希，并使用已校准 `automated_reauth` 或 `refresh_token` 策略，同时 `result.expiredBeforeRecovery:true` 与 `result.recoveredVerifySession:true` 时，`recoveryStrategy.status` 才会变为 `ready`。即使成功/过期 lifecycle coverage 都 ready，只要自动刷新或安全 re-auth recovery path 未校准，session scope 仍保持 blocked，但 manual-cookie 当前发布阻塞项可为空。


输出边界：

- `status` 只有在 readiness 和默认 fixture audit 都 ready 时才是 `ready`；任一 blocked 时为 `blocked`，否则为 `partial`。
- `protocol` 只输出布尔配置状态和 `toolLoopMode`，不输出 API key、`TABBIT_POOL_PROTOCOL_REQ_CTX`、签名上下文、cookie 或 session。
- `calibrationBacklog` 输出 `{ status, scopes:{ auth, benefits, session, upstream }, missing, nextActions }`，用于显示注册/登录、M05 副作用、session lifecycle 和真实上游 stream boundary evidence 缺口；它不改变 top-level `status`。

- `remainingWork` 是 readiness 与默认 fixture audit `nextActions` 的去重合并，只代表 core gateway/chat gate。
- `commands` 只包含安全本地命令示例：设置 `TABBIT_POOL_STATE_DIR`、运行 doctor/readiness/default audit、auth/benefits/session/upstream scoped audit 和启动 gateway。
- 不包含账号明细、`cookieJarRef`、raw fixture request/response、prompt、代码 diff 或真实用户数据。

### bin/tabbit-pool.js

package bin 暴露 `tabbit-pool`，wrapper 只做三件事：读取 `process.argv.slice(2)`、调用 `runProtocolPoolCli()`、把返回值写入 `process.exitCode`。

### Probe protocol input schema validation

`runProtocolPoolCli()` 在 `probe protocol` 调用 runner 前执行轻量 schema 校验，`probe validate` 复用同一批稳定字段校验。校验只覆盖当前 dispatch 已稳定使用的字段，未知字段会保留给真实 endpoint 校准；`recoverSession` 和 `consumeResetCoupon` 例外，它们是离线 evidence，不会进入 runner，必须提供显式脱敏 evidence input。`probe validate --operation sendMessage --json` 还会输出脱敏 `sendMessageReview`，其中 `protocolDispatchReady:true` 只代表至少一条消息字符串不是占位符，不代表真实 probe 可跳过人工内容审查。`probe validate --write-fixture` 会先校验再写入本地 fixture store，且只保留白名单 evidence/result 字段；非离线 evidence operation 使用该 flag 会在触碰 fixture store 前被拒绝。

| operation | 字段 | 规则 |
|---|---|---|
| `sendVerificationCode` / `submitRegistrationOrLogin` | `confirmSideEffect` | 如果出现，必须是 boolean；runner 仍会在触网前要求值为 `true`。 |
| `sendVerificationCode` / `submitRegistrationOrLogin` | `email` / `mobile` | 至少一个必须是非空字符串；真实 proxy OAuth 模板默认使用 `mobile`。 |
| `submitRegistrationOrLogin` | `code` | 必须是非空字符串。 |
| `sendVerificationCode` / `submitRegistrationOrLogin` | `uuid` | 如果出现，必须是 64 位字母数字 auth client value。 |
| `sendVerificationCode` / `submitRegistrationOrLogin` | `body` | 如果出现，必须是普通 JSON object；用于传入已确认的真实 auth body。 |
| `sendMessage` | `model` | 如果出现，必须是非空字符串。 |
| `sendMessage` | `messages` | 如果出现，必须是非空数组。 |
| `sendMessage` | `streamEvidence` | 如果出现，必须同时设置 `stream:true`；值必须是普通 object，`mode` 只能是 `first_token_backpressure`、`cancel_after_first_delta` 或 `error_frame`，`maxDeltas` 可省略，出现时必须是 1 到 5 的整数。 |
| `listModels` | `force` | 如果出现，必须是 boolean。 |
| `refreshQuota` | `userId` | 如果出现，必须是非空字符串；缺失时 runner 使用账号元数据里的 userId。 |
| `getNewbieExplorationMe` | `viewMode` | 如果出现，必须是 `event_gate`、`float_collapsed`、`float_expanded` 或 `activity_page`。 |
| `getNewbieExplorationMe` | `includeCompletions` / `includeRewards` | 如果出现，必须是 boolean。 |
| `getPlacementResources` | `placementCode` / `clientVersion` | 如果出现，必须是非空字符串；缺失时 `placementCode` 使用默认 `home.input_below`。 |
| `listRewardCardRecords` / `listLotteryHitRecords` | `userId` | 如果出现，必须是非空字符串；缺失时 runner 使用账号元数据里的 userId。 |
| `listRewardCardRecords` / `listLotteryHitRecords` | `offset` / `limit` | 如果出现，必须是非负整数。 |
| `getDailySignInStatus` / `dailySignIn` | `sceneCodes` | 如果出现，必须是非空字符串数组。 |
| `dailySignIn` / `participateResetCouponActivity` / `participateActivity` / `drawLottery` | `confirmSideEffect` | 如果出现，必须是 boolean；真实副作用方法缺少 `confirmSideEffect:true` 会在协议 client 内触网前失败。 |
| `dailySignIn` / `participateResetCouponActivity` | `requestNo` | 如果出现，必须是非空字符串且不超过 64 字符；协议 client 实际请求前要求存在。 |
| `listBenefitCoupons` | `userId` / `couponType` / `status` | 如果出现，必须是非空字符串；缺失 userId 时 runner 使用账号元数据。 |
| `listBenefitCoupons` / `listLotteryChanceRecords` | `offset` / `limit` | 如果出现，必须是非负整数。 |
| `participateActivity` / `drawLottery` | `body` | 如果出现，必须是普通 JSON object；协议 client 实际请求前要求存在。 |
| `getAvailableLotteryChanceCount` / `getActiveMainPools` / `listLotteryChanceRecords` | `activityId` | 如果出现，必须是非空字符串；协议 client 实际请求前要求存在。 |
| `uploadAttachment` | `attachment` | 如果出现，必须是普通 JSON object；缺失时 runner 使用默认附件探针。 |
| `uploadAttachment` | `attachment.filename` / `attachment.mimeType` / `attachment.data` | 如果出现，必须是非空字符串。 |
| `recoverSession` | `kind` / `operation` / `status` | 必须分别为 `session_recovery_strategy`、`recoverSession`、`success`；该 operation 只允许 `probe validate` 离线校验，`probe protocol` 会拒绝派发。 |
| `recoverSession` | `evidence` | 必须是普通 JSON object，且 `strategy` 为 `automated_reauth` 或 `refresh_token`、`automatedRefresh` 为 `calibrated_reauth_probe` 或 `calibrated_refresh_probe`、`observedWindowMs` 为正整数、`resultHash` 为 `sha256:` 脱敏哈希、`safe:true`、`sanitized:true`、`rawPayload:false`。 |
| `recoverSession` | `result` | 必须是普通 JSON object，并以 `expiredBeforeRecovery:true` 和 `recoveredVerifySession:true` 证明过期后恢复路径已重新通过 `verifySession`；不得写入 raw cookie/session/token。 |
| `consumeResetCoupon` | `kind` / `operation` / `status` | 必须分别为 `reset_coupon_consumption_evidence`、消费类 operation、`success`；该 operation 只允许 `probe validate` 离线校验，`probe protocol` 会拒绝派发。 |
| `consumeResetCoupon` | `evidence` | 必须是普通 JSON object，且包含 `endpointHash`、`bodyHash`、`resultHash` 三个 `sha256:` 脱敏哈希、`safe:true`、`sanitized:true`、`rawPayload:false`。 |
| `consumeResetCoupon` | `result` | 必须是普通 JSON object，并含真实消费成功信号，例如 `resetCouponConsumed:true`、`couponConsumed:true`、`consumed:true`、`used:true`、`deducted:true` 或 `consumeResult` / `couponResult` / `usageResult` success；`already_participated` / `already_claimed` 等非消费信号会被拒绝。 |
| `verifySession` | 任意字段 | 当前只要求顶层 input 是 JSON object。 |

校验失败抛出 `CliUsageError`，CLI 返回 exitCode 2，stderr 只描述字段规则，不回显原始 JSON payload，也不会调用注入的 `protocolProbeRunner.probeAccount()`。

## Protocol probe fixture 接口

### buildProtocolProbeFixture(input)

生成脱敏 `protocol_probe` fixture。输入可包含 observedAt、accountId、operation、status、account、input、result、error、advice。输出固定包含 version=1、kind、observedAt、operation、status 和 advice；account 使用 `redactAccountForDisplay()`，input/result 使用递归敏感字段脱敏，error.message 会额外屏蔽 4 到 8 位裸数字。递归 sanitizer 还会屏蔽 prompt/正文类字段（如 `content`、`prompt`、`text`、`html_content`）、stream delta、raw event/frame `data`、附件 `data` 和真实用户标识（如 `userId`、`user_id`、`user_info.id`、昵称/头像等），fixture 只保留 operation/status/结构化成功信号供 audit 判断。

~~~ts
type ProtocolProbeFixture = {
  version: 1;
  kind: "protocol_probe";
  observedAt: string;
  operation: string;
  accountId?: string;
  status: string;
  account?: RedactedAccount;
  input?: unknown;
  result?: unknown;
  error?: {
    category?: string;
    code?: string;
    status?: number;
    message?: string;
    retryable?: boolean;
    cooldownMs?: number;
  };
  advice: { category: string; severity: string; recommendation: string };
};
~~~

`RedactedAccount` 表示 `redactAccountForDisplay(account)` 的输出，不包含 `cookieJarRef`、`cookieHeader`、`session`、`sessionToken` 或 `token`。

敏感字段边界：`email` 会按展示规则脱敏；`apiKey`、`authorization`、`cookie`、`cookieHeader`、`cookieJar`、`cookieJarRef`、`password`、`secret`、`session`、`sessionToken`、`token`、`verification`、`code` 会被屏蔽；`attachment.data` / `attachments[].data` 会直接替换为 `***`；普通字符串中的邮箱、token/cookie 片段和 4 到 8 位裸数字也会被清理。`result.upstreamEvidence` 仅保留 `source`、`format` 和布尔 marker，用于真实上游 stream boundary audit；raw stream `data`、prompt、payload 和用户身份仍按 fixture sanitizer 屏蔽。

### sanitizeProtocolProbeFixture(input)

对已生成或从磁盘读取的 protocol probe fixture 再做一次展示脱敏。它会保留 `observedAt` 原始字符串用于排序和排障；如果存在 `account` 对象，会先调用 `redactAccountForDisplay()` 移除 `cookieJarRef`、`cookieHeader`、`session`、`sessionToken`、`token` 等字段，再递归屏蔽 input/result/error 中的敏感字段、`attachment.data` 和 4 到 8 位裸数字。

### new FileProtocolFixtureStore(options)

参数：`{ stateDir, fixtureDir?, fs?, now?, idFactory? }`。

| 方法 | 返回 | 说明 |
|---|---|---|
| `writeFixture(fixture)` | `Promise<string>` | 默认写入 `stateDir/fixtures/protocol-probes/<id>.json`；提供 `fixtureDir` 时写入 `<fixtureDir>/<id>.json`。写入前调用 `sanitizeProtocolProbeFixture()`，返回稳定相对 ref。 |
| `listFixtures()` | `Promise<ProtocolProbeFixtureSummary[]>` | 只枚举当前 fixture 根目录下的 JSON 文件，返回的 ref 仍为 `fixtures/protocol-probes/<name>.json`；读取审计允许 kind：`protocol_probe`、`session_recovery_strategy` 与 `reset_coupon_consumption_evidence`，未知 kind 会被忽略；按 `observedAt` 倒序、再按 ref 升序排序。 |
| `readFixture(ref)` | `Promise<ProtocolProbeFixture>` | 读取指定 ref，ref 必须解析到 protocol probe fixture 根目录内，返回再次脱敏后的 fixture。 |
| `resolveFixtureRef(ref)` | `{ filePath, ref }` | 校验并规范化 ref；缺失或路径越界会抛 `ProtocolFixtureStoreError`。 |

`ProtocolProbeFixtureSummary` 字段：`ref`、`observedAt?`、`operation?`、`status`、`accountId?`、`adviceCategory?`。

### ProtocolFixtureStoreError

结构化 fixture store 错误。常见 `code`：`FIXTURE_REF_REQUIRED`、`FIXTURE_REF_OUTSIDE_ROOT`、`FIXTURE_NOT_FOUND`、`FIXTURE_INVALID_JSON`。`exitCode` 用于 CLI：ref 缺失、越界或文件缺失为 2；非法 JSON 和其他运行时错误为 1。

### new ProtocolProbeRunner(options)

~~~ts
type ProtocolProbeRunnerOptions = {
  accountStore: { loadAccounts(): Promise<Account[]> };
  secretStore?: { readSecret(ref: string): Promise<string | null> };
  protocolClientFactory?: (account: Account) => object;
  fixtureStore?: { writeFixture(fixture: object): Promise<string> };
  now?: () => number | string | Date;
};
~~~

`probeAccount({ accountId, operation, input, writeFixture })` 支持 operation：`verifySession`、`sendVerificationCode`、`submitRegistrationOrLogin`、`sendMessage`、`listModels`、`refreshQuota`、`uploadAttachment`、`getLotteryExplorationMe`、`getNewbieExplorationMe`、`getPlacementResources`、`listRewardCardRecords`、`listLotteryHitRecords`、`getDailySignInStatus`、`dailySignIn`、`listBenefitCoupons`、`participateResetCouponActivity`、`participateActivity`、`getUsageResetCouponSku`、`getAvailableLotteryChanceCount`、`getActiveMainPools`、`listLotteryChanceRecords`、`drawLottery`。CLI 可通过 `--input-json <json>` 或 `--input-file <path>` 传入 input object；未传入时 runner 使用默认 `{}`。默认没有 protocolClientFactory 时返回 skipped/protocol_missing，不猜测真实 Tabbit endpoint。读取到 cookieJarRef secret 后会传入 runtime account，但 fixture 输出永不包含 raw session。auth operation 会把 `email`、`code`、`body`、完整 `input` 和 runtime account 传给协议 client；缺少 `confirmSideEffect:true` 时在读取 secret 或 client dispatch 前返回 `failed/invalid_request`。auth operation 可使用没有 `cookieJarRef` secret 的 provisioning 账号元数据，便于校准登录前 send-code/submit-code endpoint；其它 operation 仍要求可读取的本地 session。`sendMessage` 若显式提供 `streamEvidence`，runner 只在结果已有真实 `/api/v1/chat/completion` async marker `source:"tabbit-live"`、`real:true`、`stream:true` 且 `streamDeltas` 是 async iterable 时，按 `maxDeltas` 有界消费 delta，并只把 `backpressure`、`firstTokenFlush`、`delayedSecondChunk`、`cancellation` 或 `streamErrorFrame` 等布尔 marker 合并到 `result.upstreamEvidence`；delta 文本本身不保存到新字段。`refreshQuota` 会调用 `client.refreshQuota({ account: runtimeAccount, userId?: input.userId })`；未传 `input.userId` 时使用账号元数据中的 `userId`。只读记录查询也会把 runtime account 和可选 `input.userId` 传给客户端；`getNewbieExplorationMe` 会透传 viewMode/include flags 和 intent 字段；`getPlacementResources` 会透传 `placementCode` 和 `clientVersion`。M05 side-effect operation 会原样透传 `confirmSideEffect`，协议 client 缺少确认时触网前拒绝。`uploadAttachment` 会调用 `client.uploadAttachment({ ...input, attachment: input.attachment || defaultAttachmentProbeInput(), account: runtimeAccount })`；如果 input 没有 `attachment`，使用 `probe.txt/text/plain/base64-probe-payload` 默认附件探针。

~~~ts
type ProtocolProbeRunnerResult = {
  status: "success" | "failed" | "skipped" | string;
  account: RedactedAccount | null;
  fixture: ProtocolProbeFixture;
  advice: { category: string; severity: string; recommendation: string };
  fixtureRef?: string;
};
~~~

结果边界：

| status | error.category | 触发条件 |
|---|---|---|
| `success` | 无 | 注入的 operation 返回成功结果，或返回对象没有 `ok:false`。 |
| `failed` | `account_not_found` | `accountStore.loadAccounts()` 找不到 `accountId`。 |
| `failed` | `session_missing` | 非 auth operation 的账号没有可读取的 `cookieJarRef` secret。 |
| `skipped` | `protocol_missing` | 未配置 `protocolClientFactory`，或 client 缺少对应 operation 方法。 |
| `failed` | `invalid_request` | operation 不在 runner 支持列表，或 auth probe 缺少 `confirmSideEffect:true`。 |
| `failed` | 调用方错误分类或 `unknown` | 注入 operation 返回 `ok:false` 或抛出异常。 |

`sendMessage` probe 的默认输入是 `model:"tabbit/priority"`、单条 user `<redacted-message-content>` 占位、`stream:false`，但调用方可以通过 `input` 覆盖；auth probe 模板默认 placeholder email/code/body 和 `confirmSideEffect:false`，调用方必须基于真实 evidence 改 body 并显式设为 true 才会触发发送验证码或提交验证码；`listModels` probe 默认带 `force:true`；`refreshQuota` probe 默认从账号元数据取 `userId`，也可用 `input.userId` 覆盖；`getLotteryExplorationMe` 默认 `{}`；`getNewbieExplorationMe` 默认 `activity_page` 和 include flags；`getPlacementResources` 默认 `placementCode:"home.input_below"`；`listRewardCardRecords` 默认 offset 0/limit 10；`listLotteryHitRecords` 默认 offset 0/limit 20；M05 side-effect probe 模板默认 `confirmSideEffect:false`，调用方必须显式改为 true 才会触发 POST；`uploadAttachment` probe 默认附件是 `probe.txt`、`text/plain` 和占位 `base64-probe-payload`，fixture 中 `attachment.data` 会被脱敏。

## HTTP server 接口

### createProtocolPoolServer(input)

~~~ts
type CreateProtocolPoolServerInput = {
  apiKey?: string;
  compat: {
    handleChatCompletions(body: unknown): Promise<{ status: number; body: unknown }>;
    handleResponses(body: unknown): Promise<{ status: number; body: unknown }>;
    handleMessages(body: unknown): Promise<{ status: number; body: unknown }>;
  };
  modelsProvider?: { listModels(input?: object): Promise<unknown[]> } | (() => Promise<unknown[]>);
  health?: object | (() => object | Promise<object>);
};
~~~

返回原生 node:http Server。当前路由：

| 方法 | 路径 | 行为 |
|---|---|---|
| GET | /health | 不需要认证，返回 { status:"ok", mode:"protocol-pool", ...health }。 |
| GET | /v1/models | 需要认证，返回 OpenAI models.list shape。 |
| POST | /v1/chat/completions | 需要认证，解析 JSON 后调用 compat.handleChatCompletions；`stream:true` 且 2xx 时写 SSE，有数组 `stream.deltas` 时保留分片，没有时完整文本 fallback；有 async iterable `stream.deltas` 时 chunked flush；async iterator 抛错后写 OpenAI `data:error` 与 `[DONE]`。 |
| POST | /v1/responses | 需要认证，解析 JSON 后调用 compat.handleResponses；`stream:true` 且 2xx 时写 SSE，有数组 `stream.deltas` 时保留分片，没有时完整文本 fallback；有 async iterable `stream.deltas` 时 chunked flush；async iterator 抛错后写 `event: response.failed` 与 `[DONE]`。 |
| POST | /v1/messages | 需要认证，解析 JSON 后调用 compat.handleMessages；`stream:true` 且 2xx 时写 Anthropic SSE，有数组 `stream.deltas` 时保留分片，没有时完整文本 fallback；有 async iterable `stream.deltas` 时 chunked flush；async iterator 抛错后写 Anthropic `event: error`。 |

`/v1/models` 的模型来源只由注入的 `modelsProvider` 或默认 fallback 决定：未传 provider 时返回单个 `tabbit/priority`；传入函数 provider 时直接调用该函数；传入对象 provider 时调用 `provider.listModels()`。HTTP server 不读取 `TABBIT_POOL_PROTOCOL_*`，也不自行请求 Tabbit。若 provider 返回 `ProtocolTabbitClient.normalizeModelCatalog()` 形状，server 会把 `selectedModel` 映射到 `tabbit_selected_model`，并输出 OpenAI `object:"model"` 条目。

### readJson(req)

读取请求 body。空 body 返回 {}；坏 JSON 抛出内部 INVALID_JSON 错误，由 server 映射为 400 invalid_request_error / invalid_json。

### writeJson(res, status, body)

写入 application/json; charset=utf-8，并设置 Content-Length。

### sseData(payload)

把对象或字符串包装为 SSE `data:` frame。对象会先 JSON.stringify；字符串按原文写入。

### writeSse(res, events)

写入 `text/event-stream; charset=utf-8`、`Cache-Control: no-cache`、`Connection: keep-alive` 与 Content-Length，然后结束响应。当前用于有限数组 delta 或完整文本 fallback。

### writeSseStream(res, events)

写入 `text/event-stream; charset=utf-8`、`Cache-Control: no-cache` 与 `Connection: keep-alive`，不设置 Content-Length；随后写入预格式化 SSE frame，最后结束响应。`events` 可以是 async iterable，也可以是 `(signal) => asyncIterable` factory；HTTP route adapter 使用 factory 形式把 downstream close signal 传给 streaming SSE adapter。当前用于 async iterable `stream.deltas` 的 chunked flush。若 iterator 抛错且 headers 已发送，会调用 `errorEvents(error)` 生成 SSE error frame；默认是 OpenAI `data: {"error":...}` + `[DONE]`，Responses route 会传入 `response.failed` mapper，Anthropic route 会传入 `event:error` mapper。若客户端断开连接，writer 会 abort close signal、停止继续写 SSE，并请求当前 events iterator `return()`；streaming adapter 会进一步请求 `stream.deltas` iterator `return()`。该路径保证本地可取消 async source 不会继续被消费，但真实 Tabbit 上游取消时机仍取决于协议客户端和底层 fetch/body 行为。

### chatCompletionToSseEvents(body, stream?)

把 Chat Completion JSON 转换为 SSE frame 数组：assistant role delta、一个或多个 content delta、可选 `tool_calls` delta、finish chunk、`[DONE]`。数组 `stream?.deltas` 非空时逐 delta 输出；否则使用 `choices[0].message.content` 作为完整文本 fallback；`choices[0].message.tool_calls` 会按完整 tool call 生成兼容 delta。async iterable `stream.deltas` 不走该有限数组 helper，而由 HTTP route adapter 使用 chunked writer。

### responsesToSseEvents(body, stream?)

把 Response JSON 转换为 SSE frame 数组：`response.created`、一个或多个 `response.output_text.delta`、`response.completed`、`[DONE]`。数组 `stream?.deltas` 非空时逐 delta 输出；否则使用 `output_text` 作为完整文本 fallback。async iterable `stream.deltas` 由 HTTP route adapter 使用 chunked writer。

### anthropicMessageToSseEvents(body, stream?)

把 Anthropic message JSON 转换为 SSE frame 数组：`message_start`、每个 content block 的 `content_block_start` / `content_block_delta` / `content_block_stop`、`message_delta`、`message_stop`。数组 `stream?.deltas` 非空时，第一个 text block 逐 delta 输出多个 `content_block_delta`；否则 text delta 是完整文本块。async iterable `stream.deltas` 由 HTTP route adapter 使用 chunked writer。

### isAuthorized(req, apiKey)

接受两种本地认证形式：Authorization: Bearer <apiKey> 与 x-api-key: <apiKey>。默认 apiKey 是 sk-tabbit-local。

### openAiHttpError(status, message, type, code)

构造 { status, body:{ error:{ message, type, code } } }。HTTP route adapter 用它返回 401、400、404 和 500。

## Protocol Pool Gateway 接口

### createProtocolPoolGateway(options)

异步组合工厂，返回可启动 gateway：

~~~ts
type ProtocolPoolGateway = {
  config: object;
  store: JsonAccountStore;
  secretStore: FileSecretStore;
  accountPool: StoredAccountPool;
  protocolClientFactory: Function;
  runner: PooledRequestRunner | LocalToolLoopRunner;
  openAiCompat: OpenAICompat;
  anthropicCompat: AnthropicCompat;
  compat: object;
  server: http.Server;
  start(options?: { host?: string; port?: number }): Promise<http.Server>;
  close(): Promise<void>;
};
~~~

默认组合：`loadConfig()` -> `JsonAccountStore` + `FileSecretStore` -> `StoredAccountPool.load()` -> secret-hydrating protocol client factory -> `PooledRequestRunner` -> `LocalToolLoopRunner` -> `OpenAICompat` + `AnthropicCompat` -> `createGatewayHealthProvider()` -> `createProtocolPoolServer()`。当 `config.protocol.enabled` 为 true 时，gateway 会把 `config.protocol` 转成 `ProtocolTabbitClient` options，并与显式 `options.protocolClientOptions` 合并；显式 options 优先。`TABBIT_POOL_PROTOCOL_SEND_PATH` 会成为默认 `sendPath`，auth send/submit env 会成为默认 `authSendCodePath`、`authSendCodeMethod`、`authSubmitCodePath`、`authSubmitCodeMethod`，`TABBIT_POOL_PROTOCOL_CHAT_SESSION_ID` 会成为 `defaultChatSessionId`，`TABBIT_POOL_PROTOCOL_REQ_CTX` 会成为 `reqCtx`，`TABBIT_POOL_PROTOCOL_ATTACHMENT_UPLOAD_PATH` 会成为默认 `attachmentUploadPath`，`TABBIT_POOL_PROTOCOL_ATTACHMENT_COMPLETE_UPLOAD_PATH` 会成为默认 `attachmentCompleteUploadPath`，`TABBIT_POOL_PROTOCOL_QUOTA_USAGE_PATH` 会成为默认 `quotaUsagePath`，只读 commerce path 会成为默认 `activityLotteryPath`、`newbieExplorationPath`、`placementResourcesPath`、`rewardCardRecordsPath`、`lotteryHitRecordsPath`，M05 path 会成为对应 `signInStatusPath`、`signInPath`、`benefitCouponListPath`、`activityParticipatePath`、`usageResetCouponSkuPath`、`lotteryAvailableChancesPath`、`lotteryActiveMainPoolsPath`、`lotteryChanceRecordsPath`、`lotteryDrawPath`，`TABBIT_POOL_PROTOCOL_MODEL_CATALOG_PATH` 和 `TABBIT_POOL_PROTOCOL_MODEL_CATALOG_SCENE` 会成为模型目录配置。`config.compat.localToolLoop` 会传给默认 `LocalToolLoopRunner`，从 env 控制本地工具 allowlist、轮数、超时和结果截断；显式 `options.localToolLoop` 可覆盖这些值，显式 `options.toolLoopRunner` 则完全覆盖默认 wrapper。未显式传入 `options.modelsProvider` 且协议 env 已 opt-in 时，gateway 会创建 env-backed `modelsProvider`，由 `baseProtocolClientFactory({}).listModels()` 支撑 `/v1/models`；未 opt-in 时保持 HTTP server 的 `tabbit/priority` fallback。显式 `options.modelsProvider` 优先级最高，会覆盖 env-backed provider。

可注入项包括 config、store、secretStore、accountPool、runner、toolLoopRunner、compat、protocolClientFactory、protocolClientOptions、fetch、modelsProvider、health、now、compatNow、idFactory、hydrateSecrets、executeLocalToolUse、localToolExecutor 和 localToolLoop。测试中推荐注入 protocolClientFactory，避免真实 Tabbit 网络请求；只有 `TABBIT_POOL_TOOL_LOOP_MODE=local_executes_tools` 且注入 executor 时，gateway 才执行本地工具。

### createDefaultProtocolClientFactory(options)

返回 `(account) => new ProtocolTabbitClient(...)` 的工厂。`sendPath` 等协议参数来自 `protocolClientOptions`，不在启动工厂中硬编码。

### hydrateAccountSecrets(account, secretStore)

复制账号对象。如果账号没有 `cookie` / `cookieHeader` 但存在 `cookieJarRef`，则通过 `secretStore.readSecret(cookieJarRef)` 读取文本，并把它放入运行时账号对象的 `cookieHeader`。不会修改账号池中的原对象。

### createSecretHydratingProtocolClientFactory(baseFactory, secretStore)

包装 protocol client factory，让 `sendMessage()`、`verifySession()`、`uploadAttachment()`、`refreshQuota()`、只读 commerce 查询方法和 M05 显式 probe 方法前先 hydrate runtime account，并把 hydrate 后的 account 传给内部 client；verifySession 会把读取到的 cookie 同时作为 `session` 传入。若底层 client 缺少对应 operation，wrapper 返回或抛出 `protocol_missing` 风格失败结果，不猜测或补实现。

### listen(server, options) 与 closeServer(server)

Promise 化 node:http server 的 listen/close。`createProtocolPoolGateway().start()` 默认使用 `config.host` 与 `config.port`，但测试可传 `port: 0` 绑定临时端口。
